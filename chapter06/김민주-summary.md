# Chapter 6: 파티셔닝 (Partitioning)

## 1. 파티셔닝이란?

### 왜 파티셔닝이 필요한가?

- 데이터셋이 단일 노드의 처리 한계를 초과할 때 **여러 노드에 데이터를 분산** 저장
- 대용량 데이터셋의 **쿼리 부하 분산** → 더 많은 노드 = 더 높은 처리량
- 보통 **복제(Replication)와 함께** 사용하여 내결함성 확보

### 파티셔닝 용어 정리

| 시스템 | 파티션 용어 |
|--------|------------|
| MongoDB, Elasticsearch | **샤드 (Shard)** |
| HBase | **리전 (Region)** |
| Bigtable | **태블릿 (Tablet)** |
| Cassandra, Riak | **vnode** |
| Couchbase | **vBucket** |

### 파티셔닝의 핵심 목표

```
데이터와 쿼리 부하를 노드 전체에 고르게 분산
→ 핫스팟(Hotspot) 없이 균등 분배
→ 이론상 N개 노드 = N배 처리량
```

---

## 2. 키-값 데이터의 파티셔닝

### 2.1 키 범위 기반 파티셔닝 (Key Range Partitioning)

각 파티션에 **연속된 키 범위**를 할당하는 방식

```
파티션 1: A - D
파티션 2: E - H
파티션 3: I - M
파티션 4: N - Z
```

**장점**
- 범위 쿼리(Range Query)에 효율적
- 키를 정렬된 순서로 유지 가능

**단점: 핫스팟 위험**

```
예시: 타임스탬프를 키로 사용
→ 오늘 날짜 파티션에만 쓰기가 집중!
→ 해결책: 타임스탬프 앞에 센서 이름을 붙임
   (예: weather_station_1_2024-01-15)
```

> 실제 사용 예: **Bigtable**, **HBase**, **RethinkDB**, **MongoDB (2.4 이전)**

---

### 2.2 키 해시 기반 파티셔닝 (Hash Partitioning)

키를 **해시 함수**에 통과시켜 파티션을 결정

```
hash("user_001") → 파티션 3
hash("user_002") → 파티션 1
hash("user_003") → 파티션 5
```

**장점**
- 키가 균등하게 분산 → 핫스팟 방지
- 데이터 쏠림 현상 없음

**단점**
- 범위 쿼리 불가 → 모든 파티션에 개별 요청 필요

> 실제 사용 예: **Cassandra**, **MongoDB**, **Riak**
> 언어 내장 해시(Java의 `hashCode()` 등)는 같은 키가 다른 값을 낼 수 있어 **비적합**

---

### 2.3 일관된 해싱 (Consistent Hashing)

- CDN 업계에서 노드 추가/제거 시 **재균형화를 최소화**하기 위해 개발
- 해시 범위를 원형 링(ring)으로 구성하고, 가상 노드를 배치
- 실제로 많은 DB가 이 방식을 "일관된 해싱"이라 부르지만, 문헌의 정의와 다른 경우 많음

---

### 2.4 핫스팟 완화 (Skewed Workloads)

해시를 써도 **특정 키에 트래픽이 집중**되면 핫스팟 발생

```
예시: 유명인 SNS 게시물 키 = user_id
     특정 유명인 게시물에 수백만 요청 집중
```

**완화 방법:**
- 핫 키에 **무작위 숫자를 붙여** 여러 파티션으로 분산
  - `user_123_01`, `user_123_02`, ..., `user_123_99`
  - 읽기 시 99개 키 모두 조회 후 합산 필요
- 애플리케이션 레벨에서 핫 키 추적 로직 직접 구현 필요

---

## 3. 파티셔닝과 보조 인덱스

### 문제 상황

- 기본 키 이외의 필드로 검색이 필요할 때
- 파티션은 기본 키 기준 → 보조 인덱스는 파티션을 자연스럽게 매핑하기 어려움

---

### 3.1 문서 기반 보조 인덱스 (Local Index / Scatter-Gather)

각 파티션이 **자신의 로컬 인덱스**만 관리

```
파티션 1:           파티션 2:           파티션 3:
문서: 차량 목록     문서: 차량 목록     문서: 차량 목록
로컬 인덱스:        로컬 인덱스:        로컬 인덱스:
 color:red → [doc1]  color:red → [doc5]  color:red → [doc9]
 make:VW → [doc2]    make:VW → [doc6]    make:VW → [doc10]
```

**쿼리 흐름: Scatter-Gather**

```
클라이언트 → 모든 파티션에 쿼리 전송 (Scatter)
           ← 각 파티션 결과 수집 및 합산 (Gather)
```

**장점** → 쓰기가 빠름 (해당 파티션만 업데이트)
**단점** → 읽기가 느리고 비용이 큼 (모든 파티션 조회 필요)

> 실제 사용 예: **MongoDB**, **Riak**, **Cassandra**, **Elasticsearch**, **VoltDB**

---

### 3.2 용어 기반 보조 인덱스 (Global Index)

인덱스 자체를 **모든 파티션에 걸쳐 분산** 저장

```
color:red    → 인덱스 파티션 A (여러 파티션의 red 차량 doc ID)
color:yellow → 인덱스 파티션 B
make:VW      → 인덱스 파티션 A
make:Audi    → 인덱스 파티션 C
```

**쿼리 흐름**

```
"red 차량 조회" → 인덱스 파티션 A만 조회
                ← 해당 문서 ID들 반환
                → 해당 문서들이 있는 파티션만 조회
```

**장점** → 읽기가 빠름 (단일 파티션만 조회)
**단점** → 쓰기가 느림 (여러 파티션의 인덱스 업데이트, 분산 트랜잭션 필요)

> 실제 사용 예: **Amazon DynamoDB (Global Secondary Index)**, **Riak (Search)**, **Oracle**

---

## 4. 파티션 재균형화 (Rebalancing)

### 재균형화가 필요한 이유

- 쿼리 처리량 증가 → 노드 추가 필요
- 데이터 증가 → 디스크/RAM 증설 필요
- 장애 발생 → 다른 노드가 대신 처리

> **재균형화 원칙:** 재균형화 후에도 부하는 노드 간에 균등 분산되어야 하며, 재균형화 중에도 읽기/쓰기는 계속 동작해야 한다.

---

### 4.1 피해야 할 방법: `hash mod N`

```
노드 10개: hash(key) mod 10
노드 11개: hash(key) mod 11
→ 거의 모든 키가 다른 노드로 이동 필요!
```

→ 재균형화 비용이 너무 큼

---

### 4.2 고정 파티션 수 (Fixed Number of Partitions)

파티션 수를 노드 수보다 **훨씬 크게 설정**하고 고정

```
노드 10개, 파티션 1000개
→ 노드당 ~100개 파티션 담당

노드 추가 시:
→ 신규 노드가 기존 노드에서 파티션 일부를 가져옴
→ 파티션 총 개수는 변하지 않음!
```

**장점**
- 재균형화 시 전체 파티션 이동 최소화
- 각 파티션 단위로 이동 → 원자적(atomic) 작업

**단점**
- 파티션 수가 고정 → 미래 데이터 증가 예측 어려움
- 파티션이 너무 많으면 관리 오버헤드, 너무 적으면 재균형화 비효율

> 실제 사용 예: **Riak**, **Elasticsearch**, **Couchbase**, **Voldemort**

---

### 4.3 동적 파티셔닝 (Dynamic Partitioning)

파티션이 **지정된 크기 임계값**에 도달하면 자동 분할

```
HBase 기본값: 파티션 크기 10GB 초과 시
→ 2개로 분할 (각 ~5GB)
→ 분할된 파티션 중 하나를 다른 노드로 이동

반대로 파티션이 너무 작으면:
→ 인접 파티션과 병합 (Merge)
```

**장점**
- 파티션 수가 데이터 볼륨에 맞게 자동 조정
- 작은 데이터셋 → 적은 파티션 수 → 오버헤드 없음

**단점**
- 처음 시작 시 파티션이 1개 → 모든 쓰기가 1개 노드에 집중
- 해결: **사전 분할(Pre-splitting)** 설정으로 초기 파티션 수 지정

> 실제 사용 예: **HBase**, **RethinkDB**, **MongoDB (3.4+)**

---

### 4.4 노드 비례 파티셔닝 (Partitioning Proportional to Nodes)

파티션 수를 **노드 수에 비례**하여 결정

```
Cassandra 방식: 노드당 256개 파티션 (기본)

노드 추가 시:
→ 기존 파티션들을 무작위로 분할
→ 각 분할 파티션의 절반을 새 노드에 할당
```

**장점** → 노드 크기(데이터양/파티션)가 비교적 일정하게 유지
**단점** → 무작위 분할로 파티션 크기 불균등 발생 가능

> 실제 사용 예: **Cassandra**

---

### 자동 vs 수동 재균형화

```
자동 재균형화                수동 재균형화
─────────────────────        ─────────────────────
편리함 (운영 부담 감소)       예측 가능성 높음

위험 요소:                   장점:
- 노드 과부하 판단 오류 시    - 예상치 못한 데이터 이동 방지
  불필요한 대규모 이동        - 운영자가 명시적 제어 가능
- 연쇄 장애 가능성
```

> **Couchbase, Riak, Voldemort** → 자동 생성, **수동** 최종 적용

## 5. 파티션 요청 라우팅

### 클라이언트는 어떤 파티션/노드로 요청해야 하는가?

**서비스 탐색 (Service Discovery)** 문제

---

### 세 가지 라우팅 방식

#### 방식 1: 클라이언트 → 아무 노드 → 포워딩

```
클라이언트 ──→ 임의 노드
               ↓ (내 파티션이 아니면)
               → 올바른 노드로 포워딩
               ← 응답 반환
```

> 예: **Cassandra**, **Riak** (gossip 프로토콜로 클러스터 상태 전파)

---

#### 방식 2: 클라이언트 → 라우팅 계층 → 노드

```
클라이언트 → [라우팅 계층] → 올바른 노드
                ↑
         파티션 정보 보유
```

> 예: **ZooKeeper** 기반 라우팅 (HBase, Kafka, SolrCloud, Helix)

---

#### 방식 3: 클라이언트가 직접 파티션 인식

```
클라이언트 (파티션 매핑 정보 보유)
    ↓ 직접 올바른 노드에 접속
  노드
```

> 예: **Cassandra**, **Riak** (클라이언트 라이브러리 수준)

---

### ZooKeeper를 이용한 파티션 추적

```
ZooKeeper (분산 코디네이션 서비스)
    ├── 파티션 ↔ 노드 매핑 정보 유지
    ├── 노드 변경 시 라우팅 계층/클라이언트에 알림
    └── 라우팅 계층이 구독(subscribe)하여 최신 상태 유지
```

**ZooKeeper를 사용하는 시스템들**
- HBase, Kafka, SolrCloud, Helix (LinkedIn 개발)

---

### 병렬 쿼리 실행 (MPP)

대용량 분석 쿼리는 **여러 파티션을 동시에 쿼리**

```
MPP(Massively Parallel Processing) 쿼리 옵티마이저:
  → 복잡한 쿼리를 실행 단계로 분해
  → 각 단계를 다수의 파티션에서 병렬 실행
  → 결과 집계 후 반환
```

## 6. 정리 및 핵심 요약

### 파티셔닝 전략 비교

| 전략 | 특징 | 범위 쿼리 | 핫스팟 위험 | 사용 예 |
|------|------|-----------|------------|---------|
| **키 범위** | 정렬된 키 구간 | 가능 | 있음 | HBase, Bigtable |
| **키 해시** | 균등 분산 | 불가 | 낮음 | Cassandra, MongoDB |

### 보조 인덱스 전략 비교

| 전략 | 쓰기 | 읽기 | 사용 예 |
|------|------|------|---------|
| **문서 기반 (로컬)** | 빠름 | 느림 (scatter-gather) | MongoDB, Elasticsearch |
| **용어 기반 (글로벌)** | 느림 | 빠름 | DynamoDB GSI |

### 재균형화 전략 비교

| 전략 | 파티션 수 | 유연성 | 사용 예 |
|------|----------|--------|---------|
| **고정 파티션** | 고정 (큰 값) | 낮음 | Riak, Elasticsearch |
| **동적 파티셔닝** | 자동 조정 | 높음 | HBase, MongoDB |
| **노드 비례** | 노드 수 비례 | 중간 | Cassandra |

---

### 핵심 키워드

```
파티셔닝 = 대규모 데이터셋을 여러 노드에 분산
핫스팟   = 특정 파티션에 부하 집중 현상
샤딩     = 파티셔닝의 다른 이름
재균형화 = 노드 추가/제거 시 파티션 재배분
라우팅   = 요청을 올바른 파티션으로 전달하는 방법
```

---

### 5장 복제 vs 6장 파티셔닝

```
복제 (Chapter 5)        파티셔닝 (Chapter 6)
────────────────        ────────────────────
같은 데이터의           다른 데이터를
여러 복사본 저장        다른 노드에 저장

목적: 내결함성,         목적: 확장성,
      고가용성               부하 분산
```

> **두 기법은 보완적**: 대부분의 분산 DB는 파티셔닝 + 복제를 함께 사용
