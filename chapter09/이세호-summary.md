## 서론

1. 도입: 두 개의 시계와 분산 시스템의 딜레마
   프레데릭 브룩스의 격언("시계가 하나면 시간을 알 수 있지만, 두 개면 확신할 수 없다")으로 시작합니다. 이는 분산 시스템에서 복제본(Replica)들 사이의 데이터 불일치 문제를 관통하는 비유입니다.

결함 허용(Fault Tolerance): 시스템의 일부가 고장 나도 작동하려면 **복제(Replication)**가 필수입니다.

문제점: 데이터가 여러 곳에 있으면 복제본 간의 '불일치'가 발생할 위험이 커집니다. (예: 읽기 요청 시 오래된 데이터를 반환하거나, 동시에 여러 곳에서 쓰기가 발생하여 충돌함)

2. 일관성을 대하는 두 가지 철학
   시스템 설계자는 복제본 간의 차이를 어떻게 다룰지 결정해야 합니다.

① 최종 일관성 (Eventual Consistency)

특징: 복제 중이라는 사실을 애플리케이션 개발자에게 노출합니다. 즉, 일시적인 불일치를 허용합니다.

개발자의 역할: 일어날 수 있는 충돌이나 오래된 데이터를 개발자가 직접 처리해야 합니다.

활용: 다중 리더(Multi-leader), 리더 없는(Leaderless) 복제 방식에서 주로 사용됩니다.

장점: 오프라인 상태에서도 작업이 가능하며(로컬 우선 소프트웨어), 가용성이 높습니다.

② 강한 일관성 (Strong Consistency)

특징: 시스템이 마치 단일 노드인 것처럼 동작하도록 보장합니다.

개발자의 역할: 내부의 복제 메커니즘을 신경 쓸 필요가 없어 개발이 훨씬 단순해집니다.

단점: 성능 비용(지연 시간)이 발생하며, 특정 결함 상황에서 시스템 전체가 중단될 수 있습니다(가용성 저하).

3. 9장에서 다룰 세 가지 핵심 영역
   이 장은 '강한 일관성'을 달성하기 위한 심층적인 여정을 안내합니다.

선형성 (Linearizability): \* "강한 일관성"이라는 모호한 용어 대신, 이를 엄격하게 정의한 개념인 '선형성'을 배웁니다. 시스템을 단일 복사본처럼 보이게 만드는 구체적인 규칙입니다.

ID와 타임스탬프 (IDs and Timestamps): \* 단순한 번호 매기기 같지만, 분산 시스템에서 사건의 순서를 정하는 것은 일관성 유지와 직결된 매우 복잡한 문제입니다.

합의 알고리즘 (Consensus Algorithms): \* 결함이 발생하더라도 여러 노드가 하나의 결정(예: 누가 리더인가, 어떤 트랜잭션을 승인할 것인가)에 도달하는 방법을 탐구합니다.

4. 이론의 중요성 (경고)
   저자는 이 분야가 **"구현하기 매우 까다롭기로 악명 높다"**고 경고합니다.

결함이 없을 때는 잘 작동하는 것처럼 보이지만, 아주 드문 결함 조합이나 메시지 순서 뒤바뀜이 발생하면 시스템이 완전히 무너질 수 있습니다.

따라서 직관에만 의존하지 말고, 수많은 시행착오 끝에 정립된 이론적 모델과 증명을 바탕으로 견고한 시스템을 구축해야 함을 강조합니다.

# I. Linearizability

## 1. 서문

1. 선형성(Linearizability)이란?
   선형성은 복제된 데이터베이스를 마치 **"하나의 일관된 노드"**인 것처럼 동작하게 만드는 개념입니다. 사용자는 뒤에서 여러 대의 서버가 데이터를 복제하고 있다는 사실을 잊고, 단일 노드를 사용하는 것처럼 단순하게 생각할 수 있습니다.

다른 이름: 원자적 일관성(Atomic Consistency), 강한 일관성(Strong Consistency), 즉각적 일관성(Immediate Consistency).

핵심 철학: "데이터의 복사본은 실제로는 여러 개지만, 외부에서는 단 하나의 복사본만 존재하는 것처럼 보여야 한다."

2. 선형성의 핵심: 최신성 보장 (Recency Guarantee)
   선형성의 가장 중요한 특징은 최신 데이터를 즉각적으로 확인할 수 있어야 한다는 점입니다.

어떤 클라이언트가 쓰기(Write) 작업을 성공적으로 완료하는 순간, 그 이후에 데이터를 읽는 모든 클라이언트는 반드시 방금 기록된 최신 값을 보아야 합니다.

오래된 캐시나 복제 지연(Replication Lag)이 발생한 복제본에서 '과거의 데이터(Stale data)'를 읽는 것을 허용하지 않습니다.

3. 선형성이 깨지는 사례 (비선형적 시스템)
   텍스트에서 앨리야(Aaliyah)와 브라이스(Bryce)의 스포츠 경기 결과 확인 예시를 통해 선형성 위반 사례를 설명합니다.

> 상황 발생:

> 쓰기 완료: 경기 결과가 데이터베이스에 기록되었습니다.

> 클라이언트 A(앨리야): 새로고침을 눌러 경기 결과를 확인하고 우승팀을 알게 됩니다.

> 현실 세계의 통신: 앨리야가 옆에 있는 브라이스에게 결과를 말해줍니다.

> 클라이언트 B(브라이스): 결과를 들은 후에 새로고침을 눌렀으나, 하필 지연이 발생한 복제본에 연결되어 '경기 중'이라는 과거 데이터를 보게 됩니다.

왜 선형성 위반인가?

브라이스는 앨리야가 결과를 확인한 시점 이후에 조회를 시작했습니다. 선형적인 시스템이라면 브라이스는 최소한 앨리야가 본 것만큼 최신의 결과를 보았어야 합니다. 하지만 시스템이 '단일 복사본'처럼 행동하지 못했기 때문에(서로 다른 복제본의 상태가 달랐기 때문에) 사용자에게 혼란을 준 것입니다.

4. 요약: 선형적 시스템의 약속
   모든 작업은 어느 한 시점에 원자적으로(한 번에) 일어나는 것처럼 보입니다.

일단 최신 값이 확인되면, 그 이후의 모든 읽기는 그 값 혹은 그보다 더 나중의 값을 반환해야 합니다. 뒤로 가기(시간 역행)는 허용되지 않습니다.

## 2. What Makes a System Linearizable? - 시스템을 선형적으로 만드는 조건

1. 시스템을 선형적으로 만드는 조건
   단순히 "최신 값을 읽는다"는 것 이상으로, 선형성은 작업들 사이의 시간적 선후 관계를 엄격하게 규정합니다.

동시성 작업의 허용: 쓰기가 진행 중일 때 읽기가 수행되면, 이전 값(0)이나 새 값(1) 중 무엇을 반환해도 괜찮습니다. (네트워크 지연 등으로 정확한 처리 시점을 알 수 없기 때문)

핵심 제약 - "한 번 읽히면 끝": 누군가 한 번이라도 새 값(1)을 읽었다면, 그 이후에 발생하는 모든 읽기는 (설령 쓰기 작업이 아직 완전히 끝나지 않았더라도) 반드시 새 값(1)을 반환해야 합니다. 값이 다시 과거(0)로 되돌아가는 '시간 역행'은 절대 허용되지 않습니다.

원자적 지점(Atomic Point): 시스템 내부적으로는 쓰기가 일어나는 어느 한 시점에 값이 '0'에서 '1'로 순식간에 변한다고 가정합니다.

2. 복잡한 연산: CAS (Compare-And-Set)
   단순 읽기/쓰기 외에도 선형적 시스템은 CAS와 같은 원자적 연산을 지원합니다.

CAS(x, old, new): "x의 현재 값이 old와 같다면 new로 바꿔라"는 명령입니다.

여러 클라이언트가 동시에 값을 바꾸려 할 때, 오직 하나의 클라이언트만 성공하고 나머지는 실패하게 함으로써 **경합 조건(Race Condition)**을 방지합니다.

3. 선형성(Linearizability) vs 직렬성(Serializability)
   가장 많은 혼란을 야기하는 두 개념의 차이점입니다.

| 구분                                          | 선형성 (Linearizability)                                                                                      | 직렬성 (Serializability)                                       |
| --------------------------------------------- | ------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------- |
| **대상**                                      | 개별 객체 (한 번에 하나의 로우/키)                                                                            | 트랜잭션 (여러 객체, 여러 로우)                                |
| **핵심 보장**                                 | 최신성(Recency) 보장.<br>방금 쓴 건 바로 읽혀야 함.                                                           | 격리성(Isolation) 보장.<br>실행 결과가 순차 실행과 같아야 함.  |
| **시간 개념**                                 | 실제 물리적 시간 순서가 중요함.                                                                               | 순서가 실제 시간과 달라도 논리적으로만 맞으면 됨.              |
| **한계**                                      | 여러 객체가 연루된 '쓰기 왜곡'은 못 막음.                                                                     | 과거 데이터를 읽는 '오래된 읽기(Stale Read)'가 허용될 수 있음. |
| **💡 엄격한 직렬성 (Strict Serializability)** | 두 가지를 모두 만족하는 시스템.<br>구글의 Spanner나 FoundationDB가 이를 제공. 구현 비용(성능 저하)이 매우 큼. |                                                                |

4. 왜 선형성이 중요한가?
   선형성은 분산 시스템을 **"이해하기 쉬운 시스템"**으로 만들어 줍니다.

사용자(개발자)는 복제 지연이나 데이터 충돌을 고민할 필요가 없습니다.

하지만 이를 달성하기 위해서는 노드 간의 빈번한 통신(합의)이 필요하므로 성능(응답 시간)과 가용성 면에서 큰 대가를 치러야 합니다.

## 3. Relying on Linearizability

1. 잠금(Locking)과 리더 선출

시스템에 리더가 단 한 명만 존재해야 하는 상황(Single-leader replication)에서 선형성은 생명줄과 같습니다.

- 분산 임차(Lease): 여러 노드 중 하나만 리더가 되기 위해 '임차권'을 획득하려고 할 때, 시스템은 선형적이어야 합니다.
- 왜 필요한가? 만약 선형성이 보장되지 않으면, 두 개의 노드가 동시에 "내가 리더다"라고 착각하는 스플릿 브레인(Split Brain) 현상이 발생하여 데이터가 오염될 수 있습니다.
- 도구: Apache ZooKeeper나 etcd 같은 코디네이션 서비스가 선형적인 연산을 통해 이를 해결합니다.

2. 제약 조건과 유일성 보장

데이터베이스에서 흔히 볼 수 있는 '중복 방지' 규칙들은 선형성에 의존합니다.

- 유일성 제약(Uniqueness Constraints): 사용자 아이디나 이메일이 중복되지 않게 가입시키는 작업입니다.
- 원자적 비교 및 설정(Atomic compare-and-set): "이 이름이 비어 있으면 내 이름으로 채워라"라는 명령이 선형적이어야만, 두 사람이 동시에 같은 아이디로 가입할 때 한 명에게 정확히 에러를 보낼 수 있습니다.
- 현실적 타협: 비행기 오버부킹처럼 상황에 따라 약간의 오차를 허용할 수 있는 경우(나중에 보상해주는 방식)에는 선형성이 꼭 필요하지 않을 수도 있지만, 엄격한 데이터 정합성이 필요한 은행 잔고나 재고 관리에는 필수입니다.

3. 채널 간 타이밍 의존성 (Race Conditions)

이 부분이 가장 흥미로운 지점인데, 시스템 내부가 아닌 외부 통신 채널이 섞일 때 발생하는 문제입니다.

- 비디오 트랜스코딩 예시: 1. 웹 서버가 비디오를 저장소에 올립니다. (채널 1) 2. 서버가 메시지 큐를 통해 변환 작업자에게 알립니다. (채널 2) 3. 작업자가 메시지를 받고 저장소에서 비디오를 찾습니다.
- 위험 요소: 만약 저장소가 선형적이지 않다면, 메시지는 도착했는데 저장소 내부의 복제가 늦어져 작업자가 비디오를 찾지 못하거나 옛날 버전을 보게 되는 **경합 상태(Race Condition)**가 발생합니다.
- 결론: 두 개 이상의 통신 경로가 있을 때, 데이터의 '최신성'이 보장되지 않으면 시스템 전체의 논리가 깨집니다

## 4. Implementing Linearizable Systems

1. 단일 리더 복제 (Single-Leader) - "잠재적 선형성"

모든 읽기와 쓰기를 리더(Leader)에서만 처리한다면 이론적으로 선형적입니다. 하지만 현실적인 문제가 있습니다.

- 한계: 특정 노드가 자신이 리더가 아님에도 리더인 줄 착각하고 요청을 처리하는 '망상적 리더(Delusional Leader)' 문제가 발생할 수 있습니다.
- 리스크: 비동기 복제 환경에서는 장애 복구(Failover) 시 데이터가 유실되어 선형성이 깨질 수 있습니다.

2. 합의 알고리즘 (Consensus Algorithms) - "선형적 구현의 정석"

ZooKeeper(Zab 알고리즘 사용)나 etcd(Raft 알고리즘 사용) 같은 시스템이 여기에 해당합니다.

- 강점: 스플릿 브레인(Split Brain) 현상을 방지하도록 정교하게 설계되어 있어, 선형적인 저장소를 안전하게 구현할 수 있습니다.
- 주의점: 합의 알고리즘을 쓴다고 무조건 선형적인 것은 아닙니다. 만약 리더인지 확인하지 않고 로컬 노드에서 바로 읽기를 허용하면 '오래된 데이터(Stale read)'가 나올 수 있습니다.

3. 다중 리더 및 리더리스 복제 - "선형성 보장 어려움"

이 방식들은 대개 성능과 가용성을 위해 선형성을 포기합니다.

- 다중 리더 (Multi-Leader): 여러 노드에서 동시에 쓰기가 발생하고 나중에 동기화하므로, 데이터 충돌이 필연적입니다. 따라서 선형적이지 않습니다.
- 리더리스 (Leaderless, Dynamo 스타일): \* **정족수(Quorum, w+r>n)**를 사용해도 엄격한 의미의 선형성을 보장하기 어렵습니다.
- LWW (Last Write Wins): 시계 동기화 오차(Clock Skew) 때문에 이벤트의 실제 순서를 보장할 수 없어 거의 확실하게 선형적이지 않습니다.

| 복제 방식     | 선형성 가능 여부 | 주요 걸림돌                                 |
| ------------- | ---------------- | ------------------------------------------- |
| 단일 리더     | 가능함 (조건부)  | 리더의 착각(망상), 장애 복구 시 데이터 유실 |
| 합의 알고리즘 | 매우 높음        | 리더 확인 없는 읽기 수행 시 문제 발생 가능  |
| 다중 리더     | 불가능           | 동시 쓰기 발생 및 비동기 복제로 인한 충돌   |
| 리더리스      | 거의 불가능      | 시계 오차(LWW), 정족수 보장의 한계          |

### 4-1. Linearizability and quorums

1. 정족수(w+r>n)가 선형성을 보장하지 못하는 이유

- 흔히 w+r>n 조건을 만족하면 항상 최신 데이터를 읽을 것이라 생각하지만, 네트워크 지연이 개입하면 이야기가 달라집니다.
- 상황: 복제본이 3개인 시스템에서 쓰기(w=3), 읽기(r=2)를 수행하는 중입니다.
- 문제 발생 (그림 10-6 시나리오): 1. 클라이언트 A가 읽기를 수행하여 복제본들로부터 {새 값(1), 옛 값(0)}을 받습니다. 결과적으로 새 값 1을 인식합니다.

2. 클라이언트 B가 A의 읽기가 끝난 후에 읽기를 시작합니다. 하지만 B가 접근한 정족수 노드들은 아직 복제가 완료되지 않아 {옛 값(0), 옛 값(0)}을 보냅니다. B는 0을 인식합니다.

- 결과: 시간상 나중에 일어난 읽기(B)가 이전 읽기(A)보다 더 과거의 데이터를 반환하게 되며, 이는 선형성 위반입니다.

2. 해결 방법과 그 대가

- 리더리스 시스템에서도 선형성을 강제할 방법은 있지만, 성능 손실이 매우 큽니다.
- 동기적 읽기 수리(Synchronous Read Repair): 읽기 과정에서 최신 값을 확인했다면, 그 결과를 클라이언트에게 돌려주기 전에 모든 노드에 최신 값을 즉시 업데이트해야 합니다. (Cassandra가 이 방식을 사용하지만, 시계 오차 문제로 완벽한 선형성은 어렵습니다.)
- 쓰기 전 읽기: 쓰기 수행 전, 정족수 노드에서 가장 최신 타임스탬프를 먼저 확인하여 새 데이터가 확실히 더 큰 값을 갖도록 보장해야 합니다.
- Riak의 사례: Riak은 성능 저하를 우려하여 이러한 동기적 수리를 수행하지 않으므로 선형적이지 않습니다.

3. 결정적인 한계: CAS 연산의 불가능
   단순한 읽기/쓰기는 위와 같은 비용을 치르고 선형적으로 만들 수 있을지 몰라도, CAS(Compare-and-Set) 같은 복잡한 연산은 불가능합니다.

- 이유: CAS는 "값이 X일 때만 Y로 바꿔라"라는 원자적 합의가 필요합니다.
- 결론: 리더리스 시스템에서 엄격한 CAS를 구현하려면 결국 **합의 알고리즘(Consensus Algorithm)**이 필요하며, 이는 리더리스의 본래 설계 철학에서 벗어나게 됩니다.

### 5. The Cost of Linearizability

1. 선형성 vs 가용성 (네트워크 분할 상황)

- 네트워크에 장애가 발생하여 노드 간 연결이 끊겼을 때(Network Partition), 시스템은 반드시 선택을 해야 합니다.
- 선형성 선택 (CP): 데이터의 일관성이 중요하다면, 리더와 연결되지 않은 노드는 요청을 거부하거나 기다려야 합니다. 즉, 가용성을 포기합니다.
- 가용성 선택 (AP): 서비스가 계속 유지되는 것이 중요하다면, 각 노드는 최신 상태가 아닐지라도 자신이 가진 데이터를 반환합니다. 즉, 선형성을 포기합니다.

2. CAP 정리의 오해와 진실

- 저자는 CAP 정리가 역사적으로는 의미가 있었지만, 실제 시스템 설계에는 오히려 혼란을 준다고 지적합니다.
- "3개 중 2개 선택"은 틀린 말: 네트워크 장애(P)는 우리가 선택할 수 있는 옵션이 아니라 발생하는 '사고'입니다. 따라서 평상시에는 일관성(C)과 가용성(A)을 모두 가질 수 있지만, 장애가 났을 때만 둘 중 하나를 선택하는 것입니다.
- CAP의 한계: '가용성'에 대한 정의가 실제 서비스 운영에서 말하는 가용성과 다르고, 네트워크 지연이나 노드 다운 같은 다양한 장애 상황을 고려하지 못합니다.
- PACELC 원칙: CAP를 보완한 개념입니다.
  - 장애 시(P): 가용성(A)과 일관성(C) 중 선택.
  - 정상 시(E): 낮은 지연 시간(L)과 일관성(C) 중 선택.

3. 선형성과 성능 (Latency)

- 놀랍게도 선형성을 포기하는 가장 큰 이유는 '장애 대비'보다 '성능' 때문인 경우가 많습니다.
- CPU 사례: 현대의 멀티코어 CPU조차 선형적이지 않습니다. 각 코어마다 **캐시(Cache)**를 가지고 있기 때문인데, 만약 선형성을 강제하면 메모리 접근 속도가 엄청나게 느려져서 컴퓨터 성능을 제대로 낼 수 없습니다.
- 네트워크의 한계: 선형성을 보장하려면 읽기/쓰기 응답 시간이 네트워크의 불확실한 지연 시간에 비례해서 늘어날 수밖에 없습니다. (Attiya와 Welch의 증명)

4. 최종 요약

   | 구분      | 선형성 보장 (CP)                        | 선형성 포기 (AP/Latency 중심)                 |
   | --------- | --------------------------------------- | --------------------------------------------- |
   | 장점      | 데이터가 항상 최신이며 일관됨           | 장애에도 서비스 유지, 매우 빠른 응답 속도     |
   | 단점      | 네트워크 장애 시 서비스 중단, 느린 속도 | 데이터가 꼬이거나 옛날 데이터를 볼 수 있음    |
   | 대표 예시 | 금융 결제, 리더 선출, 유일성 검사       | 소셜 미디어 피드, 멀티코어 CPU 캐시, 카산드라 |

# II. ID Generators and Logical Clocks

## 1. 서문

1. 단일 노드 ID 생성기: 선형성의 표본

- 가장 단순한 방법은 서버 한 대가 번호표를 나눠주는 것입니다.
- 장점: 선형적입니다. A라는 메시지가 먼저 등록 완료되었다면, 그 뒤에 등록되는 B는 반드시 더 큰 ID를 갖습니다. 즉, ID 순서가 곧 시간 순서입니다.
- 단점 (실무에서 못 쓰는 이유):
  - 단일 장애점(SPOF): 그 서버가 죽으면 시스템 전체에서 ID 발급이 중단됩니다.
  - 지연 시간: 지구 반대편에서도 ID 하나 받으려고 멀리 있는 서버까지 다녀와야 합니다.
  - 병목 현상: 초당 수만 건의 쓰기가 발생하면 서버 한 대가 감당하기 힘듭니다.

2. 분산 ID 생성기: 선형성을 포기한 대안들

- 실무에서는 성능과 가용성을 위해 선형성을 조금씩 희생한 방식들을 사용합니다.

3. 왜 타임스탬프 기반 ID는 완전히 선형적이지 않을까?

| 방식            | 설명                                  | 선형성/순서 보장 여부                          |
| --------------- | ------------------------------------- | ---------------------------------------------- |
| 샤딩 (Sharding) | 짝수/홀수 등 다수 서버가 각자 ID 발급 | X (서로 다른 서버에서 순서가 뒤집힐 수 있음)   |
| ID 블록 할당    | 각 노드가 미리 할당받은 범위에서 발급 | X (나중에 할당받은 노드가 먼저 쓸 수 있음)     |
| 랜덤 UUID       | 무작위로 긴 문자열 ID 생성            | X (순서 개념 자체 없음)                        |
| 타임스탬프 기반 | '시간+서버ID+순번' 조합(Snowflake 등) | △ (거의 맞지만, 서버 시계 오차로 꼬일 수 있음) |

- Twitter Snowflake, UUID v7 같이 시간 기반 ID는 얼핏 정렬이 맞아보이지만, 분산 시스템 특유의 **시계 오차(Clock Skew)** 때문에 순서가 엉킬 수 있습니다.
- 시나리오: 서버 A의 시계가 서버 B보다 1초 빠릅니다.
- 결과: 서버 A에서 1초 전에 생성된 ID가 서버 B에서 방금 생성된 ID보다 더 큰 값을 가질 수 있습니다. 논리적으로는 나중에 일어난 사건이 더 작은 ID를 갖게 되는 **"시간 역전"**이 발생하죠.

4. 논리적 시계(Logical Clocks)의 등장

- 텍스트 마지막에 언급된 내용입니다. 실제 '물리적 시간'에 의존하면 선형성을 보장하기 위해 원자시계 같은 비싼 장비가 필요하니, 대신 이벤트 간의 인과관계만 따지는 **논리적 시계(Lamport Clock, Vector Clock)**를 사용하자는 아이디어입니다.

## 2. Logical Clocks

1. 램포트 타임스탬프 (Lamport Timestamps)

- 가장 기초적인 논리적 시계로, 1978년 레슬리 램포트가 제안했습니다.
- 구조: (카운터, 노드 ID)의 쌍으로 이루어집니다. 노드 ID를 포함하여 고유성을 보장합니다.
- 작동 원리:
  1.  각 노드는 자신의 카운터를 관리하며, 연산마다 1씩 증가시킵니다.
  2.  메시지를 보낼 때 카운터를 실어 보냅니다.
  3.  메시지를 받은 노드는 자신의 카운터와 받은 카운터 중 큰 값을 선택한 뒤 1을 더해 자신의 카운터를 갱신합니다.
- 특징: **인과성(Causality)**을 보존하는 전체 순서(Total Ordering)를 제공합니다. 즉, A가 B보다 먼저 일어났다면 항상 A의 ID < B의 ID입니다.

2. 하이브리드 논리 시계 (Hybrid Logical Clocks, HLC)

- 물리적 시계의 장점(실제 시간 반영)과 램포트 시계의 장점(순서 보장)을 합친 방식입니다.
- 배경: 램포트 시계는 "오늘 오후 3시에 달린 댓글" 같은 시간 기반 조회가 불가능합니다.
- 특징: \* 물리적 시간을 기반으로 하되, NTP 등으로 인해 시계가 뒤로 가거나 멈춰도 램포트 알고리즘처럼 **단조 증가(Monotonic)**하도록 강제합니다.
- CockroachDB 같은 현대 분산 DB에서 실제 사용됩니다.
- 특수 하드웨어(원자시계 등) 없이도 대략적으로 동기화된 시계만 있으면 작동합니다.

3. 벡터 시계 (Vector Clocks)와의 차이점

- 이 부분이 실무적인 설계에서 매우 중요합니다.

| 구분        | 램포트 / 하이브리드 시계                       | 벡터 시계 (Vector Clocks)                     |
| ----------- | ---------------------------------------------- | --------------------------------------------- |
| 순서        | 모든 이벤트에 순서를 매김 (전체 순서)          | 선후 관계가 있는 것만 매김 (부분 순서)        |
| 병행성 감지 | 불가능 (동시에 일어난 일도 억지로 순서를 정함) | 가능 (두 사건이 동시에 일어났음을 알 수 있음) |
| 크기        | 작고 고정됨 (몇 바이트)                        | 큼 (노드 개수만큼 비례해서 커짐)              |
| 용도        | 유일 ID 생성, 스냅샷 격리 구현                 | 데이터 충돌 감지 및 해결                      |

4. 💡 중요한 결론: 논리적 시계 ≠ 선형성

- 텍스트에서 가장 강조하는 부분 중 하나입니다. "램포트 시계는 선형성을 보장하지 않는다."
- 이유: 논리적 시계는 "A가 B보다 먼저 일어났다"는 **순서(Causality)**는 알려주지만, "지금 이 데이터가 가장 **최신(Recency)**인가?"는 알려주지 못합니다.
- 단순히 ID의 크기만 비교해서는 네트워크 너머에 이보다 더 큰 ID를 가진 '더 최신 데이터'가 있는지 알 방법이 없기 때문입니다.

# III. Consensus

## 1. 서문

1. 모든 길은 '합의'로 통한다

- 우리가 앞서 "어렵다"고 했던 작업들은 사실 모두 합의 문제의 다른 이름들입니다.
- 리더 선출: "누가 리더인가?"에 대해 모든 노드가 동의해야 함. (스플릿 브레인 방지)
- 원자적 CAS: "이 값이 현재 X인가, 그래서 Y로 바꿀 수 있는가?"에 대해 모든 노드가 동의해야 함.
- 유일성 제약: "이 아이디가 이미 존재하는가?"에 대해 모든 노드가 동의해야 함.

2. 합의 알고리즘의 종류

- 합의는 너무나 중요해서 이미 검증된 강력한 알고리즘들이 존재합니다.
- 전통의 강자: Paxos (가장 유명하지만 이해하기 어렵기로 악명 높음), Viewstamped Replication.
- 현대의 대세: Raft(이해하기 쉬워 etcd 등에서 사용), Zab(ZooKeeper에서 사용).
- 비잔틴 합의 (BFT): 노드가 단순히 죽는 게 아니라, 거짓말을 하거나 악의적으로 행동하는 상황까지 대비하는 알고리즘 (주로 블록체인에서 사용).

3. FLP 불가능성 정리 (The FLP Result)

- 공부하시다 보면 반드시 마주치게 될 유명한 이론적 한계입니다.
- 내용: "노드가 하나라도 죽을 가능성이 있는 비동기 시스템에서는, 합의를 항상 완벽하게 끝낼 수 있는 알고리즘은 존재하지 않는다."
- 왜 그런가? 비동기 시스템에서는 노드가 '죽은 건지' 아니면 단순히 '네트워크가 느린 건지' 구분할 수 없기 때문에, 무한정 기다리다가 합의가 끝나지 않을 수 있다는 뜻입니다.
- 실무에서의 해법: 다행히 FLP 정리는 '시계'나 '타임아웃'을 전혀 쓰지 못한다는 아주 엄격한 가정하에서의 결론입니다. 실무에서는 타임아웃이나 무작위성을 도입해서 이 한계를 극복하고 합의를 잘 해내고 있습니다.

## 2. The Many Faces of Consensus

1. 합의의 4가지 필수 조건 (정의)

- 합의 알고리즘이 "제대로 작동한다"고 말하려면 다음 네 가지를 반드시 만족해야 합니다.
- 균등 합의 (Uniform Agreement): 어떤 두 노드도 서로 다른 결정을 내릴 수 없음.
- 무결성 (Integrity): 한 번 결정하면 번복할 수 없음.
- 유효성 (Validity): 결정된 값은 반드시 누군가가 제안한 값이어야 함. (무조건 null로 결정하는 편법 방지)
- 종결성 (Termination): 죽지 않은 노드는 결국 어떤 값이라도 결정해야 함. (결함 허용의 핵심)

2. 합의와 동등한 문제들 (The Equivalence)

- 저자는 다음 문제들 중 하나만 풀 수 있으면 나머지 모두를 풀 수 있다고 설명합니다.
- 원자적 CAS (Compare-and-Set): \* "현재 값이 A면 B로 바꿔라"는 연산은 합의와 같습니다.
- 여러 노드가 동시에 CAS를 시도할 때, 누가 성공할지 합의하는 것이 곧 합의 알고리즘이기 때문입니다.
- 공유 로그 (Total Order Broadcast):
  - 메시지를 모든 노드에 동일한 순서로 배달하는 것입니다.
  - "다음에 올 로그 항목이 무엇인가?"라는 질문에 대해 매번 합의를 시도하면 공유 로그가 만들어집니다.
- 원자적 커밋 (Atomic Commitment):
  - 2단계 커밋(2PC)처럼 모든 참여자가 YES/NO를 결정하는 것입니다.
  - 합의와 비슷하지만, **"한 명이라도 NO라고 하면 무조건 중단(Abort)"**해야 한다는 더 까다로운 유효성 조건이 붙습니다.

3. 합의 수(Consensus Number)의 개념

- 모든 연산이 똑같이 강력한 것은 아닙니다. 알고리즘이 몇 개의 노드 사이에서 합의를 끌어낼 수 있는지를 나타냅니다.
- Fetch-and-add: 합의 수가 2입니다. 두 노드 사이의 합의는 가능하지만, 그 이상의 노드에서는 누가 이겼는지 서로 알 방법이 없어 종결성을 만족하지 못할 수 있습니다.
- CAS / 공유 로그: 합의 수가 **∞(무한대)**입니다. 아무리 노드가 많아도 결함을 허용하며 합의를 이끌어낼 수 있습니다.

4. 실무적 트레이드오프: 결함 허용의 한계

- 합의 알고리즘은 마법이 아니며 물리적인 한계가 있습니다.
- 과반수 쿼럼 (Majority Quorum): 종결성(Termination)을 보장하려면 **전체 노드의 절반 이상(f+1)**이 살아있어야 합니다. 5대 중 3대가 죽으면 시스템은 안전을 위해 동작을 멈춥니다.
- 안전성 vs 활성 (Safety vs Liveness): 대규모 장애가 발생하면 시스템이 멈출 수는 있지만(활성 상실), 잘못된 결정을 내려 데이터가 꼬이는 일(안전성 위반)은 절대 발생하지 않도록 설계됩니다.

💡 최종 요약

"합의는 단순히 '예/아니오'를 정하는 것을 넘어, 분산 시스템의 순서를 정하고(로그), 상태를 동기화하며(CAS), 거래를 확정 짓는(커밋) 모든 행위의 근간입니다."

## 3. Consensus in Practice

1. 실무의 주인공: 공유 로그 (Shared Logs)

- 이론적으로는 CAS나 단일 값 합의도 가능하지만, 실무에서 쓰이는 대부분의 합의 시스템(Raft, Zab, Multi-Paxos)은 공유 로그 형태를 취합니다.
- 상태 머신 복제 (State Machine Replication): 모든 복제본이 로그에 기록된 명령을 동일한 순서로 실행하면, 결과적으로 모두 동일한 상태에 도달한다는 원리입니다.
- 활용도: \* 데이터베이스 복제: 쓰기 작업을 로그로 기록.
- 직렬성 트랜잭션: 로그 순서대로 트랜잭션을 실행하여 완벽한 순서 보장.
- 펜싱 토큰 (Fencing Tokens): ZooKeeper의 zxid처럼 로그의 순번을 이용해 오래된 리더의 요청을 거부함.

2. 리더 선출의 딜레마와 에포크(Epoch)

- "리더를 뽑으려면 합의가 필요하고, 합의를 하려면 리더가 필요하다"는 순환 논리를 에포크 개념으로 해결합니다.
- 에포크 번호 (Ballot/View/Term number): 리더가 바뀔 때마다 증가하는 번호입니다.
- 리더의 자격 검증:
  - 투표: 노드는 자신이 아는 가장 높은 에포크 번호보다 낮은 요청은 거절합니다.
  - 쿼럼 확인: 리더는 로그를 추가하기 전, 여전히 자신이 최신 에포크의 리더인지 과반수(Quorum)의 확인을 받아야 합니다.
  - 안전 장치: 새로운 리더는 반드시 이전 리더가 커밋한 모든 로그를 가지고 있어야 합니다. (Raft는 로그가 가장 최신인 노드만 리더가 될 수 있게 제한하고, Paxos는 리더가 된 후 로그를 맞춥니다.)

3. 합의 알고리즘의 빛과 그림자

- 합의 알고리즘은 분산 시스템의 "올바른 리더 복제"를 가능하게 하지만, 공짜는 아닙니다.

- 장점 (Pros) 단점 (Cons)
- 자동 장애 복구: 사람의 개입 없이 리더 교체 가능. 엄격한 과반수 요구: 5대 중 3대가 죽으면 시스템 전체 중단.
- 스플릿 브레인 방지: 두 리더가 동시에 활약하는 사고 차단. 성능 저하: 노드를 늘릴수록 통신 비용이 커져 오히려 느려짐.
- 데이터 유실 없음: 커밋된 데이터는 절대 사라지지 않음. 네트워크 민감도: 타임아웃 설정이 잘못되면 무한 리더 선출 루프에 빠짐.

💡 최종 결론

실무에서 합의 알고리즘은 **"성능을 조금 내주더라도, 절대 데이터가 깨지면 안 되는 핵심 장치"**로 사용됩니다. Kafka나 etcd 같은 도구들이 이 까다로운 알고리즘들을 미리 구현해 두었기 때문에, 우리는 이를 믿고 고수준의 분산 시스템을 설계할 수 있는 것이죠.

## 4. Coordination Services

1. 코디네이션 서비스란?

- 겉보기에는 키-값(Key-Value) 저장소처럼 보이지만, 일반 DB와는 목적이 완전히 다릅니다.
- 데이터의 양: 아주 적은 양의 데이터(설정 값, 리더 정보 등)만 메모리에 보관합니다.
- 목적: 대규모 분산 시스템(Kubernetes, Spark 등)의 노드들이 서로 발을 맞추게(Coordinate) 돕는 보조 역할을 합니다.
- 기반: 앞서 배운 **합의 알고리즘(Raft, Zab 등)**을 내장하여, 어떤 장애 상황에서도 데이터의 일관성을 잃지 않습니다.

2. 4대 핵심 기능 (분산 시스템의 필수 도구)

- 코디네이션 서비스가 제공하는 이 기능들은 분산 시스템 구축을 훨씬 쉽게 만들어줍니다.
- 잠금(Lock) 및 임차(Lease): \* 원자적 CAS 연산을 활용해 "이 작업은 나만 하겠다"는 권한을 얻습니다. 여러 노드가 경쟁해도 합의 알고리즘 덕분에 딱 한 명만 성공합니다.
- 펜싱(Fencing) 지원: \* 노드가 일시적으로 멈췄다 살아났을 때(좀비 노드), 자신이 옛날 리더임을 깨닫게 해주는 **단조 증가 ID(zxid 등)**를 생성해 줍니다.
- 장애 감지 (Failure Detection): \* **하트비트(Heartbeat)**를 주고받으며 노드의 생존을 확인합니다. 특정 시간 동안 소식이 없으면 그 노드가 가진 권한(임차권)을 자동으로 회수합니다.
- 변경 알림 (Change Notifications/Watch): \* 데이터가 바뀌면 연결된 클라이언트에게 즉시 알려줍니다. 클라이언트는 계속 DB를 조회(Polling)할 필요 없이 가만히 기다리면 됩니다.

3. 실무에서의 활용 사례
   ① 작업 할당 및 리더 선출

수천 대의 서버가 있을 때 "누가 대장(Leader)인가?" 혹은 "어떤 서버가 7번 데이터를 담당하나?"라는 정보를 관리합니다.

노드가 새로 들어오거나 죽으면, 코디네이션 서비스가 이를 감지해 **작업을 다시 분배(Rebalancing)**하도록 돕습니다.

② 서비스 디스커버리 (Service Discovery)

"지금 접속 가능한 API 서버의 IP 주소는 무엇인가?"를 관리합니다.

새로운 서버가 뜨면 자기 IP를 등록하고, 클라이언트는 변경 알림을 통해 최신 서버 목록을 실시간으로 업데이트받습니다.

4. 장단점 및 주의사항

   | 장점                                                           | 단점 및 한계                                                          |
   | -------------------------------------------------------------- | --------------------------------------------------------------------- |
   | **높은 신뢰성**: 과반수가 살아있으면 절대 데이터가 깨지지 않음 | **쓰기 성능 한계**: 합의 과정 때문에 쓰기 속도가 일반 DB보다 느림     |
   | **자동 장애 복구**: 사람 개입 없이 시스템이 스스로 복구됨      | **운영의 복잡성**: 합의 알고리즘 특성상 타임아웃 튜닝 등이 까다로움   |
   | **효율성**: 수천 대 노드 관리를 3~5대 서버에 '외주' 줄 수 있음 | **읽기 최신성**: '옵저버(Observer)' 노드를 쓰면 선형성이 깨질 수 있음 |

💡 최종 요약

"코디네이션 서비스는 분산 시스템의 '경찰관'과 같습니다. 교통 정리(잠금)를 하고, 사고 여부를 감지하며(장애 감지), 바뀐 규칙을 확성기로 알립니다(알림). 이 경찰관이 있기에 수천 대의 서버가 하나의 거대한 시스템처럼 움직일 수 있는 것입니다."

## IV. Summary

1. 선형성 (Linearizability): "단일 복사본의 환상"

- 가장 먼저 배운 개념이자 강한 일관성의 표준입니다.
- 핵심: 여러 대의 서버에 데이터가 흩어져 있어도, 외부에서 볼 때는 단 한 대의 서버가 원자적으로 처리하는 것처럼 보이게 하는 약속입니다.
- 용도: 최신 데이터 읽기, 중복 방지, 경합 상태(Race Condition) 해결.
- 대가: 네트워크 지연에 민감하며, 성능이 느려질 수 있습니다.

2. ID 생성과 논리적 시계: "순서의 문제"

- 선형성을 구현하기 위한 도구들을 살펴보았습니다.
- ID 생성기: 단일 노드는 선형적이지만 장애에 취약하고, 분산 생성기(Snowflake 등)는 빠르지만 선형성을 보장하기 어렵습니다.
- 논리적 시계 (Lamport/HLC): 실제 시간이 아닌 이벤트의 인과관계를 따집니다. "A가 B의 원인이다"는 순서는 잡을 수 있지만, "누가 우주에서 제일 최신인가(선형성)"는 말해주지 못합니다.

3. 합의 (Consensus): "분산 시스템의 정답"

- 결함 허용(Fault Tolerance)과 선형성을 동시에 잡기 위한 최종 병기입니다.
- 원리: 여러 노드가 단일한 작업 시퀀스에 동의하게 만듭니다. 덕분에 일부 노드가 죽어도 시스템은 마치 한 대의 완벽한 서버처럼 동작합니다.
- 동등한 문제들: 아래의 문제 중 하나만 풀면 나머지도 다 풀립니다.
- 원자적 CAS / 락과 임차 / 유일성 제약 / 공유 로그 / 원자적 커밋.

4. 실무의 대안: 코디네이션 서비스

- 이 복잡한 이론을 직접 구현하는 것은 너무 위험하기 때문에, 검증된 도구를 사용합니다.
- 도구: ZooKeeper, etcd.
- 역할: 리더 선출, 서비스 디스커버리, 설정 관리 등 **"작지만 절대 깨지면 안 되는 데이터"**를 관리하는 '분산 시스템의 뇌' 역할을 수행합니다.

5. 마지막 선택: 강한 일관성 vs 가용성 (CAP)

- 모든 시스템에 합의와 선형성이 필요한 것은 아닙니다.
- 강한 일관성이 필요할 때: 금융 거래, 리더 선출, 결제 시스템 → 합의 기반 시스템(CP) 사용.
- 가용성과 성능이 우선일 때: SNS 피드, 로그 수집, 대규모 서비스 → 리더리스/다중 리더(AP/최종 일관성) 사용.

💡 학습 마무리 한 마디

사용자님은 이제 "선형성이 왜 비싼지", "ID 생성기가 왜 순서를 뒤바꿀 수 있는지", 그리고 **"왜 중요한 결정에는 합의 알고리즘이 필요한지"**를 이해하는 수준에 도달하셨습니다. 이 지식은 단순히 이론에 그치지 않고, 나중에 대규모 아키텍처를 설계할 때 "여기서는 정합성을 위해 성능을 조금 포기하자" 혹은 **"여기서는 최종 일관성으로 충분하다"**는 현명한 판단을 내리는 밑거름이 될 것입니다.
