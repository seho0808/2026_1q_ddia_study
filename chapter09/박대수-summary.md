분산 시스템 하 불변의 진실: 시스템 내의 모든 것들이 잘못될 수 있다. 완벽한 보장이란 존재하지 않는다.

그렇기에 결함에 예민하게 반응하게 하거나(매우 쉬움 but, 상용 서비스에는 어울리지 않음), 혹은 결함에 둔감하게 반응하도록 해야 한다(매우 어려움, but 받아들여야만 함).

코드 문제가 아닌, 네트워크 등 외부 변수로 인해 발생하는 장애에 잘 견뎌내는 분산 시스템을 설계하는 방법에 대한 이야기

1. 일관성 보장
- 트레이드 오프 때문에 최종적 일관성을 추구하는 편
- 결론이 나기 전까지는 수 많은 동시성 이슈를 묵인하기 때문에 매우 약한 보장
- 쓰기 작업 중간에 읽기가 발생할 경우 어느 시점에 읽었느냐에 따라 정확하지 않은 데이터를 읽게될 수 있음


2. 선형성
- 최신성 보장
- 시스템에 데이터 복사본이 딱 하나인 것 처럼 행동하는 것
- 모든 연산이 어느 시점에 원자적으로 수행된 것처럼 보여줌
- 이를 위해 연산들은 정의된 순서대로 잘 실행되어야 함
- 레지스터가 갱신된 값을 반환한 이후의 요청은 항상 갱신된 값만을 반환해야 함(이는 아직 쓰기 작업이 진행 중이라 응답이 없는 상태여도 동일함)

2-1. 선형성을 추구하는 경우
- 잠금과 리더 선출
    - 단일 리더 복제
    - 모든 노드가 동시에 잠금을 요청하고, 획득에 성공한 노드가 리더가 됨
    - 코디네이터 사용

- 제약 조건과 유일성 보장
    - 데이터베이스에서 주로 사용
    - 딱 하나만 있어야 하는 정보(이메일 정보, 휴대전화 번호 등)를 누가 선점하느냐를 구분할 때 필요
    - compare and set
    - 항공기 좌석처럼 유일성 보장이 꼭 필요하지 않은 경우가 많으니 신중하게 써야 함

- 채널 간 타이밍 의존성
    - 하나의 작업을 위해 소통하는 채널이 두 개 이상일 때 발생하는 문제
    - 웹 서버가 원본 이미지를 받아 파일 저장소에 저장하고, 메시지 큐를 통해 이미지 크기를 수정해 함께 저장하는 경우 -> 웹 서버와 메시지 큐 둘 중 어느 작업이 먼저 끝나냐에 따라 이슈 발생 가능


2-2. 선형성 구현
    - 단일 리더 복제
        - 리더나 동기식으로 갱신된 팔로워에서 실행된 읽기는 선형적이 될 수 있음
        - 리더가 아닌 노드가 자신이 리더라고 착각할 수 있음
        - 네트워크 지연, 패킷 소실 등으로 일부 팔로워는 오랜 시간 갱신이 되지 않을 수 있는데, 이 팔로워를 읽은 경우 선형성이 깨질 수 있음
    
    - 합의 알고리즘
        - 선형적임
        - 스플릿 브레인과 복제본이 뒤쳐지는 문제를 막을 수단이 포함되어 있음

    - 다중 리더 복제와 리더 없는 복제는 비선형적이거나 그렇게 될 가능성이 높음
        - 얘네는 보통 선형성을 최대한 보장하기 위해 정족수 개념을 사용하려 하지만, 네트워크 지연 이슈로 인해 선형성이 깨질 가능성이 있음


2-3. 선형성의 비용
    - 만약 두 개의 데이터 센터가 있고, 각자가 클라이언트와 소통은 되지만, 데이터 센터 간 통신이 끊긴 상황 가정
    - 단일 리더 복제(선형성 중시): 리더가 있는 데이터 센터는 문제 없음, 리더가 없는 데이터 센터로 리디렉팅 되는 경우 장애 경험
    - 다중 리더 복제(선형성 희생): 각 데이터 센터에 리더가 있기 때문에 독자적으로 작업 가능, but 데이터 센터 간 통신이 복구된 후 발생한 동시성 이슈를 해결해야 함


2-4. CAP 정리
    - C(일관성), A(가용성), P(분단 내성): 네트워크 분단이 생겼을 때 일관성과 가용성 중 하나를 선택해야 함
    - 어떤 선형성 데이터베이스라도 복제 서버 간 네트워크 장애로 인해 발생하는 문제는 피할 수 없다.
    - 선형성을 요구하는 경우 일부 복제 서버가 (리더와 연결된) 다른 복제 서버와 연결이 끊기면, 일부 복제 서버는 연결이 끊긴 동안 요청을 처리할 수 없다
    - 선형성을 희생한 경우 위의 경우가 발생해도 가용한 상태를 유지한다. 다만 그 동작은 선형적이지 않기에 통신이 정상회된 이후 (어쩌면)고통스러운 합치 작업이 불가결하다.


2-5. 선형성과 네트워크 지연
    - 선형성은 웹 서비스들이 매우 중요하게 여기는 일관성, 정확성에 큰 역할을 하기 때문에 이론적으로 중요하게 다뤄짐
    - 하지만 실제로는 거의 사용되지 않음
    - 안정성을 위해 성능을 너무 크게 지불해야 함, 효율적이지 않음
    - 결국 선형성을 지키기 위해서는 작업의 어떤 시점에서는 모든 요청과 응답의 결과를 수렴해야 하는 시점이 발생하는 데, 이 때 네트워크 장애가 발생하면 비용이 무한대로 증가할 수 있음(응답이 안올테니까)

3. 순서화 보장
    - 분산 시스템의 가장 중요한 개념
    - 순서화, 선형성, 합의 사이에는 깊은 연결 관계가 있음

3-1. 순서화와 인과성
    - 순서화는 인과성을 보존하는 데 도움을 줌
    - 인과성은 이벤트에 순서를 부과함(원인 -> 결과)
    - 인과적으로 일관적(EX) 스냅숏 격리)

3-2. 인과적 순서가 전체 순서가 아님
    - 전체 순서(선형성): 어떤 두 요소를 비교할 수 있게 해줌, 항상 비교가 가능(EX) 자연수)
    - 부분 순서(인과성): 서로 포함되지 않는 두 집합 간의 우열 비교가 불가능 해서 집합 내의 우열 비교만 가능하고 전체 순서는 정립할 수 없는 경우
    - 콘서트 티켓 구매를 요청(A)하면 티켓이 구매되는 로직(B)이 있다 쳤을 때 A -> B 는 확실히 보장되지만, 만약 이를 관리하는 서버가 3개 일 때 1서버: A->B, 2서버: A->B, 3서버: A->B 간의 순서 우열은 비교할 수 없음, 그래서 A1->B1->A2->B2->A3->B3의 순서로 진행됐을 거다라는 확신을 줄 수 없음

3-3. 선형성은 인과적 일관성보다 강하다
    - 선형성은 인과성을 내포함
    - 선형적인 서비스는 인과성이 자동으로 유지됨
    - 다만 선형성은 비용이 크기 때문에 선호되지 않음 + 선형성을 포기하고도 인과성을 유지할 수 있음

3-4. 인과적 의존성 담기
    - 어떤 연산이 다른 연산보다 먼저 실행됐는지 알아야 함(부분 순서)
    - 정해진 연산 순서가 있는 경우, 모든 복제 서버가 해당 연산 순서를 존중하도록 강제해야 함
    - 중간에 갑자기 다른 순서의 연산이 끼어들어선 안됨
    - 만에 하나 응답이 누락된 연산이 있을 발견될 경우, 해당 연산의 응답이 올 때까지 기다리거나, 아니면 해당 연산만 다시 처리해서 응답을 받아낸 후 마저 진행해야 함

3-5. 일련번호 순서화
    - 매 요청마다 인과적 의존성을 모두 추적하는 것은 대단히 비효율적(오버헤드)
    - 일련번호, 타임스탬프(논리적 시계) -> 인과성에 일관적인 전체 순서 정렬 가능

3-6. 비인과적 일련번호 생성기
    - 인과성에 일관적이지 않은 방식: 
        - 각 노드 별 독립적으로 일련번호 집합 생성: 노드 당 초당 연산수가 다를 수 있음 [1, 3, 5] / [2, 4, 6, 8] 이 있다 할 때 1보다 4가 더 먼저 생성된 일련번호라고 확신할 수 없음
        - 타임스탬프(물리적 시계): 시계 스큐(서버 별 시계 왜곡)에 종속적임
        - 일련번호 블록 할당: A -> B 로 간다할 때 A는 1-1000(노드 1) 사이의 일련번호를 받고, B는 1001-2000(노드 2) 사이 일련번호를 받을 수 있음 

3-7. 램포트 타임스탬프
    - 타임스탬프(카운터, 노드 ID) 이렇게 세 가지의 식별자를 조합해 만든 타임스탬프
    - 모든 요청과 응답에 추적한(혹은 갱신한) 카운터의 최댓값을 포함시켜 별도의 작업 없이도 실시간으로 인과성을 추적함
    - 만약 2개의 노드 중 노드 1에서만 작업 카운트가 계속 쌓이다가 노드 2에 작업이 들어가면, 노드 1에서 쌓인 카운터와 자신의 카운터를 비교해 노드 1의 작업 결과가 인과적으로 최신 값임을 확인하고 이를 바탕으로 추가 작업을 이어감
    - 만약 두 개의 노드에 같은 카운터로 요청이 들어가버렸을 경우 타임스탬프 비교로 처리

3-8. 타임스탬프 순서화로 충분하지 않은 경우
    - 램포트 타임스탬프 방식은 사후 일관성 처리에 효과적(일단 요청을 처리한 후 램포트 타임스탬프를 비교해서 다시 정립하는 방식이니까)
    - 인과성을 '당장' 확인해야 하는 경우에는 쓸 수 없음 -> 연산 전체 순서가 계획된 것만으로는 충분하지 않음, 언제 그 순서가 (이론의 여지가 전혀 없이) 확정되어 불변의 상태가 되는지도 알아야 함

3-9. 전체 순서 브로드캐스트
    - 노드 사이에 메시지를 교환하는 프로토콜
    - 두 가지 속성을 항상 만족해야 함
        - 신뢰성 있는 전달: 어떤 메시지도 손실돼선 안됨, 메시지가 한 노드에 전달되면 모든 노드에도 전달돼야 함
        - 전체 순서가 정해진 전달: 메시지는 모든 노드에 같은 순서로 전달돼야 함

3-10. 전체 순서 브로드캐스트 사용
    - 주키퍼나 ETCD 같은 합의 서비스에서 전체 순서 브로드캐스트를 실제로 사용함 -> 전체 순서 브로드캐스트와 합의 사의의 강한 연관성
    - 데이터베이스 복제에 필요함 -> 상태 기계 복제
    - 직렬성 트랜잭션
    - 로그를 생성하고 읽는 것과 비슷함
    - 펜싱 토큰 구현에도 유용

3-11. 전체 순서 브로드캐스트를 이용해 선형성 저장소 구현
    - 전체 순서 브로드캐스트는 비동기 식 -> 메시지는 고정된 순서로 가지만, 언제 전달될 지는 미지수
    - 선형성은 최신성이 핵심인데, 어떻게 하지? -> compare-and-set
    - 메시지를 로그에 추가해 전체 노드에 브로드캐스트를 날린 후 돌아오는 응답으로 중복 여부 확인
    - 순차적 일관성, 타임라인 일관성으로도 불림
    - 선형적 쓰기는 보장되지만, 선형적 읽기는 보장되지 않음
        - 노드 1과 노드 2가 있을 때, 노드 2에 이미 ggangtong 유저가 있는 경우, 노드 1에서 ggangtong 유저를 쓰기할 때가 돼서야 ggangtong 유저가 이미 존재함을 알 수 있음, 즉 읽기로는 확인이 안될 수 있음

3-12. 선형성 저장소를 사용해 전체 순서 브로드캐스트 구현
    - increment-and-get이나 compare-and-set 연산으로 전체 순서 메시지에 일련번호를 부여해 관리하면 됨
    - 메시지 4와 메시지 6을 받았다면, 메시지 5가 안왔다는 뜻이니 기다렸다 받으면 됨
    - 선형성 레지스터와 전체 순서 브로드캐스트는 둘 다 합의와 동등하다

4. 분산 트랜잭션과 합의
    - 합의: 여러 노드들이 뭔가에 동의하게 만드는 것
    - 극장의 같은 좌석 예약과 같이 서로 공존할 수 없는 연산 중 어떤 것이 승자가 돼야 하는 지 결정하는 데 유용함
    - 중요한 경우
        - 리더 선출: 단일 리더 복제의 경우에 여러 노드 사이에 네트워크 결함이 발생할 경우 리더가 둘 이상 선출되는 경우가 발생할 수 있음
        - 원자적 커밋: 여러 노드나 파티션에 걸쳐 트랜잭션이 일어나는 경우 모든 노드가 트랜잭션 결과에 동의해야만 커밋이 되도록 해야 함

    - 분산 시스템에서는 노드는 언제나 죽을 가능성이 있기 때문에 항상 합의에 이를 수 있는 알고리즘은 없다 라는 합의 불가능성 논제가 있음

4-1. 원자적 커밋과 2단계 커밋(2PC)
    - 트랜잭션의 결과는 커밋 성공이나 어보트이다.
    - 원자성: 트랜잭션이 전부 커밋되거나 전부 어보트 되도록 보장하여 일부만 성공하는 엣지 케이스를 막아주는 개념

4-2. 단일 노드에서 분산 원자적 커밋으로
    - 단일 노드 트랜잭션: 디스크에 지속성 있게 쓰여지는 순서에 의존(디스크가 커밋 레코드 쓰기를 마치는 시점)
    - 다중 노드 트랜잭션: 다중 객체 트랜잭션, 용어 파티셔닝된 보조 색인 -> 원자성 보장 위반 가능성 높음
        - 어떤 노드는 제약 조건 위반/충돌 감지로 어보트 <> 다른 노드는 커밋 완료
        - 네트워크 장애로 일부 노드에서 커밋 요청 손실
        - 일부 노드에서 트랜잭션 커밋 전 서버 장애로 롤백됐지만 나머지 노드에선 커밋 완료
    - 트랜잭션 커밋은 되돌릴 수 없어야 함, 커밋된 이후부터는 다른 클라이언트에게 노출될 수 있기 때문

4-3. 2단계 커밋
    - 다수의 복제 노드와 하나의 코디네이터가 존재
    - 코디네이터가 각 노드의 데이터 쓰기를 지시한 이후, 모든 노드로부터 커밋 준비 완료 응답을 반음(1단계)
    - 모든 노드로부터 커밋 준비 완료 응답을 받은 후, 코디네이터는 모든 노드에 실제 커밋 명령을 내림(2단계)
    - 각 단계에서 한 노드라도 어보트를 내는 경우, 모든 노드의 작업을 어보트하고 롤백함

4-4. 코디네이터 장애
    - 2단계 커밋은 노드 장애에는 강하지만, 코디네이터 장애에는 속수무책임
    - 각 단계마다 노드들이 코디네이터에 응답과 요청을 보낼 텐데, 코디네이터가 이에 대해 반응하지 않으면 무제한으로 기다려야 함
    - 네트워크 장애 등 사유로 코디네이터에게 트랜잭션 작업 관련해서 추가 요청과 응답을 받지 못하고 있는 상황을 의심스러운/불확실한 트랜잭션 상태라고 함
    - 코디네이터가 죽는 걸 원천 봉쇄할 순 없으니 코디네이터의 모든 커밋, 어보트 요청 기록과 왜 그런 결정을 했는지를 남긴 로그를 항상 남겨야 함

4-5. 3단계 커밋
    - 2단계 커밋은 코디네이터가 복구하기를 기다리느라 멈출 수 있기 때문에 블로킹 원자적 커밋 프로토콜
    - 이론상 논블로킹 원자적 커밋 프로토콜이 가능하긴 함
    - 완벽한 장애 감지기가 필요한데, 이는 불안정한 네트워크 환경 상 불가능함
    - 실용적이지 않은 아이디어

4-6. 현실의 분산 트랜잭션
    - 데이터베이스 내부 분산 트랜잭션(표준 설정에서 복제와 파티셔닝 사용하는 경우): 데이터베이스 노드 사이에 내부 트랜잭션을 지원함, 트랜잭션에 참여하는 모든 노드는 동일한 데이터베이스 소프트웨어를 실행
    - 이종 분산 트랜잭션: 둘 이상의 다른 기술을 가진 참여자로 구성된 트랜잭션(메세지 브로커 같은 비데이터베이스 시스템 포함) -> 매우 어려움

4-7. 정확히 한 번 메세지 처리
    - 메시지 큐에서 나온 메시지는 그 메시지를 처리하는 데이터베이스 트랜잭션이 커밋에 성공했을 때만 처리되도록 구현 -> 메시지 확인과 데이터베이스 쓰기를 단일 트랜잭션에서 원자적 커밋으로 처리
    - 결과적으로 정확히 한 번 처리되도록 구현하는 것이 중요함

4-8. XA 트랜잭션
    - 이종 기술에 걸친 2단계 커밋 구현 표준 
    - 애플리케이션이 네트워크 드라이버나 클라이언트 라이브러리를 사용해 참여자 데이터베이스나 메시징 서비스와 통신한다고 가정함
    - XA API 사용
    - 코디네이터 존재 -> 코디네이터 로그 존재
    - 모든 통신이 클라이언트 라이브러리를 거쳐야 하기 때문에 데이터베이스 서버와 코디네이터가 직접 연결되지 않음

4-9. 의심스러운 상태에 있는 동안 잠금을 유지하는 문제
    - 데이터베이스는 트랜잭션이 커밋되거나 어보트할 때까지 잠금을 스스로 해제할 수 없음 -> 잠긴 로우는 누가 풀어주지 않는 이상 접근이 불가능해짐

4-10. 코디네이터 장애 복구
    - 트랜잭션 장애가 발생함녀 고아가 된 트랜잭션(의심스러운 트랜잭션)이 발생함 -> 구현 방식에 따라 수동으로 해소(커밋 OR 어보트)하지 않으면 영원히 잔류할 수도 있음
    - XA 구현에는 이런 의심스러운 트랜잭션을 어떻게 처리할 지 참여자가 직접 선택하는 '경험적 결정'이 존재 -> 2단계 커밋 약속 체계를 위반할 수 있음 -> 원자성을 깰 수 있음 -> 패닉 버튼

4-11. 분산 트랜잭션 제약
    - XA 트랜잭션은 코디네이터가 일종의 데이터베이스여야 함 -> 트랜잭션 결과 저장
    - 코디네이터가 복제되지 않고 단일 장비에서만 실행되면 시스템 단일 장애점이 됨 -> 상당수의 코디네이터 구현이 고가용성을 제공하지 않거나, 가장 기초적인 복제만 제공
    - 코디네이터가 애플리케이션 서버에 포함되면 코디네이터 로그를 보관해야 해서 stateful한 상태가 됨
    - XA는 광범위한 데이터 시스템과 호환돼야 하므로 최소 공통 분모가 돼야 함, 즉 표준화된 프로토콜을 유지할 수 없음. -> 여러 시스템에 걸친 교착상태를 파악할 수 없고, 직렬성 스냅숏 격리를 이용할 수 없음
    - XA가 아닌 데이터베이스 내부 분산 트랜잭션은 SSI도 쓸 수 있지만, 2PC가 성공적으로 트랜잭션커밋을 하려면 모든 참여자가 응답해야 한다는 점이 여전히 남음. 분산 트랜잭션은 장애를 증폭시키는 경향이 있음.

4-12. 내결함성을 지닌 합의
    - 균일한 합의: 하나 이상의 노드들이 값을 제안하고, 합의 알고리즘이 그 값 중 하나를 결정함
        - 균일한 동의: 어떤 두 노드도 다르게 결정하지 않는다. 즉, 어떤 노드가 값을 결정했다면, 그 뒤에 결정을 내리는 모든 프로세스는 반드시 같은 값을 결정해야 한다.
        - 무결성: 어떤 노드도 두 번 결정하지 않는다.
        - 유효성: 한 노드가 값 v를 결정한다면 v는 어떤 노드에서 제안된 것이다.
        - 종료: 죽지 않는 모든 노드는 결국 어떤 값을 결정한다. -> 일부 노드에서 장애가 나더라도 결국은 끝을 봐야 한다.
    - 대부분 합의 알고리즘은 비잔틴 결함이 없거나 1/3 미만이라고 가정함

4-13. 합의 알고리즘과 전체 순서 브로드캐스트
    - 뷰스탬프 복제, 팍소스, 라프트
    - 대다수는 균일한 합의 알고리즘을 직접 사용하지 않음 -> 전체 순서 브로드캐스트 알고리즘 선호
    - 전체 순서 브로드캐스트는 모든 노드에게 메시지가 정확히 한 번, 같은 순서로 전달돼야 함 -> 브로드캐스트 횟수만큼 합의를 한 것과 동일한 효과
        - 균일한 동의: 브로드캐스트된 메시지는 모두에게 동일한 순서로 전달되고 반박되지 않음으로 모든 노드가 동의한 것과 같은 효과
        - 무결성: 한 메시지는 정확히 한 번 전달되기 때문에 메시지가 중복되지 않음, 즉 같은 프로세스에서 두 번 결정하지 않음
        - 유효성: 메시지는 브로드캐스트되기 때문에 오염되거나 조작될 수 없다.
        - 종료: 메시지는 손실되지 않는다. 메시지가 결국 모든 노드에 전달되니까.

4-14. 단일 리더 복제와 합의
    - 리더를 누가 선택하느냐에 따라 단일 리더 복제에서도 합의를 고민해야 하게 됨
    - 사람이 직접 고를 경우: '독재자' 방식 합의 알고리즘. -> 수동 선출 방식 -> 종료 속성을 항상 만족할 수 없음
    - 노드들이 선출할 경우: 스플릿 브레인 문제 발생(네트워크 장애로 노드 20개 중 [10개 <-> 10개] 이런 식으로 두 집단으로 쪼개져 버리면?)

4-15. 에포크 번호 붙이기와 정족수
    - 합의 프로토콜은 리더가 유일하다고 보장하기 어려움, 대신 좀 더 약한 종류의 보장이 가능
    - 에포크 번호, 투표 번호(팍소스), 뷰 번호(뷰스탬프 복제)
    - 각 에포크 내에서는 리더가 유일함을 보장
    - 그룹 내에서 리더 선출 때마다 에포크 번호를 단조 증가시킴 -> 에포크 번호가 다른 두 리더 사이 충돌이 생기면 번호가 높은 리더가 이김
    - 각 리더는 자신의 결정을 과반수 이상의 노드에게 보내고, 정족수가 그 제안을 찬성하고 응답하기를 기다림 -> 노드는 에포크 번호가 더 높은 다른 리더를 인식하지 못했을 때만 찬성표를 던짐
    - 결국 안건이 있을 때마다 리더 선출과 리더 제안 찬반 투표라는 두 개의 투표를 각 노드는 수행해야 함 -> 그리고 투표를 하는 정족수는 항상 겹쳐야 함
    - 2PC와 유사한 점이 있음(차이점 - 조율자 선택 방식 차이: 리더/선출 + 과반수 VS 코디네이터/독재자 + 전체)

4-16. 합의의 제약
    - 합의가 모든 곳에 쓰이지는 않음, 트레이드 오프 문제
    - 노드가 제안에 투표하는 과정은 동기식 복제 <> 데이터베이스는 종종 비동기 복제를 사용함(성능)
    - 합의 시스템은 항상 엄격한 과반수 충족을 요구(노드 1대의 장애를 견디려면 최소 세 대의 노드가 필요해짐)
    - 보통 합의 알고리즘에서는 멤버십 수를 고정함 -> 동적 멤버십 확장은 매우 어려운 개념
    - 장애 노드 감지에 타임아웃에 주로 의존 -> 네트워크 지연 변동이 심한 경우 리더 재선출 이슈가 잦아져 성능 이슈 발생
    - 결국 합의 알고리즘은 네트워크 문제에 매우 민감한데, 현행 네트워크 인프라는 매우 불안정함

4-17. 멤버십과 코디네이션 서비스
    - 주키퍼, etcd: 분산 키-값 저장소/코디네이터 설정 서비스
    - 완전히 메모리 안에 들어올 수 있는 작은 양의 데이터 보관을 위해 설계
    - 전체 순서 브로드캐스트를 위한 로그 데이터
    - 주키퍼는 선형성 원자적 연산을 사용함 -> compare-and-set -> 분산 잠금은 보통 클라이언트 장애가 난 경우 결국엔 해제되도록 만료 시간이 있는 임차권을 사용함
    - 임차권을 통한 임시 잠금을 통해 보호할 때, 펜신 토큰이 필요함 -> 주키퍼는 트랜잭션 id와 버전 번호로 해결
    - 클라이언트와 주키퍼 서버는 하트비트를 주기적으로 교환, 세션 타임아웃보다 긴 기간 동안 하트비트가 멈추면 주키퍼는 세션이 죽었다고 선언함
    - 세션에서 획득한 잠금은 세션이 타임아웃 됐을 떄 자동 해제 설정 가능(단명 노드)
    - 클라이언트는 다른 클라이언트가 생성한 잠금과 값을 읽고, 거기에 변경이 있는지 감시할 수도 있음


4-18. 작업을 노드에 할당하기
    - 아래와 같은 상황에 주키퍼/처비 모델이 잘 작동함
        - 여러 프로세스나 서비스 중 하나가 리더나 주 구성요소로 선택돼야할 때
        - 파티셔닝된 자원이 있는데, 이 자원을 어느 노드에 할당해야 할 지 결정하는 경우

    - 원자적 연산, 단명 노드, 알림을 잘 사용하면 관리자 개입 없이 자동 처리하도록 구현 가능
    - 후에 수천 대의 노드가 생겼을 때, 매번 이 수천대의 노드가 리더 투표를 하는 것은 비효율적 -> 그래서 고정된 몇 개의 노드(3~5개) 사이에서 과반수 투표로 리더를 선출한 후, 해당 리더를 통해 수천대에 전체 순서 브로드캐스트를 하는 방식을 사용함
    - 코디네이트 작업의 일부(합의, 연산 순서화, 장애 감지)를 외부 서비스에 위탁하기도 함

4-19. 서비스 찾기
    - 주키퍼, etcd 등은 특정 서비스에 연결하려면 어떤 ip 주소로 접속해야 하는지 알아내는 용도로 자주 쓰임

4-20. 멤버십 서비스

    - 장애 감지와 합의를 연결하면 어떤 노드가 살아있는 것으로 여겨져야 하는지, 죽은 것으로 여겨져야 하는 지 동의할 수 있음
    - 당연히 잘못 선언될 가능성은 늘 존재, 하지만 어느 노드를 믿어야 할 지 알아야 하기 때문에 늘 유효한 방식
