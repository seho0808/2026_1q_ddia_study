# Chapter 9 - 일관성과 합의 (Consistency and Consensus)
## 1. 배경

### 분산 시스템에서 발생하는 문제들

분산 시스템은 **부분 장애(Partial Failure)** 가 발생해도 올바르게 동작해야 한다.

| 문제 | 설명 |
|------|------|
| 네트워크 지연 & 패킷 손실 | 메시지가 늦거나 아예 사라질 수 있음 |
| 노드 장애 | 언제든지 충돌하거나 재시작될 수 있음 |
| 클럭 스큐 (Clock Skew) | 서버마다 시간이 다를 수 있음 |
| 메시지 재정렬 | 전송 순서와 수신 순서가 다를 수 있음 |

> 이런 불확실성 속에서 **어떤 보장(guarantee)을 제공할 수 있을까?**

### 접근 방식: 추상화(Abstraction)

- 분산 시스템의 복잡성을 숨기는 **추상화**를 사용
- 애플리케이션이 신경 쓸 부분을 줄여줌
- 대표적 추상화: **트랜잭션**, **선형성**, **합의**

## 2. 선형성

### 2-1. 정의

> "시스템이 데이터의 복사본을 하나만 가지고, 그 복사본에 대한 모든 연산이 원자적(atomic)인 것처럼 보이게 하는 보장"

- 가장 강력한 일관성 모델
- **Atomic Consistency**, **Strong Consistency** 라고도 불림

### 2-2. 핵심 속성

- 읽기는 항상 **최신 쓰기 값**을 반환
- 한 클라이언트가 새 값을 읽었다면, **이후 모든 클라이언트도 새 값을 읽어야 함**
- 단일 글로벌 타임라인 존재
- 원자적 CAS(Compare-And-Set) 연산 지원

### 2-3. 선형성 vs 직렬성(Serializability)

| 구분 | 선형성 | 직렬성 |
|------|--------|--------|
| 대상 | 단일 객체의 읽기/쓰기 | 멀티 객체 트랜잭션 |
| 시간 | 실시간 순서 보장 | 트랜잭션 순서 보장 |
| 레벨 | 개별 연산 | 트랜잭션 묶음 |

> 둘 다 제공하면 **Strict Serializability (강한 직렬성)**

### 2-4. 선형성이 필요한 경우

- **리더 선출 (Leader Election)** - 분산 락
- **유일성 제약 (Uniqueness Constraint)** - 사용자명 중복 방지
- **채널 간 타이밍 의존성** - 파일 업로드 후 처리 시작

### 2-5. 선형성의 비용

```
선형성 높음  ->  성능 비용 높음  ->  가용성 낮음
```

- 네트워크 왕복 대기 필요
- **CAP 정리**에 따라 네트워크 파티션 발생 시 선형성 포기해야 함

### 2-6. CAP 정리

> Consistency / Availability / Partition Tolerance - 세 가지를 동시에 만족할 수 없다

- 네트워크 파티션(P)은 **피할 수 없음**
- 따라서 **C(일관성)와 A(가용성) 중 하나를 선택**해야 함

```
CP 시스템: 파티션 시 일부 요청 거부 (일관성 우선)
AP 시스템: 파티션 시 오래된 값 반환 가능 (가용성 우선)
```

> CAP 정리는 단순화된 모델 - 실제는 더 많은 트레이드오프가 존재

## 3. 순서 보장

### 3-1. 순서와 인과성 (Causality)

- **인과성(Causality)**: 원인이 결과보다 먼저 관찰되어야 함
- 인과적으로 의존된 연산들은 **순서**가 있어야 함
- 동시에 발생한 이벤트는 순서 없음 (어느 쪽이 먼저여도 무방)

### 3-2. 인과적 일관성 (Causal Consistency)

- 선형성보다 **약한 모델**
- 인과 관계에 있는 연산의 순서만 보장
- 네트워크 파티션에서도 **가용 가능**
- 선형성 없이도 많은 문제 해결 가능

> 일관성 강도: **선형성 > 인과적 일관성 > 최종적 일관성**

### 3-3. Lamport 타임스탬프

`(counter, nodeID)` 쌍으로 전체 순서(total order)를 생성하는 방법

```
send:    counter += 1
receive: counter = max(local_counter, received_counter) + 1
compare: 낮은 counter가 먼저, 같으면 nodeID로 비교
```

**한계**: 전체 순서는 보장하지만 인과성을 파악할 수는 없음

### 3-4. 전체 순서 브로드캐스트 (Total Order Broadcast)

- 모든 노드에 **동일한 순서로** 메시지를 전달
- 두 가지 보장:
  - **신뢰성 (Reliable delivery)**: 메시지는 결국 모든 노드에 전달됨
  - **전체 순서 (Totally ordered delivery)**: 모든 노드가 동일한 순서로 수신

> 선형성 <-> 전체 순서 브로드캐스트는 **동등하게 강력**

## 4. 분산 트랜잭션과 합의

### 4-1. 원자적 커밋 문제 (Atomic Commit)

- 트랜잭션이 여러 노드에 걸쳐 있을 때 **모두 커밋 or 모두 롤백** 보장 필요
- 단일 노드에서는 간단하지만, 분산 환경에서는 복잡

### 4-2. 2단계 커밋 (Two-Phase Commit, 2PC)

```
Phase 1 - Prepare
  코디네이터 -> 모든 참여자: "커밋할 준비됐나?"
  참여자 -> 코디네이터: Yes / No (Yes면 잠금 획득)

Phase 2 - Commit or Abort
  모든 Yes -> 코디네이터: "커밋하라" 전송
  하나라도 No -> 코디네이터: "중단하라" 전송
```

**2PC의 문제점**

| 문제 | 설명 |
|------|------|
| 코디네이터 장애 | 참여자가 in-doubt 상태로 블로킹 |
| 단일 장애점 (SPOF) | 코디네이터 없으면 진행 불가 |
| 성능 비용 | 두 번의 라운드트립, 잠금 유지 |

> **in-doubt 트랜잭션**: 코디네이터가 죽으면 참여자는 커밋/롤백을 알 수 없어 무한 대기

### 4-3. 합의 (Consensus)

> "여러 노드가 값에 대해 동의하고, 그 동의가 번복되지 않는 것"

**합의 알고리즘의 요구사항**

| 속성 | 설명 |
|------|------|
| Uniform Agreement | 모든 노드가 동일한 값 결정 |
| Integrity | 결정된 값은 반드시 제안된 값 중 하나 |
| Validity | 올바른 노드만 값 결정 가능 |
| Termination | 결국 모든 노드가 값을 결정 |

### 4-4. FLP 불가능성 정리

> "단 하나의 프로세스 장애도 허용하는 비동기 시스템에서는 합의가 불가능하다" 
- 비동기 시스템에서 장애 탐지가 불완전하기 때문
- 실제에서는 **타임아웃**과 **랜덤 지연**으로 우회

### 4-5. 합의 알고리즘 - Raft / Paxos

**Raft 알고리즘**
- **리더 선출**: 과반수 투표로 리더 결정
- **로그 복제**: 리더가 엔트리를 팔로워에 복제
- **과반수(Quorum) 동의** 필요

```
노드 5개 -> 3개 이상 동의 필요
노드 3개 -> 2개 이상 동의 필요
```

**Paxos vs Raft**

| | Paxos | Raft |
|--|-------|------|
| 이해도 | 어렵다 | 이해하기 쉽게 설계됨 |
| 실무 활용 | 오래된 구현 다수 | etcd, CockroachDB 등 |
| 리더 | 선택적 | 항상 존재 |

### 4-6. ZooKeeper와 분산 코디네이션

ZooKeeper, etcd는 합의 알고리즘을 내부적으로 구현하고 다음을 제공:

- **리더 선출** - 분산 락 기반
- **서비스 디스커버리** - 노드 등록/해제
- **멤버십 관리** - 클러스터 구성 변경
- **설정 관리** - 동적 설정 공유

> 직접 합의 알고리즘을 구현하기보다 ZooKeeper 같은 **검증된 서비스를 활용**하는 것이 권장됨

## 5. 핵심 요약

### 일관성 모델 계층

```
선형성 (Linearizability)         <- 가장 강력, 실시간 보장, 성능 비용 큼
인과적 일관성 (Causal Consistency) <- 실용적, 파티션 허용, 인과 순서 보장
최종적 일관성 (Eventual Consistency) <- 가장 약함, 높은 가용성
```

### 장단점 비교

| 개념 | 보장 | 트레이드오프 |
|------|------|-------------|
| 선형성 | 단일 복사본처럼 동작 | 성능 저하, 파티션 시 가용성 저하 |
| 인과적 일관성 | 인과 순서 보장 | 전체 순서 미보장 |
| 2PC | 원자적 커밋 | 블로킹, 코디네이터 SPOF |
| 합의 (Raft 등) | 노드 간 합의 | 과반수 필요, 구현 복잡 |

### 기억할 핵심 포인트

- 추상화를 잘 활용하면 복잡한 분산 시스템 문제를 단순하게 해결할 수 있다
- 선형성은 강력하지만 비싸다. 진짜 필요한지 생각하라
- 합의는 불가능하지 않다. 단지 어렵고 제약이 있을 뿐이다
