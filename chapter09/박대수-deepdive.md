# Chapter 09 - XA 트랜잭션 Deep Dive

## 1. XA의 본질: 분산 트랜잭션의 이상향 vs 현실

### XA가 해결하려는 문제
XA(eXtended Architecture)는 1990년대 X/Open에서 제정한 **이종 분산 트랜잭션 표준**이다. 핵심 목표는:

1. **이종 시스템 통합**: 서로 다른 데이터베이스(MySQL + PostgreSQL)나 메시징 시스템을 하나의 트랜잭션으로 묶기
2. **원자성 보장**: All-or-nothing semantics across heterogeneous systems
3. **표준화**: 벤더 독립적인 API 제공

### XA의 작동 원리
```java
// 의사코드로 본 XA 트랜잭션 플로우
XADataSource ds1 = // MySQL
XADataSource ds2 = // PostgreSQL

XAResource xaRes1 = ds1.getXAConnection().getXAResource();
XAResource xaRes2 = ds2.getXAConnection().getXAResource();

Xid xid = // 글로벌 트랜잭션 ID

// Phase 1: Prepare
xaRes1.prepare(xid);
xaRes2.prepare(xid);

// Phase 2: Commit/Rollback
if (allPrepared) {
    xaRes1.commit(xid, false);
    xaRes2.commit(xid, false);
} else {
    xaRes1.rollback(xid);
    xaRes2.rollback(xid);
}
```

## 2. XA의 근본적 딜레마: 이상향 vs 실용성

### 2.1 코디네이터의 모순적 위치
**문제점**: XA는 코디네이터를 "트랜잭션 매니저"로 정의하지만, 실제 구현에서 코디네이터는:

- **애플리케이션 서버에 내장**: Stateless 애플리케이션이 Stateful 코디네이터가 됨
- **단일 장애점**: 코디네이터 장애 시 전체 트랜잭션 시스템 마비
- **복제의 어려움**: 트랜잭션 상태를 복제하는 것이 복잡

**DDIA의 비판**: "코디네이터가 일종의 데이터베이스여야 함" — 코디네이터가 트랜잭션 결과를 저장해야 하는 아이러니

### 2.2 코디네이터 로그의 치명적 역할: "Stateless의 역설"
XA 트랜잭션 매니저(TM)는 단순히 명령을 전달하는 전달자가 아니라, **상태를 기록하는 저장소**여야 한다.

- **WAL(Write-Ahead Log)의 필수성**: TM은 참여자들에게 commit 명령을 보내기 직전, "이제 커밋하기로 결정했다"는 사실을 자신의 로컬 디스크에 로그로 기록(Sync)해야 한다.
- **고립된 리소스(Orphaned Resources)**: 애플리케이션 서버(TM 내장)가 로그를 남기지 않고 prepare 직후에 크래시된다면?
  - DB(참여자)들은 '준비 완료' 상태로 락을 쥐고 기다리지만, 살아난 TM은 이전 상태를 모르므로 명령을 내리지 못한다.
  - 이 상태의 데이터는 관리자가 수동으로 해소하기 전까지 영구적으로 잠기며(Deadlock과 유사) 전체 시스템의 가용성을 무너뜨린다.
- **결론**: 분산 트랜잭션을 안정적으로 운영하려면 TM 자체가 DB 수준의 고가용성(HA)과 영속성을 가져야 하는 역설에 직면한다.

### 2.3 의심스러운 트랜잭션: 블로킹의 근본 원인
**문제점**: 2PC의 블로킹 특성이 XA에서 극대화된다.

```
시나리오:
1. 코디네이터: "준비됐어?" → 참여자들: "준비됐어!"
2. 코디네이터: "커밋!" (하지만 네트워크 장애로 전달 실패)
3. 참여자들은 영원히 기다림... 잠금 유지... 시스템 정체
```

**실제 영향**:
- 데이터베이스 커넥션 풀 고갈
- 애플리케이션 스레드 블로킹
- 전체 시스템 가용성 저하

### 2.4 "경험적 결정": XA 약속의 위반
**XA의 비밀 무기**: "Heuristic Decisions" (경험적 결정)

```java
// XA API의 위험한 기능
xaResource.commit(xid, TMONEPHASE | TMNOWAIT); // 강제 커밋
xaResource.rollback(xid); // 강제 롤백
```

**문제점**:
- 원자성 파괴: 일부 참여자는 커밋, 일부는 롤백
- 데이터 불일치 발생
- "패닉 버튼"으로 전락

**DDIA의 지적**: "2단계 커밋 약속 체계를 위반할 수 있음"

## 3. XA의 성능과 확장성 한계

### 3.1 동기화 오버헤드
- 모든 참여자가 준비될 때까지 대기
- 네트워크 왕복 증가
- 데이터베이스 잠금 시간 연장

### 3.2 교착상태 감지 불가
**XA의 비극적 한계**: "XA는 광범위한 데이터 시스템과 호환돼야 하므로 최소 공통 분모가 돼야 함"

결과적으로:
- 교착상태 감지 불가
- SSI(Serializable Snapshot Isolation) 사용 불가
- 성능 최적화 제한

### 3.3 Read-Only 최적화: 2PC의 비용 절감 전략
모든 참여자가 2단계(Prepare-Commit)를 다 거치는 것은 성능상 매우 비효율적이다. 이를 최적화하기 위해 **Read-Only 프로토콜**이 사용된다.

**작동 원리**: 특정 참여자가 해당 트랜잭션 내에서 데이터를 조회(Select)만 하고 변경하지 않았다면, Prepare 요청을 받았을 때 `XA_RDONLY` 응답을 보낸다.

**이점**:
- **Phase 2 생략**: TM은 해당 참여자에게 commit이나 rollback을 보낼 필요가 없음을 즉시 인지하고 목록에서 제외한다.
- **락 조기 해제**: 참여자는 Prepare 직후 즉시 락을 해제하고 트랜잭션을 종료할 수 있어, DB 리소스 점유 시간을 획기적으로 줄인다.

**의의**: 이 최적화는 읽기 비중이 높은 분산 작업에서 XA의 오버헤드를 줄여주는 실질적인 장치이다. 다만 본질적인 블로킹 문제는 여전하다.

## 4. 현실 세계의 XA: 어디에 적합할까?

### 적합한 사용 사례
1. **금융 시스템**: 데이터 일관성이 절대적 우선
2. **레거시 시스템 통합**: 서로 다른 데이터베이스 강제 통합
3. **비즈니스 크리티컬 배치**: 오프라인 배치 처리

### 부적합한 사용 사례
1. **고성능 웹 애플리케이션**: 지연 시간 민감
2. **마이크로서비스**: 서비스 간 느슨한 결합 선호
3. **실시간 시스템**: 블로킹이 치명적

## 5. XA 이후의 세상: 대안적 접근법

### 5.1 Saga 패턴 (비동기 보상)
```java
// Saga: 실패 시 보상 트랜잭션 실행
try {
    orderService.createOrder();
    paymentService.charge();
    inventoryService.reserve();
} catch (Exception e) {
    paymentService.refund();
    inventoryService.release();
}
```

**장점**: 블로킹 없음, 부분 실패 처리 가능  
**단점**: 복잡한 보상 로직, 최종 일관성

### 5.2 Saga vs XA: 격리성(Isolation)의 부재와 데이터 오염

#### Saga 패턴 상세

**정의**: Saga는 긴 트랜잭션을 **여러 개의 짧은 로컬 트랜잭션**으로 쪼개고, 각 단계가 실패하면 **역순으로 보상 트랜잭션(Compensating Transaction)**을 실행해 원자성을 근사하는 패턴이다. 1987년 Hector Garcia-Molina 논문에서 제안되었다.

**구조**:
- **포워드 트랜잭션**: T1 → T2 → T3 순으로 실행
- **보상 트랜잭션**: C1 ← C2 ← C3 (실패 지점부터 역순 롤백)
- 각 Ti는 독립 커밋되므로, 중간 상태가 즉시 다른 트랜잭션에 노출된다.

**구현 방식**:

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **오케스트레이션** | 중앙 Saga 오케스트레이터가 각 서비스 호출·보상 순서 제어 | 순서 보장 용이, 상태 관리 집중 | 오케스트레이터 SPOF 가능 |
| **코레오그래피** | 각 서비스가 이벤트를 발행·구독하며 자기 보상 로직 실행 | 분산·느슨한 결합 | 흐름 파악 어려움, 순환 의존 위험 |

**실패 시나리오 예시** (주문 → 결제 → 재고):
```
1. 주문 생성(T1) ✓ → 2. 결제(T2) ✓ → 3. 재고 예약(T3) ✗ 실패
→ 보상: C3(재고 해제) → C2(결제 환불) → C1(주문 취소)
```

**보상의 한계**: 보상 트랜잭션은 "되돌리기"가 아니라 **반대 작업**이다. 예: 결제 환불은 원래 결제와 별도 API이며, 환불 실패·타임아웃·중복 호출 등 추가 장애 지점이 생긴다. **멱등성** 설계가 필수다.

---

Saga는 가용성을 위해 XA를 포기하지만, 그 대가로 **ACID의 'I'(격리성)**를 잃는다.

| 구분 | XA | Saga |
|------|----|------|
| 격리성 | 모든 참여자 커밋 전까지 데이터 비노출 | 로컬 트랜잭션 각각 커밋 → 중간 데이터 노출 |
| Dirty Read | 없음 | 가능 (예: 주문 완료 전 결제만 성공 시 통계 서비스가 해당 결제액 합산) |
| 가용성 | 블로킹으로 저하 | 높음 |

**선택의 기준**:
- **금융/회계**: "잠깐이라도 데이터가 꼬이면 안 된다" → XA (강력한 격리성)
- **이커머스/MS**: "결제 취소 보상 로직을 짜더라도 일단 주문이 밀리면 안 된다" → Saga (높은 가용성)

## 6. 결론: XA는 여전히 필요한 악마인가?

**XA의 가치**: 분산 트랜잭션의 골든 스탠다드, 강력한 일관성·격리성 보장

**XA의 현실**: 높은 복잡성, 성능 비용, 운영 부담, TM 로그 영속성 문제

**요약**: XA는 격리성(Isolation)을 보장하여 데이터 정합성을 지키지만, TM의 로그 영속성 문제로 인해 관리 부담이 크며 성능적 한계를 가진다. Read-Only 최적화 같은 기법을 쓰기도 하지만, 본질적인 블로킹 문제는 여전하다. 반면 Saga는 가용성이 높으나 격리성이 결여되어 비즈니스 차원의 복잡한 보상 로직과 데이터 오염 방지 전략이 추가로 요구된다.

**현대적 접근**: 상황에 따른 선택
- **강력한 일관성 필요**: XA 또는 합의 기반 시스템(etcd, ZooKeeper)
- **성능 우선**: Saga, 이벤트 기반 아키텍처
- **하이브리드**: 도메인별 전략 선택

**DDIA의 핵심 메시지**: *"분산 트랜잭션은 장애를 증폭시키는 경향이 있음"*

실제 시스템 설계에서는 **비용 vs 이익**을 철저히 계산해야 한다. XA는 강력한 도구지만, 망치를 들고 모든 것을 못으로 취급해서는 안 된다.
