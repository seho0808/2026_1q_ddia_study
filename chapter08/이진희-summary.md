# 8장. 분산 시스템의 골칫거리 - 요약

## 개요

분산 시스템에서 잘못될 수 있는 것들에 대한 비관적인 개요. 단일 컴퓨터는 결정적으로 동작하지만, 분산 시스템은 일부는 정상이고 일부는 고장난 **부분 장애(partial failure)** 상태가 발생할 수 있다. 부분 장애는 비결정적이어서 다루기 어렵다.

---

## 결함과 부분 장애

- **단일 컴퓨터**: 하드웨어가 올바르면 항상 같은 결과(결정적)
- **분산 시스템**: 일부는 정상, 일부는 예측 불가능하게 고장 → **부분 장애(비결정적)**

### HPC vs 클라우드

| 구분 | 슈퍼컴퓨터(HPC) | 클라우드 |
|------|----------------|---------|
| 용도 | 과학 계산 | 인터넷 서비스 |
| 하드웨어 | 특화 장비 | 상용 장비 |
| 장애 처리 | 전체 작업 중단 후 재시작 | 장애 감내하며 계속 동작 |

인터넷 서비스는 **신뢰성 없는 구성 요소**로 **신뢰성 있는 시스템**을 구축해야 한다.

---

## 신뢰성 없는 네트워크

인터넷/데이터센터 네트워크는 **비동기 패킷 네트워크** — 메시지 도착 여부와 시간을 보장하지 않는다.

### 요청 시 발생 가능한 문제
1. 요청 손실
2. 요청이 큐에서 지연
3. 원격 노드 장애
4. 원격 노드 일시 응답 불가 (GC 등)
5. 응답 손실
6. 응답 지연

### 타임아웃
- 유일한 결함 감지 수단이지만 적정값 설정이 어렵다
- 너무 길면 장애 감지가 느림 / 너무 짧으면 오탐 발생
- 비동기 네트워크는 **기약 없는 지연(unbounded delay)** 이 있음
- 해결책: **Phi Accrual failure detector** — 응답 시간 분포를 기반으로 타임아웃 자동 조정

### 네트워크 혼잡과 큐 대기
패킷 지연의 주요 원인:
- 여러 노드가 동시에 같은 목적지로 패킷 전송 → **네트워크 혼잡**
- CPU 코어가 바쁘면 OS가 큐에 넣어 대기
- 가상화 환경에서 수십 ms 멈춤
- TCP **흐름 제어(flow control)** 에 의한 송신 속도 제한

### 동기 vs 비동기 네트워크
- **전화망(회선 교환)**: 회선 예약 → **제한 있는 지연(bounded delay)**
- **이더넷/IP(패킷 교환)**: 큐 대기 → **기약 없는 지연**

---

## 신뢰성 없는 시계

분산 시스템에서 각 노드의 시계는 서로 다를 수 있다.

### 두 종류의 시계

| 시계 | 용도 | 특징 |
|------|------|------|
| **일 기준 시계** (time-of-day) | 날짜/시각 | NTP 동기화, 역방향 점프 가능 |
| **단조 시계** (monotonic) | 경과 시간 측정 | 항상 앞으로만 흐름, 동기화 불필요 |

### 시계의 문제점
- 수정 시계의 **드리프트(drift)** — 온도/전압에 따라 오차 발생
- NTP 동기화 정확도는 네트워크 지연만큼만 보장
- **윤초** 발생 시 1분이 59초 또는 61초
- 가상화 환경에서 추가 오차
- 시계를 이용한 이벤트 순서 결정(**LWW**, Last Write Wins)은 위험

### 해결책
- **논리적 시계(logical clock)**: 증가하는 카운터로 이벤트 상대 순서만 측정
- **구글 TrueTime API**: 시계 신뢰 구간을 명시적으로 보고, Spanner가 스냅숏 격리에 활용

---

## 프로세스 중단

노드가 임의 시점에 상당한 시간 동안 멈출 수 있다.

### 중단 원인
- **Stop-the-world GC** — 수 분 지속 가능
- 가상 장비 **서스펜드/재개**
- OS 컨텍스트 스위치 / 하이퍼바이저 스위치
- 동기식 디스크 I/O 대기
- 스왑(페이징)으로 인한 페이지 폴트
- SIGSTOP 신호

멈춘 노드는 재개 후 멈췄다는 사실을 알아채지 못한다. → **임차권(lease)** 기반 리더십이 위험한 이유

### 완화 방법
- GC 중단을 계획적 중단으로 간주 → 해당 노드 요청을 다른 노드가 처리

---

## 지식, 진실, 그리고 거짓말

### 정족수(Quorum)
노드는 자신의 판단을 100% 믿을 수 없으므로 **과반수 투표(quorum)** 에 의존한다.

### 펜싱 토큰(Fencing Token)
잠금/임차권 획득 시마다 단조 증가하는 토큰 발급. 저장소 서버는 더 큰 토큰을 처리했다면 이전 토큰 요청을 거부 → 오래된 리더의 잘못된 쓰기 방지

### 비잔틴 결함
- 노드가 고의로 거짓 응답을 보내는 경우 → **비잔틴 장군 문제**
- 이 책에서 다루는 시스템은 노드가 신뢰성은 없지만 **정직하다**고 가정
- 비잔틴 내결합성 솔루션은 비용이 커서 일반 시스템에서는 실용적이지 않음
- 단, 하드웨어 오류·버그·잘못된 설정 등 **약한 형태의 거짓말**은 체크섬·입력 검증 등으로 방어

## 시스템 모델

알고리즘이 가정하는 결함의 종류를 정형화한 추상화.

### 타이밍 모델
| 모델 | 설명 | 현실성 |
|------|------|--------|
| 동기식 | 지연/중단/시계 오차 모두 상한 있음 | 비현실적 |
| **부분 동기식** | 대부분 동기식이지만 가끔 한계 초과 | 현실적 |
| 비동기식 | 타이밍 가정 없음 | 너무 제한적 |

### 노드 장애 모델
| 모델 | 설명 |
|------|------|
| crash-stop | 죽으면 영원히 중단 |
| **crash-recovery** | 죽었다가 나중에 복구 가능 |
| 비잔틴 | 임의의 악의적 동작 |

→ 실제로는 **crash-recovery + 부분 동기식** 조합이 가장 유용

### 안전성(Safety) vs 활동성(Liveness)
- **안전성**: "나쁜 일은 일어나지 않는다" — 위반 시 되돌릴 수 없음
- **활동성**: "좋은 일은 결국 일어난다" — 미래에 만족될 가능성이 있음

---

## 정리

분산 시스템의 세 가지 핵심 난제:

1. **네트워크**: 패킷은 언제든 손실/지연 가능, 응답 없음 = 전달 여부 알 수 없음
2. **시계**: 노드 간 시계 불일치, 갑작스러운 점프, 오차 측정 어려움
3. **프로세스**: GC 등으로 임의 중단 후 재개 시 멈췄다는 사실을 인지 못함

**부분 실패**는 분산 시스템의 본질적 특성. 결함 감지(타임아웃)도 어렵고, 감지 후 처리(정족수, 펜싱)도 복잡하다. 다음 장에서는 이 문제들을 해결하는 알고리즘을 다룬다.
