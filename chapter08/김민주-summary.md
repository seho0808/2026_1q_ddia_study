# Chapter 8. 분산 시스템의 골칫거리
### Designing Data-Intensive Applications

## 1. 왜 분산 시스템은 어려운가?

### 단일 컴퓨터 vs 분산 시스템

| 구분 | 단일 컴퓨터 | 분산 시스템 |
|------|------------|------------|
| 장애 방식 | **전체 실패** (all-or-nothing) | **부분 실패** (partial failure) |
| 예측 가능성 | 결정론적 | 비결정론적 |
| 디버깅 | 상대적으로 쉬움 | 매우 어려움 |

### 핵심 개념: 부분 장애 (Partial Failure)

> 시스템 일부는 정상 동작하고, 다른 일부는 예측 불가능한 방식으로 실패하는 상황

- 네트워크를 통해 메시지를 보냈을 때 **성공 여부를 알 수 없다**
- 요청이 유실됐는지, 응답이 유실됐는지, 원격 노드가 죽었는지 **구분 불가**
- 분산 시스템 설계의 핵심은 **부분 장애를 인정하고 견딜 수 있도록** 설계하는 것

---

## 2. 신뢰할 수 없는 네트워크

### 네트워크 패킷의 운명

요청을 보냈을 때 응답이 없다면 어떤 일이 일어난 걸까?

```
[클라이언트] --요청--> ???  --응답--> [클라이언트]
```

가능한 시나리오:
1. 요청이 유실됨
2. 요청이 큐에서 대기 중 (원격 노드는 아직 못 받음)
3. 원격 노드가 죽었음
4. 원격 노드가 일시적으로 멈춤 (ex. GC 중)
5. 원격 노드가 처리했지만, 응답이 유실됨
6. 응답이 큐에서 대기 중 (네트워크 지연)

> **결론**: 타임아웃이 발생하면 **무슨 일이 일어났는지 알 수 없다**

---

### 실제 네트워크 지연 분포

- 네트워크 지연은 **비대칭적**이고 **꼬리가 긴 분포(long-tail distribution)**
- 99.9%는 빠르지만 0.1%는 매우 느릴 수 있음
- **최악의 경우**를 기준으로 설계해야 함

### 타임아웃 설정의 딜레마

| 타임아웃 | 장점 | 단점 |
|---------|------|------|
| **짧게** | 장애를 빠르게 감지 | 오탐(false positive) 증가 |
| **길게** | 오탐 감소 | 장애 감지 지연 |

> 적절한 타임아웃 값은 **실험적으로** 결정해야 함 (PHI Accrual, Erlang 방식 등)

---

### 네트워크 파티션

```
  [노드 A] ========= X ========= [노드 B]
           네트워크 파티션 발생
```

- 두 노드가 서로를 죽었다고 **오판**할 수 있음
- 각자 계속 동작하면서 **일관성 깨짐** 발생 가능

---

## 3. 신뢰할 수 없는 시계

### 컴퓨터의 두 가지 시계

#### 1) 일 기준 시계 (Time-of-day clock)
- `System.currentTimeMillis()`, `clock_gettime(CLOCK_REALTIME)`
- **NTP**로 동기화 → 시간이 **뒤로 돌아갈 수 있음**
- 이벤트 순서 측정에 **부적합**

#### 2) 단조 시계 (Monotonic clock)
- `System.nanoTime()`, `clock_gettime(CLOCK_MONOTONIC)`
- 항상 **앞으로만** 증가
- 경과 시간 측정에 **적합** (절대 시각은 의미 없음)

---

### 분산 시스템에서 시계가 위험한 이유

**시나리오: 마지막 쓰기 승리 (Last Write Wins)**

```
클라이언트 A: x = 1  (타임스탬프: 42.004초)
클라이언트 B: x = 2  (타임스탬프: 42.003초) ← 시계가 약간 느림

→ A의 쓰기가 나중에 발생했지만 B의 쓰기가 "승리"
→ 데이터 소실!
```

> 분산 시스템에서 시계 기반 이벤트 순서는 **신뢰하면 안 된다**

---

### 동기화 정확도의 한계

| 환경 | 클럭 오차 |
|------|---------|
| 데이터센터 NTP | 수 밀리초 |
| 인터넷 NTP | ~100ms |
| GPS 수신기 | ~1µs |
| 쿼츠 크리스탈(드리프트) | 수십 ms/일 |

### 구글 TrueTime API (Spanner)

- 시간을 **구간**으로 표현: `[earliest, latest]`
- 실제 시간이 이 구간 내에 있음을 **보장**
- 트랜잭션 커밋 시 구간이 겹치지 않을 때까지 **의도적으로 기다림**
- 하지만 이런 수준의 정밀도는 일반 환경에서 **구현하기 매우 어려움**

---

## 4. 프로세스 중단 (Process Pause)

### 프로세스가 갑자기 멈추는 이유

- **GC (Stop-the-world Garbage Collection)**: 수십 초간 멈출 수 있음
- **가상 머신 일시 중단**: VM 마이그레이션, 절전 모드
- **OS 컨텍스트 스위치**: CPU 스케줄링
- **디스크 I/O**: 동기 파일 시스템 작업
- **페이지 폴트**: 스왑 영역 접근

---

### 위험한 시나리오: 리더 노드

```
1. 노드 A: 리더 선출, 임차 획득 (유효기간 30초)
2. 노드 A: GC 발생, 50초간 중단
3. 노드 B: 임차 만료 → 새 리더 선출
4. 노드 A: GC 종료 → "나는 여전히 리더" 착각
   → 두 리더가 동시에 쓰기 수행!
```

> 프로세스는 자신이 얼마나 멈춰 있었는지 **알 수 없다**

### 리스 (Lease) 기반 잠금의 위험

```python
# 위험한 코드
while lease.valid():      # ← 이 체크 후 GC가 발생하면?
    process_request()     # ← GC 이후 리스가 만료됐지만 계속 실행
```

**안전하게 만들려면**: 펜싱 토큰(Fencing Token) 사용 필요

---

## 5. 지식, 진실, 거짓말

### 노드는 아무것도 확신할 수 없다

- 네트워크 장애인지, 원격 노드가 죽었는지 **모른다**
- 자신의 시계가 정확한지 **모른다**
- 자신이 GC로 얼마나 멈춰 있었는지 **모른다**

> **"쿼럼(Quorum)"**: 다수결로 진실을 결정한다

---

### 펜싱 토큰 (Fencing Token)

분산 잠금을 **안전하게** 구현하는 방법:

```
1. 클라이언트 A가 잠금 획득 → 토큰 33 발급
2. 클라이언트 A가 GC로 인해 멈춤
3. 클라이언트 B가 잠금 획득 → 토큰 34 발급
4. 클라이언트 B가 스토리지에 쓰기 (토큰 34)
5. 클라이언트 A 재개 → 스토리지에 쓰기 시도 (토큰 33)
6. 스토리지가 토큰 33 < 34이므로 거부!
```

**핵심**: 스토리지 서버가 **단조 증가 토큰**을 검증함으로써 오래된 잠금 소유자를 차단

---

### 비잔틴 장애 (Byzantine Fault)

| 장애 유형 | 정의 | 예시 |
|---------|------|------|
| **충돌-중단 (Crash-stop)** | 노드가 단순히 멈춤 | 서버 다운 |
| **충돌-복구 (Crash-recovery)** | 멈췄다가 재시작 | 재부팅 |
| **비잔틴 (Byzantine)** | **거짓 정보를 전달** | 해킹, 버그, 하드웨어 오류 |

> 대부분의 분산 데이터베이스는 **비잔틴 장애를 가정하지 않음**  
> (같은 조직의 신뢰할 수 있는 노드라고 가정)

---

## 6. 시스템 모델과 현실

### 타이밍 모델

| 모델 | 설명 | 현실성 |
|------|------|--------|
| **동기 모델** | 지연 상한 존재, 시계 오차 상한 존재 | 현실과 거리가 멈 |
| **부분 동기 모델** | 대부분은 동기적, 가끔 한계 초과 | **현실에 가장 가까움** |
| **비동기 모델** | 타이밍에 대한 가정 없음 | 매우 제한적 |

---

### 노드 장애 모델

| 모델 | 설명 |
|------|------|
| **충돌-중단** | 노드는 죽기만 하고, 죽은 이후엔 아무것도 안 함 |
| **충돌-복구** | 노드가 죽었다가 다시 살아날 수 있음, 안정적 저장소 사용 |
| **비잔틴** | 노드가 거짓말하거나 악의적 행동 가능 |

> 현실에서 가장 유용한 모델: **부분 동기 + 충돌-복구**

---

### 알고리즘의 정확성 (Correctness)

좋은 분산 알고리즘이 만족해야 하는 속성:

- **안전성 (Safety)**: "나쁜 일은 절대 일어나지 않는다"
  - ex) 두 노드가 동시에 리더가 되지 않는다
- **활동성 (Liveness)**: "결국에는 좋은 일이 일어난다"  
  - ex) 노드가 죽더라도 결국 새 리더를 선출한다

> **Trade-off**: 네트워크 파티션 상황에서 안전성과 활동성을 **동시에 보장할 수 없다** → CAP 정리

---

## 7. 핵심 요약

### 분산 시스템에서 가정해야 할 것들

```
- 네트워크 패킷은 유실, 지연, 중복될 수 있다
- 노드의 시계는 다른 노드와 동기화되지 않을 수 있다
- 프로세스는 언제든지 임의의 시간 동안 멈출 수 있다
- 노드는 자신이 얼마나 멈췄는지, 메시지가 전달됐는지 알 수 없다
```
