# 이번 챕터가 언급한 내용들이 왜 와닿지가 않을까? 큰 분산 시스템을 운용하고 있지 않아서인가..?

1. "백만 분의 일"의 마법 (규모의 차이)
   텍스트에서도 언급되었듯, '백만 분의 일' 확률로 일어나는 장애는 내 노트북이나 소규모 서버 2~3대에서는 평생 한 번 볼까 말까 한 일입니다. 하지만 구글이나 넷플릭스처럼 수만 대의 서버를 운영하면, 그 '백만 분의 일' 확률의 장애가 매일 아침 출근할 때마다 터지는 일상이 됩니다.
   개인 프로젝트: 확률 0%에 수렴 (네트워크는 항상 완벽해 보임)
   대규모 시스템: 확률 100% (어딘가는 지금 이 순간에도 고장 나 있음)
2. 현대 도구들의 "완벽한 은폐"
   우리가 쓰는 현대의 프레임워크와 클라우드 서비스(AWS, Kubernetes, Kafka 등)는 사실 이 9장에서 다룬 끔찍한 일들을 대신 처리해주기 위해 만들어진 것들입니다.
   TCP가 패킷 유실을 대신 복구해주고,
   Kubernetes가 죽은 노드를 알아서 재시작해주며,
   Managed DB가 복제본 간의 시계 차이를 어느 정도 감춰줍니다.
   우리는 아주 잘 닦인 아스팔트 위에서 운전하고 있기 때문에, 그 아래에 얼마나 복잡한 배수구와 진흙탕(네트워크 결함, 시계 오차 등)이 있는지 체감하기 어려운 것이죠.

3. "Localhost"의 배신
   우리는 보통 개발을 할 때 localhost에서 테스트합니다. localhost 환경은:
   네트워크 지연이 거의 0에 가깝고(결정론적),
   시계가 나 자신의 시계이므로 오차가 없으며,
   프로세스가 멈추면(Break point) 세상도 같이 멈추는 환상적인 곳입니다.
   이런 **'완벽한 실험실'**에 익숙해져 있다 보니, 물리적 세계의 지저분한 현실(비결정론)이 비현실적으로 느껴지는 것입니다.

어떻게 하면 "와닿게" 할 수 있을까?
이론만으로는 절대 와닿지 않습니다. 가장 좋은 방법은 내 시스템에 **'인위적인 고통'**을 주는 것입니다.
Fault Injection (결함 주입): 코드에 sleep(random)을 넣어 네트워크 지연을 흉내 내보거나, 패킷을 강제로 5% 정도 유실시켜 보세요.
Chaos Engineering: 잘 돌아가는 서버 하나를 예고 없이 죽여(kill -9)보세요. 내 서비스가 여전히 멀쩡하게 돌아가는지 확인하는 순간, 마틴 클레프만이 왜 그렇게 "비관주의자"가 되라고 했는지 뼈저리게 느끼게 됩니다.

# 이러한 환경을 테스팅하는데에는 chaos engineering이 전부인가? 보통 어떻게 테스팅하는가? big tech best practices

카오스 엔지니어링(Chaos Engineering)이 가장 유명하긴 하지만, 분산 시스템의 그 지저분한 '비결정론적' 특성을 잡아내기 위해 빅테크 기업들은 훨씬 더 다층적이고 정교한 테스트 전략을 사용합니다.

단순히 "서버를 죽여보는 것"을 넘어, 실무에서 사용하는 Big Tech의 Best Practice 4가지를 정리해 드릴게요.

1. 결정적 시뮬레이션 테스팅 (Deterministic Simulation Testing, DST)
   현재 분산 시스템 테스팅의 **'끝판왕'**으로 불리는 기법입니다. FoundationDB, TigerBeetle, AWS Lambda 팀 등이 사용하여 극강의 신뢰성을 확보한 방법입니다.
   핵심 원리: 네트워크 지연, 시계 점프, 스레드 스케줄링 등 모든 '무작위성'을 가상화(Mocking)하여 시뮬레이션 환경 안에 가둡니다.
   왜 하는가? 카오스 엔지니어링은 '재현'이 어렵습니다. 하지만 DST는 난수 발생기(Seed)를 고정하면 수조 분의 일 확률로 발생하는 버그도 똑같이 100% 재현할 수 있습니다.
   효과: 실제 시간으로는 수십 년이 걸릴 테스트를 시뮬레이션상에서 단 몇 시간 만에 수행하며 수만 가지의 장애 조합을 검증합니다.
2. Jepsen 테스팅 (검증 및 모델 체킹)
   분산 데이터베이스나 메시지 큐를 만드는 팀들이 필수적으로 거치는 과정입니다.
   핵심 원리: 시스템에 무작위로 장애(네트워크 단절, 파티션 등)를 일으키면서, 동시에 아주 정교한 논리적 검증기를 돌려 데이터 일관성이 깨지는지 실시간으로 확인합니다.
   Big Tech 활용: 새로운 분산 알고리즘을 도입할 때 TLA+ 같은 언어로 설계를 검증하고, 구현체는 Jepsen 스타일의 테스트로 '안전성(Safety)' 성질이 깨지는지 확인합니다.
3. 섀도잉 및 카나리 배포 (Production-level Testing)
   이론적으로 아무리 테스트해도 실제 사용자의 트래픽 패턴은 예측할 수 없습니다. 그래서 빅테크는 운영 환경의 트래픽을 이용합니다.
   트래픽 섀도잉 (Traffic Shadowing): 실제 사용자 요청을 복사해서 '테스트 버전' 서버에도 보냅니다. 테스트 서버가 에러를 내거나 데이터를 날려도 실제 사용자에게는 영향이 없으므로, 운영 환경과 똑같은 부하와 네트워크 상황에서 안전하게 테스트할 수 있습니다.
   카나리 배포 (Canary Release): 아주 극소수의 사용자(예: 1%)에게만 새 버전을 노출하고, 9장의 내용처럼 "응답 속도가 느려지거나", "시계 오차로 인한 에러"가 발생하는지 모니터링한 뒤 점진적으로 확대합니다.
4. 장애 복구 훈련 (DiRT - Disaster Recovery Testing)
   구글(Google)에서 매년 수행하는 전사적 훈련으로 유명합니다.
   핵심 원리: 단순히 소프트웨어 버그를 찾는 게 아니라, '시스템 + 사람 + 프로세스' 전체를 테스트합니다.
   방법: 예고 없이 특정 데이터센터 전체의 전원을 차단하거나, 특정 지역의 네트워크를 끊어버립니다. 이때 시스템이 자동으로 복구되는지, 엔지니어들이 매뉴얼대로 대응하는지 확인합니다.
   교훈: 기술적 결함만큼이나 "사람의 판단 실수"가 큰 장애를 만든다는 것을 깨닫게 해줍니다.
   요약하자면, 빅테크의 테스팅 여정은 이렇습니다.
   로직 검증: TLA+ 등을 이용해 설계가 논리적으로 맞는지 확인.
   DST: 시뮬레이션으로 수만 가지의 지저분한 에지 케이스(Edge cases)를 선제적으로 제거.
   Fault Injection: 단위/통합 테스트 단계에서 의도적으로 에러를 주입.
   Chaos Engineering: 실제 운영 환경 근처에서 예상치 못한 변수를 확인.
   Shadowing/Canary: 실제 트래픽으로 마지막 안전망 구축.

# 딥다이브 가능한 주제들

### 1. 네트워크 장애 시뮬레이션: "혼돈의 전도사" 되어보기

이론으로 배운 "6가지 응답 부재 원인"을 로컬 환경에서 강제로 만들어보는 실습입니다.

- **실습 도구:** `Toxiproxy` (Shopify 제작) 또는 `Linux tc (Traffic Control)`
- **과제:**
- **Latency Injection:** 마이크로서비스 간 통신에 5초 이상의 지연을 주어, 애플리케이션의 타임아웃 설정이 의도대로 작동하는지 확인하세요.
- **Packet Loss:** 패킷 유실율을 10%~20%로 설정했을 때, TCP 재전송이 전체 응답 속도(P99 Latency)에 미치는 영향을 메트릭으로 관찰하세요.
- **실제 구현:** `Chaos Mesh` 같은 도구를 Kubernetes 환경에 설치하고, 특정 포드(Pod)의 네트워크를 주기적으로 끊었을 때 서비스 가용성이 유지되는지 테스트해 보세요.

---

### 2. 시계와 순서: "시간의 마법사" 탈출하기

물리적 시계의 위험성을 이해했다면, 논리적 시계와 인과 관계를 직접 구현해 볼 차례입니다.

- **실습 과제:**
- **Lamport Clock 구현:** 여러 노드 간에 메시지를 주고받을 때, 각 메시지에 카운터를 붙여 이벤트의 선후 관계를 확정하는 간단한 채팅 서버를 만들어 보세요.
- **Vector Clock 도전:** 인과 관계가 깨진(Concurrent write) 상황을 감지하고 사용자에게 충돌 해결을 요청하는 로직을 작성해 보세요.

- **딥다이브:** \* **Google Spanner의 TrueTime:** 구글이 어떻게 GPS와 원자시계를 이용해 "오차 범위($\epsilon$)"를 계산하고, 커밋 대기(Commit Wait)를 통해 외부 일관성을 보장하는지 백서를 읽어보세요.

---

### 3. 분산 락과 좀비 프로세스: "펜싱 토큰" 구현

GC 중단으로 인해 발생하는 '좀비 노드' 문제를 코드로 해결해 보는 가장 실무적인 연습입니다.

- **실습 환경:** Redis(Redlock) 또는 ZooKeeper
- **과제:**

1. **문제 재현:** 자바 스레드에서 `Thread.sleep()`이나 디버거 브레이크포인트를 이용해 락 임차권(Lease)이 만료된 상황을 강제로 만듭니다.
2. **해결책 적용:** 스토리지 계층(예: MySQL 또는 간단한 파일 시스템)에서 **펜싱 토큰(Fencing Token)** 검증 로직을 추가하세요.

- 구현 로직: `Update Data Set value = X where token > last_seen_token`
- 이때 낮은 토큰을 가진 좀비 노드의 요청이 정확히 거부(Reject)되는지 로그로 확인합니다.

---

### 4. 공식 검증 (Formal Methods): "수학으로 증명하는 버그"

단순한 유닛 테스트로는 발견할 수 없는 분산 시스템의 엣지 케이스(Edge case)를 찾는 프로의 도구입니다.

- **실습 도구:** `TLA+` 또는 `Jepsen`
- **과제:**
- **TLA+ 기초:** "두 노드가 동시에 락을 잡을 수 없다"는 안전성(Safety) 속성을 TLA+ 문법으로 작성하고 모델 체커를 돌려보세요.
- **Jepsen Test 읽기:** 분산 데이터베이스(MongoDB, PostgreSQL, Kafka 등)의 결함을 무자비하게 찾아내는 [Jepsen 분석 리포트](https://jepsen.io/analyses)를 하나 골라 정독해 보세요. "네트워크 파티션 상황에서 이 DB가 정말 광고하는 대로 일관성을 지키는가?"에 대한 통찰을 얻을 수 있습니다.

# 내가 스트레스 받는 것: 위에 처럼 나열은 되어있는데, 그냥 무한한건지 아니면 알려진 문제가 한정적인지..? 저거 ddia 9챕터인가 그럼

공부하시면서 충분히 스트레스 받으실 만한 대목입니다. 끝도 없이 쏟아지는 "안 되는 이유"들을 보고 있으면, "이걸 다 고려해서 코드를 짜는 게 인간이 가능한 영역인가?"라는 현타가 오기 마련이죠.

결론부터 말씀드리면, **현상(Cause)은 무한하지만, 모델(Model)은 한정적입니다.** 즉, 엔지니어가 대응해야 할 **'문제의 종류'**는 의외로 깔끔하게 정리가 되어 있습니다.

---

## 1. 현상은 무한하지만, 범주는 'Big 3'로 수렴합니다

상어가 케이블을 씹든, 인턴이 전원 선을 뽑든, 아니면 데이터센터에 홍수가 나든, 분산 시스템 엔지니어 입장에서 그 원인은 중요하지 않습니다. 결과적으로 다음 **세 가지 범주** 안에만 들어오면 됩니다.

- **네트워크 (The Network):** 메시지가 가다가 사라지거나, 늦게 오거나.
- **시계 (The Clock):** 시간이 거꾸로 가거나, 옆 동네랑 시간이 다르거나.
- **프로세스 (The Process):** 코드가 실행되다가 갑자기 얼어버리거나(GC 등), 죽어버리거나.

DDIA의 이 파트가 공포스러운 이유는, 우리가 당연하게 믿었던 "시간"과 "순서"라는 기초 체력을 박살 내기 때문입니다. 하지만 이 '지옥'에도 끝은 있습니다.

---

## 2. 혼돈을 가두는 상자: "시스템 모델 (System Models)"

학자들과 선배 엔지니어들은 이 무한한 혼돈을 통제하기 위해 **'가정의 울타리'**를 쳤습니다. 우리가 모든 사고를 다 막을 순 없으니, "우리 시스템은 최소한 이 정도 상황은 견딘다"라고 선언하는 것이죠.

### (1) 타이밍 모델 (시간의 불확실성을 가두기)

| 모델                             | 특징                                                      | 현실성                                     |
| -------------------------------- | --------------------------------------------------------- | ------------------------------------------ |
| **동기식 (Synchronous)**         | 지연 시간이 무조건 X초 이내라고 가정.                     | 거의 불가능 (특수 하드웨어만 가능)         |
| **비동기식 (Asynchronous)**      | 지연 시간에 아무 제한이 없음. 시계도 안 믿음.             | 너무 엄격해서 알고리즘 설계가 매우 어려움. |
| **부분 동기식 (Partially Sync)** | 대부분은 잘 되는데, 가끔 미친 듯이 느려질 수 있다고 가정. | **대부분의 실무 시스템이 택하는 모델.**    |

### (2) 노드 결함 모델 (죽음의 종류를 가두기)

- **Crash-stop:** 죽으면 끝이다. (깔끔함)
- **Crash-recovery:** 죽어도 다시 살아날 수 있다. (데이터베이스 등 실무 모델)
- **Byzantine:** 노드가 거짓말을 하거나 공격을 한다. (블록체인 아니면 보통 무시)

---

## 3. 왜 무한하게 느껴질까? (심리적 장벽)

지금 느끼시는 스트레스의 정체는 아마 **"비결정론(Non-determinism)"** 때문일 겁니다.
단일 장비에서는 `A -> B`가 당연한 진리였는데, 여기서는 `A가 갈 수도 있고, B가 먼저 올 수도 있고, 둘 다 안 갈 수도 있다`는 확률의 세계로 들어가야 하니까요.

하지만 걱정 마세요. DDIA 후반부로 갈수록 이 지옥 같은 상황들을 해결하기 위한 **'합의(Consensus)'**와 **'복제(Replication)'** 알고리즘들이 등장합니다. 이 알고리즘들은 "상어가 케이블을 씹어도, 시계가 5초 틀려도, 우리 시스템은 데이터 안 깨먹는다"라는 것을 **수학적으로 보장**해 줍니다.

## 4. 스트레스를 줄이는 마인드셋

- **"모든 것을 막을 필요는 없다":** 모든 장애를 막는 게 아니라, **'안전성(Safety)'** 속성 하나만 지키면 됩니다. (예: "데이터가 중복될 순 있어도, 틀린 데이터가 저장되지는 않게 하겠다.")
- **"추상화의 힘을 믿으세요":** 우리가 직접 TCP 패킷 유실을 처리하지 않듯, 나중에는 Raft나 Paxos 같은 표준 프로토콜이 이 복잡한 문제들을 대신 해결해 줍니다.

지금 보시는 파트는 **"우리가 싸워야 할 적군 리스트"**를 파악하는 단계입니다. 적군이 많아 보이지만, 결국 **'네트워크, 시계, 멈춤'**이라는 세 명의 대장급 빌런만 잘 감시하면 됩니다.

혹시 정리하신 내용 중에 특히 **"이건 진짜 이해 안 가고 억지 같다"** 싶은 구체적인 케이스가 있나요? 그 부분만 집중적으로 패보겠습니다. Would you like me to clarify a specific failure scenario that feels particularly confusing?
