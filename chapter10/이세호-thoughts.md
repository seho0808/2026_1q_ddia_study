- 이 책을 보고 든 인상:
  - "배치 처리는 실시간 대용량 트래픽 처리 보다는 훨씬 쉬우니 그렇게 어려워할 것 없다"라고 말하는 느낌이었음.
  - 사실 이 책에서 어려웠던 부분은 분산 시스템에서 합의하고 그런것들인데, 배치 처리에서는 애초에 시간 제약이 실시간이 아니다보니 훨씬 느슨함.

# 스파크는 언제부터 써야할까? 내가 알기론 진짜 수백 테라급부터 효율이 올라간다 들었던 것 같은데.

- 결론부터 말씀드리면, 데이터의 양이 단일 서버의 메모리(RAM)를 훨씬 초과하고, 처리 시간이 감당 안 될 때가 바로 스파크를 도입할 시점입니다.
- 수백 테라 기준에 대하여: 과거에는 분산 처리 오버헤드(네트워크 통신, 직렬화 등)가 커서 아주 큰 데이터에서만 이득이 있었지만, 지금은 수백 GB에서 수 TB 정도만 되어도 스파크를 쓰는 것이 훨씬 효율적입니다.
- 싱글 노드의 역습: 요즘은 단일 서버도 RAM이 256GB, 512GB씩 꽂히기도 하죠. 만약 데이터가 100GB 정도라면 스파크보다 Pandas나 최근 뜨고 있는 Polars, DuckDB를 한 대의 고성능 서버에서 돌리는 게 훨씬 빠를 수 있습니다. 스파크는 여러 대의 컴퓨터가 통신하며 맞추는 "조율 시간"이 필요하기 때문입니다.
- 스파크를 써야 하는 진짜 신호:
  - 메모리 부족(OOM): Memory Error가 계속 나고, 서버 사양을 아무리 올려도 답이 없을 때.
  - 처리 시간: 단일 서버에서 10시간 걸리는 작업이 분산 처리를 통해 30분으로 줄어들 수 있을 때.
  - 에코시스템: 이미 데이터가 HDFS나 S3 같은 분산 저장소에 있고, 이를 그대로 활용해야 할 때.

# HDFS는 뭘까? 실제로 써보질 않으니 와닿지가 않는듯함.

실제로 써보지 않으면 그냥 "어려운 파일 시스템" 같지만, 쉽게 비유하자면 **"여러 대의 컴퓨터 하드디스크를 하나처럼 묶은 거대한 가상 드라이브"**입니다.

### 왜 그냥 하드디스크를 안 쓰고 HDFS를 쓰나요?

우리가 영화 파일을 저장할 때 10GB짜리는 내 컴에 저장하면 되지만, 만약 **1PB(1,024TB)짜리 파일**이 있다면 어떻게 할까요? 세상에 1PB짜리 하드디스크는 없습니다.

1. **파일 쪼개기 (Blocks):** HDFS는 큰 파일을 **128MB(기본값) 단위의 블록**으로 잘게 쪼갭니다.
2. **나눠 담기:** 이 쪼개진 조각들을 수십, 수백 대의 서버(DataNodes)에 골고루 나눠서 저장합니다.
3. **복제 (Replication):** 이게 핵심인데, 조각 하나를 최소 3군데 서버에 복사해둡니다. 한 대가 고장 나도 데이터가 사라지지 않게 하려는 **'안전장치'**입니다.
4. **장부 관리 (NameNode):** "A 파일의 첫 번째 조각은 1번 서버에 있고, 두 번째 조각은 5번 서버에 있어"라는 장부를 들고 있는 대장 컴퓨터가 하나 있는데, 이를 **NameNode**라고 부릅니다.

### 비유로 이해하기

> 당신이 거대한 도서관의 관장입니다. 책이 너무 많아서 한 건물에 다 못 넣습니다. 그래서 10개의 건물에 책을 나눠 담고, 본관(NameNode)에는 "어떤 책이 몇 번 건물의 몇 번째 칸에 있는지" 적힌 거대한 카탈로그를 둡니다. 누군가 책을 찾으러 오면 카탈로그를 보고 위치를 알려주는 방식입니다.

### 스파크와 HDFS의 관계

둘은 환상의 콤비입니다.

- **HDFS:** 데이터를 안전하고 거대하게 **저장**하는 곳.
- **Spark:** HDFS에 흩어져 있는 데이터 조각들을 각 서버에서 동시에 **계산**하는 엔진.

스파크는 데이터가 저장된 바로 그 서버에서 계산을 수행하는 **'데이터 지역성(Data Locality)'**을 활용해 네트워크 전송 비용을 아낍니다. "야, 1번 서버에 있는 데이터 조각, 네가 거기서 바로 계산해서 결과만 알려줘!"라고 명령하는 식이죠.

# 스파크 코드 예시

스파크는 보통 **Python(PySpark)**을 가장 많이 사용합니다. 과거의 저수준 방식인 **RDD**와 요즘 대세인 고수준 방식인 **DataFrame** 두 가지 예시를 통해 스파크가 데이터를 어떻게 요리하는지 보여드릴게요.

---

## 1. 현대적인 방식: DataFrame API (추천)

마치 SQL이나 Pandas를 쓰는 느낌입니다. 내부적으로 스파크가 가장 많이 최적화(DAG 생성)를 해주는 방식입니다.

```python
from pyspark.sql import SparkSession
from pyspark.sql.functions import col, desc

# 1. 팀장님(Driver) 소환: 스파크 세션 시작
spark = SparkSession.builder \
    .appName("LogAnalysis") \
    .get_OrCreate()

# 2. 데이터 불러오기 (HDFS나 S3에서 가져온다고 가정)
# 실제로는 spark.read.csv("hdfs://...") 처럼 씁니다.
data = [
    ("216.58.210.78", "/index.html", 200),
    ("216.58.210.78", "/css/style.css", 200),
    ("172.217.161.142", "/index.html", 200),
    ("216.58.210.78", "/cart", 404),
    ("1.1.1.1", "/index.html", 200)
]
df = spark.createDataFrame(data, ["ip", "path", "status"])

# 3. 변환(Transformation): 아직 계산 안 함! 지도만 그리는 중 (Lazy Evaluation)
# - 상태코드가 200인 것만 필터링해서 IP별로 개수 세기
result_df = df.filter(col("status") == 200) \
              .groupBy("ip") \
              .count() \
              .orderBy(desc("count"))

# 4. 액션(Action): "자, 이제 계산 시작해!" (실제 실행)
result_df.show()

# 세션 종료
spark.stop()

```

---

## 2. 고전적인 방식: RDD (저수준 API)

앞서 언급한 **MapReduce**의 철학이 그대로 담긴 방식입니다. "Hello World" 격인 **단어 세기(Word Count)** 예시입니다.

```python
# 텍스트 파일을 한 줄씩 읽어서 RDD 생성
text_file = spark.sparkContext.textFile("hdfs://path/to/large_text.txt")

# 함수형 프로그래밍 스타일로 데이터 가공
word_counts = text_file.flatMap(lambda line: line.split(" ")) \
                       .map(lambda word: (word, 1)) \
                       .reduceByKey(lambda a, b: a + b)

# 결과 출력 (Action)
for word, count in word_counts.take(10):
    print(f"{word}: {count}")

```

---

## 💡 코드에서 눈여겨볼 점

1. **메서드 체이닝:** `.filter().groupBy().count()` 처럼 작업을 줄줄이 엮습니다. 스파크는 이걸 보고 **"아, 필터링 먼저 하고 그룹화하는 게 효율적이겠네?"** 하고 스스로 순서를 최적화합니다.
2. **분산 저장의 힘:** 위 코드에서 `df`나 `text_file`은 내 컴퓨터 메모리에 있는 게 아닙니다. 클러스터 수십 대에 쪼개져 있는 상태이고, 스파크는 그 조각들에 명령을 동시에 내리는 것입니다.
3. **HDFS와의 연결:** `spark.read.csv("hdfs://...")` 한 줄이면, 스파크가 HDFS의 네임노드(장부)에게 물어보고 수백 대의 데이터노드에서 파일 조각들을 동시에 빨아들입니다.

# 중간에 서로 모두 결과를 알아야하는 groupby 같은건 어떻게 도는거지? 중앙에서 합치나

아주 날카로운 질문입니다! 맞습니다. `groupBy`나 `join`처럼 **"나뿐만 아니라 다른 놈들이 가진 데이터도 다 확인해야 하는 작업"**은 분산 처리에서 가장 어렵고 비용이 많이 드는 단계입니다.

결론부터 말씀드리면, **중앙(Driver)에서 한꺼번에 합치지 않습니다.** 만약 수백 테라의 데이터를 중앙 서버 한 대에서 합치려 하면 그 서버는 바로 터져버리겠죠(OOM).

대신 스파크는 **셔플(Shuffle)**이라는 과정을 통해 **"끼리끼리 모이는 새로운 그룹핑"**을 수행합니다. 그 원리를 단계별로 설명해 드릴게요.

---

### 1단계: 로컬에서 먼저 요약 (Map-side Combine)

일단 각 일꾼(Executor)은 자기가 가진 데이터 안에서 최대한 할 수 있는 걸 먼저 합니다.

- 예를 들어 `A`라는 키의 개수를 세는 `groupBy`라면, 1번 일꾼은 자기한테 있는 `A`가 10개라는 걸 미리 계산해 둡니다. (이걸 해야 네트워크로 보낼 데이터 양이 줄어듭니다.)

### 2단계: 꼬표 던지기 (Partitioning & Hashing)

이제 "누가 어떤 키를 담당할지" 규칙을 정합니다. 보통 **해시(Hash) 함수**를 씁니다.

- 규칙 예: `hash(Key) % 3(일꾼 수)`
- 결과가 `0`이면 0번 일꾼에게, `1`이면 1번 일꾼에게 보내기로 약속하는 거죠.

### 3단계: 진짜 셔플 (The Shuffle - 핵심)

이제 네트워크를 통해 데이터가 대이동을 시작합니다.

- 모든 일꾼이 서로에게 데이터를 던집니다. 0번 일꾼은 모든 일꾼으로부터 "해시 결과가 0인 데이터 조각"들을 다 받아옵니다.
- 이때 데이터가 거미줄처럼 얽히며 이동하기 때문에 **셔플(Shuffle, 카드를 섞다)**이라고 부릅니다. 이 과정에서 네트워크 대역폭을 가장 많이 잡아먹고 속도가 느려집니다.

### 4단계: 최종 합체 (Reduce-side Merge)

이제 각 일꾼의 손에는 **"특정 키(예: 'A')에 해당하는 전 세계의 모든 조각들"**이 모였습니다.

- 0번 일꾼은 이제 안심하고 자기 손에 모인 조각들만 다 더하면, 전 클러스터를 통틀어 `A`가 몇 개인지 정확히 알 수 있게 됩니다.
- 이 결과를 다시 HDFS에 쓰거나 드라이버에게 보냅니다.

---

### 💡 왜 중앙(Driver)에서 안 합치나요?

1. **메모리 한계:** 드라이버는 보통 한 대입니다. 수천 대의 일꾼이 보낸 데이터를 한 대의 메모리에 다 올릴 수 없습니다.
2. **네트워크 병목:** 모든 데이터가 한 곳으로 쏠리면 네트워크 망이 마비됩니다.
3. **병렬 처리 상실:** 중앙에서 합치는 순간, 그동안 자랑하던 "분산 처리"의 장점이 사라지고 "싱글 스레드" 작업이 되어버립니다.

### 🍱 비유로 이해하기: "전국 지점 매출 집계"

전국 100개 편의점의 상품별 매출을 집계한다고 해봅시다.

- **나쁜 방법 (중앙 합체):** 모든 점장이 본사 사장님 한 명에게 영수증을 다 보냄 → 사장님 과로사.
- **스파크 방법 (셔플):** 1. 각 지점에서 "삼각김밥 10개, 라면 5개" 식으로 먼저 요약합니다.

2. 약속을 정합니다. "과자류는 강남점으로, 음료수는 강북점으로 다 모여!"
3. 각 지점은 요약된 장부를 들고 담당 지점으로 찾아갑니다. (셔플)
4. 강남 점장은 전국에서 온 과자 장부만 합치고, 강북 점장은 음료 장부만 합칩니다.
5. 마지막에 각 점장이 합계만 본사에 보고합니다.

---

**"셔플이 일어나면 성능이 떨어진다"**는 말이 바로 이 데이터 이동 비용 때문입니다. 그래서 스파크 고수들은 이 셔플을 최소화하도록 코드를 짜는 데 집중하곤 하죠.
