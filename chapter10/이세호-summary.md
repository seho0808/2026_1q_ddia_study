# 서문

## 1. 온라인 시스템 vs. 일괄 처리 시스템

데이터 시스템은 크게 사용자의 요청에 즉각 반응하는 **온라인 시스템**과 대량의 데이터를 몰아서 처리하는 **오프라인(일괄 처리) 시스템**으로 나뉩니다.

| 구분            | 온라인 시스템 (Online)          | 일괄 처리 시스템 (Batch/Offline)       |
| --------------- | ------------------------------- | -------------------------------------- |
| **주요 작업**   | 요청(Request) 및 응답(Response) | 대규모 계산, 데이터 변환, 분석         |
| **예시**        | 웹 브라우징, API 호출, DB 쿼리  | AI 모델 학습, ETL, 대규모 분석         |
| **성능 지표**   | **응답 시간(Response Time)**    | **처리량(Throughput)**                 |
| **데이터 특징** | 데이터 수정(Mutation) 빈번      | 입력 데이터는 읽기 전용(Immutable)     |
| **종료 시점**   | 실시간으로 지속됨               | 작업 완료 시 종료 (수 분 ~ 수 일 소요) |

---

## 2. 일괄 처리의 핵심 철학: 불변성(Immutability)

일괄 처리의 가장 큰 특징은 입력 데이터를 변경하지 않고, 결과물을 처음부터 다시 생성한다는 점입니다. 이를 통해 다음과 같은 강력한 이점을 얻습니다.

- **인적 결함 내성 (Human Fault Tolerance):** 코드에 버그가 있어 출력이 잘못되었다면, 코드를 수정하고 작업을 다시 실행하면 됩니다. 기존 데이터를 덮어쓰지 않으므로 복구가 매우 쉽습니다.
- **시간 여행 (Time Travel):** 과거의 출력물을 보관하고 있다면 언제든 이전 상태로 되돌아갈 수 있습니다.
- **비가역성 최소화:** 실수가 치명적인 결과를 초래하지 않으므로 애자일(Agile)한 개발과 실험이 가능합니다.
- **효율적인 자원 사용:** 일반적인 트랜잭션 DB(OLTP)보다 대용량 데이터를 처리할 때 컴퓨팅 자원을 훨씬 효율적으로 사용합니다.

---

## 3. 기술의 진화: MapReduce에서 현대적 프레임워크까지

일괄 처리 기술은 지난 20년간 급격히 발전해 왔습니다.

### 1단계: MapReduce의 시대 (2004년~)

- Google이 발표한 알고리즘으로, 범용 하드웨어에서 대규모 병렬 처리를 가능하게 했습니다.
- Hadoop, CouchDB 등의 기반이 되었으나, 현재는 낮은 추상화 수준과 성능 문제로 인해 구글 내에서도 거의 사용되지 않는 **오래된 기술**이 되었습니다.

### 2단계: 현대적 프레임워크의 등장

- **Spark, Flink:** MapReduce보다 정교한 캐싱과 실행 전략을 사용하여 성능을 극대화했습니다.
- **데이터 흐름(Dataflow) API:** 단순한 Map/Reduce 모델을 넘어 DataFrame API, 쿼리 언어 등 사용성이 높은 모델을 지원합니다.

### 3단계: 클라우드와 오케스트레이션

- **저장소의 변화:** 분산 파일 시스템(HDFS)에서 S3와 같은 **객체 스토리지(Object Storage)**로 이동했습니다.
- **워크플로 관리:** 과거의 Oozie, Azkaban 대신 **Airflow, Dagster, Prefect**와 같이 복잡한 작업 흐름을 관리하는 도구들이 주류가 되었습니다.
- **경계의 모호함:** Snowflake, BigQuery 같은 클라우드 데이터 웨어하우스가 발전하면서 일괄 처리와 데이터 웨어하우스 간의 경계가 사라지고 있습니다.

> "시스템은 한 사람의 영향력에 너무 강하게 휘둘리면 성공할 수 없다." (Donald Knuth)

이 문구처럼 일괄 처리 시스템은 다양한 관점의 실험을 수용할 수 있는 견고한 구조를 지향합니다. 비록 실시간성은 떨어지지만, **데이터의 정합성을 보장하고 복구 가능성을 높이며 대규모 데이터를 통합**하는 데 있어 대체 불가능한 도구입니다.

# Batch Processing with Unix Tools

## 1. 실전 예제: 웹사이트 인기 페이지 TOP 5 추출

웹 서버의 거대한 액세스 로그 파일에서 가장 많이 요청된 페이지 5개를 찾는 과정을 Unix 파이프라인으로 구현한 예시입니다.

### [Unix 명령행 분석]

```bash
cat /var/log/nginx/access.log | awk '{print $7}' | sort | uniq -c | sort -rn | head -n 5

```

| 단계  | 명령어             | 역할                                                     |
| ----- | ------------------ | -------------------------------------------------------- |
| **1** | `cat`              | 로그 파일을 읽어 스트림으로 보냄                         |
| **2** | `awk '{print $7}'` | 각 행에서 7번째 필드(요청 URL)만 추출                    |
| **3** | `sort`             | URL을 알파벳순으로 정렬 (중복된 URL들이 모이게 됨)       |
| **4** | `uniq -c`          | 인접한 중복 행을 합치고, 각 행의 출현 횟수(Count)를 출력 |
| **5** | `sort -rn`         | 숫자 기준(`-n`)으로 역순(`-r`, 큰 숫자부터) 정렬         |
| **6** | `head -n 5`        | 상위 5개 결과만 남기고 나머지는 버림                     |

---

## 2. Unix 도구 vs. 사용자 정의 프로그램 (Python)

두 방식은 겉보기엔 취향 차이 같지만, **실행 흐름(Execution Flow)**에서 결정적인 차이가 있습니다.

- **Python (인메모리 해시 테이블 방식):**
- URL을 키로, 카운트를 값으로 하는 **해시 테이블(딕셔너리)**을 메모리에 유지합니다.
- **장점:** 고유 URL 개수가 적을 때 매우 빠릅니다.
- **단점:** 고유 URL 개수가 너무 많아 메모리(RAM) 용량을 초과하면 프로그램이 중단됩니다.

- **Unix 파이프라인 (정렬 기반 방식):**
- 해시 테이블 대신 **정렬(Sorting)**에 의존합니다.
- **장점:** 메모리보다 큰 데이터셋도 처리 가능합니다. GNU `sort`는 메모리가 부족하면 자동으로 디스크를 사용(Spilling to disk)하며 여러 CPU 코어를 병렬로 활용합니다.
- **결과:** 수 기가바이트의 로그도 수 초 내에 안정적으로 처리합니다.

---

## 3. 핵심 기술 통찰: 정렬 vs. 해싱

이 비교는 데이터 시스템 설계에서 매우 중요한 개념입니다.

> **작업 세트(Working Set)의 크기**가 핵심입니다.
>
> - 고유 항목(Unique URLs)의 수가 메모리에 들어갈 정도라면 **해시 테이블**이 효율적입니다.
> - 데이터가 메모리보다 훨씬 크다면, **병합 정렬(Merge Sort)** 방식이 디스크 순차 읽기/쓰기 패턴을 활용하므로 훨씬 확장성이 좋습니다.

---

## 4. Unix 도구의 한계와 다음 단계

Unix 도구는 매우 강력하지만 결정적인 한계가 있습니다.

1. **단일 머신 제약:** 오직 한 대의 컴퓨터 자원만 사용할 수 있습니다.
2. **확장성:** 데이터가 단일 머신의 디스크나 CPU 처리 능력을 넘어서면 대응이 어렵습니다.

**이 지점이 바로 MapReduce와 같은 분산 일괄 처리 프레임워크가 등장하는 지점입니다.** 여러 대의 컴퓨터에 데이터를 나누어 저장하고 병렬로 Unix와 유사한 로직을 수행하는 것이 다음 논의의 핵심입니다.

# Batch Processing in Distributed Systems

## 1. 분산 파일 시스템 (DFS) vs. 객체 스토리지 (Object Store)

분산 환경에서 데이터를 저장하는 방식은 크게 두 가지로 나뉩니다.

### 분산 파일 시스템 (예: HDFS)

- **블록 단위 저장:** 파일을 거대한 블록(예: 128MB)으로 쪼개어 여러 **데이터 노드(DataNode)**에 분산 저장합니다. 블록이 클수록 관리해야 할 메타데이터가 줄어들고 디스크 탐색 비용이 낮아집니다.
- **데이터 지역성 (Data Locality):** 네트워크 대역폭을 아끼기 위해 데이터가 있는 서버에서 바로 계산을 수행할 수 있습니다.
- **메타데이터 관리:** **이름 노드(NameNode)**나 별도의 키-값 저장소를 통해 파일 위치와 권한 정보를 관리합니다.

### 객체 스토리지 (예: Amazon S3)

- **URL 기반 접근:** `s3://bucket/key` 형태의 고유 주소를 가집니다.
- **불변성(Immutability):** 파일을 수정하려면 전체를 다시 써야 하며, 디렉토리 개념이 없습니다(슬래시`/`는 단순히 파일 이름의 일부).
- **비원자적 이름 변경:** 파일 이름을 바꾸려면 복사 후 삭제해야 하므로 큰 디렉토리의 이름을 바꾸는 작업이 매우 무겁습니다.
- **분리된 계층:** 저장소와 계산 자원이 분리되어 있어 각각 독립적인 확장이 가능합니다.

| 비교 항목     | 분산 파일 시스템 (DFS)                 | 객체 스토리지 (Object Store)      |
| ------------- | -------------------------------------- | --------------------------------- |
| **최적화**    | 대용량 데이터, 데이터 지역성 기반 계산 | 확장성, 유연성, 클라우드 네이티브 |
| **수정 방식** | 추가(Append) 지원 가능                 | 전체 다시 쓰기 (Overwrite)        |
| **디렉토리**  | 실제 계층 구조 존재                    | 접두사(Prefix) 기반의 가상 구조   |

---

## 2. 분산 작업 오케스트레이션 (Job Orchestration)

Unix 커널이 단일 머신의 자원을 관리하듯, **Kubernetes**나 **YARN** 같은 오케스트레이터는 클러스터 전체의 자원을 관리합니다.

- **리소스 매니저:** 클러스터 전체의 CPU, 메모리, GPU 가용 상태를 파악하는 "두뇌" 역할을 합니다.
- **스케줄러:** 작업 요청을 받고 어떤 노드에서 실행할지 결정합니다. 이는 **NP-Hard** 문제(최적해를 구하기 매우 어려운 문제)이므로, 실제로는 휴리스틱 알고리즘(FIFO, 우선순위 큐 등)을 사용합니다.
- **태스크 실행기 (Executor):** 각 노드에서 실제 코드를 실행하고 상태를 보고합니다 (예: `kubelet`, `NodeManager`).

---

## 3. 워크플로와 DAG (Directed Acyclic Graph)

단일 Unix 명령어를 파이프(`|`)로 잇는 것처럼, 분산 환경에서는 여러 작업(Job)을 연결하여 **워크플로(Workflow)**를 만듭니다.

- **DAG 구조:** 작업 간의 의존 관계를 방향성 있는 비순환 그래프로 표현합니다.
- **관리 도구:** **Airflow, Dagster, Prefect** 등이 사용되며, 이전 단계의 작업이 성공해야만 다음 단계가 실행되도록 제어합니다.
- **데이터 전달:** Unix 파이프는 메모리 버퍼를 쓰지만, 분산 워크플로는 보통 중간 결과물을 DFS나 객체 스토리지에 기록하여 작업 간 결합도를 낮춥니다.

---

## 4. 결함 내성 (Fault Tolerance)

일괄 처리 시스템은 온라인 시스템보다 장애 대응이 유연합니다.

- **재실행:** 작업이 실패하면 다른 노드에서 다시 실행하면 그만입니다. 입력 데이터가 불변이기 때문에 결과는 동일합니다.
- **스팟 인스턴스 (Spot Instances):** 언제든 중단될 수 있는 저렴한 서버를 활용하기에 적합합니다. 스케줄러에 의해 작업이 강제 종료(Preemption)되어도 다시 시작하면 되기 때문입니다.
- **중간 데이터 처리:**
  - **MapReduce:** 중간 결과를 매번 DFS에 써서 안정성을 확보하지만 느립니다.
  - **Spark/Flink:** 중간 결과를 메모리에 두거나 체크포인트를 활용하여 성능과 안정성 사이의 균형을 맞춥니다.

### 요약 및 통찰

분산 일괄 처리는 결국 **"거대한 데이터를 어떻게 효율적으로 쪼개고(DFS), 누가 어디서 처리할지 결정하며(Scheduler), 장애 시 어떻게 안전하게 다시 시작할 것인가(Fault Tolerance)"**에 대한 해답을 찾아가는 과정입니다.

# Batch Processing Models

## 1. MapReduce: 일괄 처리의 조상

MapReduce는 Unix 도구 체인(`awk | sort | uniq`)의 논리를 분산 환경으로 확장한 모델입니다.

- **Map 단계:** 입력 레코드를 읽어 **키-값(Key-Value) 쌍**을 추출합니다. (상태 비저장)
- **Shuffle 단계:** 모든 키-값 쌍을 키 기준으로 정렬하여, 동일한 키를 가진 데이터들이 동일한 리듀서로 모이도록 보냅니다.
- **Reduce 단계:** 정렬된 키와 그에 해당하는 값들의 리스트를 반복(Iterate)하며 최종 결과(합계, 카운트 등)를 생성합니다.

> **한계:** 작업 간의 데이터를 항상 분산 파일 시스템(HDFS 등)에 써야 하므로 속도가 느리고, 조인(Join) 같은 복잡한 로직을 직접 구현하기가 매우 까다롭습니다.

---

## 2. 데이터플로 엔진 (Spark, Flink)

MapReduce의 단점을 보완하기 위해 등장한 현대적인 엔진들입니다. 전체 워크플로를 하나의 거대한 작업으로 처리합니다.

- **유연성:** Map -> Reduce라는 엄격한 교차 구조 대신, 필터링, 조인, 그룹화 등 다양한 연산자를 자유롭게 배치할 수 있습니다.
- **최적화:**
  - **연산자 병합:** 셔플이 필요 없는 여러 단계(Map, Filter 등)를 하나의 태스크로 묶어 오버헤드를 줄입니다.
  - **메모리 활용:** 중간 결과물을 디스크가 아닌 메모리에 유지하여 I/O 비용을 획기적으로 낮춥니다.
  - **파이프라이닝:** 이전 단계가 다 끝나기 전에도 데이터가 준비되는 대로 다음 연산자가 실행될 수 있습니다.

---

## 3. 핵심 알고리즘: 셔플(Shuffle)과 조인(Join)

대규모 데이터를 정렬하고 합치는 과정은 일괄 처리의 심장부와 같습니다.

### 셔플(Shuffle)

- 분산 환경에서의 **정렬 알고리즘**입니다.
- 각 매퍼가 리듀서별로 파일을 만들고, 리듀서가 이를 가져와서 병합 정렬(Merge Sort)을 수행합니다. 이를 통해 페타바이트급 데이터도 정렬할 수 있습니다.

### 정렬-병합 조인(Sort-Merge Join)

1. **Map:** 두 테이블(예: 활동 로그와 사용자 DB)에서 조인 키(User ID)를 추출합니다.
2. **Shuffle:** 동일한 User ID를 가진 데이터들이 동일한 리듀서로 모이도록 정렬합니다.
3. **Reduce:** 리듀서가 정렬된 데이터를 읽으며 두 정보를 합칩니다. (이때 보조 정렬을 통해 사용자 정보를 활동 로그보다 먼저 읽게 할 수 있습니다.)

---

## 4. 프로그래밍 모델의 진화: SQL과 데이터프레임

기술의 초점이 "어떻게 실행하느냐"에서 **"어떻게 쉽게 작성하느냐"**로 이동했습니다.

- **SQL:** 모든 분석가와 개발자에게 익숙한 언어입니다. 쿼리 최적화기(Optimizer)가 SQL을 효율적인 물리적 실행 계획으로 자동 변환해 줍니다.
- **데이터프레임(DataFrames):** R과 Pandas에서 유래된 모델로, 표 형식의 데이터를 함수 호출(join, filter 등) 방식으로 조작합니다.
- **Spark/Daft:** 사용자 코드를 즉시 실행하지 않고, 전체 쿼리 계획을 세워 최적화한 후 분산 실행합니다.
- **클라우드 데이터 웨어하우스와의 결합:** BigQuery나 Snowflake 같은 시스템들도 이제 SQL뿐만 아니라 데이터프레임 API를 지원하며 일괄 처리 시스템과 경계가 허물어지고 있습니다.

---

### 요약: MapReduce vs 현대적 데이터플로

| 특징                 | MapReduce                  | 데이터플로 엔진 (Spark/Flink)      |
| -------------------- | -------------------------- | ---------------------------------- |
| **중간 데이터 저장** | 항상 디스크 (HDFS/S3)      | 주로 메모리, 필요시 로컬 디스크    |
| **실행 구조**        | 고정된 Map -> Reduce 단계  | 유연한 DAG (방향성 비순환 그래프)  |
| **프로그래밍**       | 저수준 API (직접 구현)     | SQL, 데이터프레임 등 고수준 API    |
| **성능**             | 상대적으로 느림 (I/O 병목) | 매우 빠름 (최적화 및 파이프라이닝) |

# Batch Use Cases

## 1. ETL (추출, 변환, 적재)

일괄 처리는 운영 데이터베이스에서 데이터를 가져와 분석용 시스템으로 옮기는 ETL 작업에 가장 널리 쓰입니다.

- **병렬 처리의 효율성:** 데이터 필터링이나 필드 추출 같은 작업은 "당혹울 정도로 병렬적(Embarrassingly Parallel)"인 특성이 있어 일괄 처리 시스템에서 매우 빠르게 수행됩니다.
- **강력한 스케줄링:** **Airflow, Dagster, Prefect**와 같은 워크플로 스케줄러를 통해 복잡한 단계를 관리하며, 실패 시 재시도(Retry) 기능을 통해 파이프라인의 신뢰성을 보장합니다.
- **데이터 거버넌스:** 최근에는 **데이터 메시(Data Mesh)**나 **데이터 계약(Data Contract)** 같은 표준을 도입해 조직 내 여러 팀이 스스로 파이프라인을 관리하고 데이터를 공유하는 구조로 발전하고 있습니다.

## 2. 분석 (Analytics) 및 데이터 레이크하우스

분석 쿼리(OLAP)를 수행할 때도 일괄 처리 엔진이 핵심적인 역할을 합니다.

- **데이터 레이크하우스:** **Apache Iceberg**와 같은 테이블 포맷을 사용해 객체 스토리지(S3 등)에 저장된 대량의 데이터를 SQL로 직접 쿼리합니다.
- **사전 집계(Pre-aggregation):** 쿼리 속도를 높이기 위해 주기적으로 데이터를 요약하여 OLAP 큐브나 전용 시스템(Druid, Pinot 등)에 미리 적재해 둡니다.
- **애드혹(Ad-hoc) 분석:** 분석가가 비즈니스 문제를 해결하기 위해 즉석에서 실행하는 쿼리 역시 일괄 처리 프레임워크의 빠른 실행 엔진을 활용합니다.

## 3. 머신러닝(ML) 및 LLM

AI 모델을 만들고 운영하는 과정은 거대한 일괄 처리 작업의 연속입니다.

- **피처 엔지니어링:** 원시 데이터를 모델 학습에 적합한 수치형 데이터(벡터)로 변환합니다.
- **모델 학습 및 일괄 추론:** 대규모 데이터셋을 입력받아 모델 가중치를 계산하거나, 실시간 응답이 필요 없는 대량의 예측 데이터를 한꺼번에 생성합니다.
- **LLM 데이터 준비:** 웹사이트의 텍스트를 추출하고, 중복을 제거하며, 단어를 **임베딩(Embedding)**으로 변환하는 전처리 과정에 **Ray**나 **Kubeflow** 같은 프레임워크가 사용됩니다.
- **그래프 처리:** 추천 엔진 등에 쓰이는 복잡한 관계 데이터를 처리하기 위해 **Pregel** 모델이나 **Spark GraphX** 같은 일괄 처리 도구를 활용합니다.

## 4. 파생 데이터 제공 (Serving Derived Data)

일괄 처리로 계산된 결과(추천 목록, 보고서 등)를 실제 서비스용 DB로 옮기는 과정입니다.

- **직접 쓰기의 위험성:** 일괄 처리 작업이 운영 DB에 직접 데이터를 쓰면 네트워크 오버헤드가 크고, 운영 중인 서비스의 성능을 떨어뜨리며, 실패 시 원자성을 보장하기 어렵습니다.
- **권장 솔루션:**
  - **스트리밍 시스템(Kafka) 활용:** 일괄 처리 결과를 Kafka로 보내 완충(Buffer) 역할을 하게 하고, DB가 이를 안정적으로 소비하게 합니다.
  - **벌크 로드(Bulk Load):** DB 전용 파일 포맷을 일괄 처리 단계에서 직접 생성한 후, DB의 벌크 임포트 기능을 통해 한꺼번에 갈아 끼우는 방식입니다. 매우 빠르고 안정적입니다.

### 결론 및 통찰

일괄 처리는 낮은 지연 시간보다는 **대규모 처리량(Throughput)**과 **결함 내성(Fault Tolerance)**에 최적화되어 있습니다. "언제든 다시 돌려도 똑같은 결과가 나온다"는 불변성의 원칙 덕분에, 데이터가 거대해질수록 시스템의 안정성을 유지하는 가장 강력한 수단이 됩니다.

# 요약

## 1. 일괄 처리의 기초: 유닉스(Unix) 철학

일괄 처리의 근본 원리는 멀리 있지 않습니다. 우리가 흔히 쓰는 유닉스 명령행 도구에 그 정수가 담겨 있습니다.

- **기본 프리미티브:** `awk`, `sort`, `uniq` 등은 데이터를 정렬하고 숫자를 세는 일괄 처리의 가장 기본적인 동작을 보여줍니다.
- **불변성과 재현 가능성:** 입력 데이터는 **읽기 전용(Immutable)**이며, 작업 결과는 항상 새로운 출력을 생성합니다. 이 덕분에 버그가 발생해도 부작용 없이 언제든 작업을 다시 실행(Rerun)하고 디버깅할 수 있습니다.

---

## 2. 분산 일괄 처리의 3대 요소

시스템 규모가 커지면서 일괄 처리 프레임워크는 크게 세 가지 계층으로 정교화되었습니다.

| 계층                               | 역할                                           | 주요 기술                |
| ---------------------------------- | ---------------------------------------------- | ------------------------ |
| **저장소 (Storage)**               | 대용량 파일을 블록 단위로 복제하고 관리        | HDFS, S3 (객체 스토리지) |
| **오케스트레이션 (Orchestration)** | 작업의 실행 시점, 위치 결정 및 자원 할당       | YARN, Kubernetes (K8s)   |
| **계산 (Computation)**             | 실제 데이터를 읽고 변환하는 비즈니스 로직 수행 | MapReduce, Spark, Flink  |

> **참고:** 요약문에서는 단일 작업의 자원을 관리하는 **작업 오케스트레이터(Job Orchestrator)**와 여러 작업 간의 의존성(DAG)을 관리하는 **워크플로 오케스트레이터(Workflow Orchestrator, 예: Airflow)**를 구분하여 설명했습니다.

---

## 3. 처리 모델의 진화: MapReduce에서 데이터플로까지

데이터를 처리하는 방식은 더 유연하고 성능이 뛰어난 방향으로 진화했습니다.

- **MapReduce:** `Map`(데이터 추출)과 `Reduce`(데이터 집계)라는 정형화된 함수를 사용합니다.
- **데이터플로 엔진 (Spark, Flink):** 연산자를 보다 자유롭게 배치할 수 있는 데이터플로 API를 제공하며, 중간 결과물을 메모리에 유지하여 성능을 비약적으로 향상시켰습니다.
- **셔플(Shuffle) 알고리즘:** 일괄 처리의 핵심입니다. 서로 다른 노드에 흩어진 데이터를 그룹화, 조인, 집계하기 위해 특정 키를 기준으로 데이터를 재배치하는 과정입니다.

---

## 4. 사용성 개선: SQL과 데이터프레임

엔지니어뿐만 아니라 분석가들도 대규모 데이터를 다룰 수 있도록 인터페이스가 고도화되었습니다.

- **선언적 쿼리:** SQL과 DataFrame API는 "어떻게(How)" 처리할지가 아니라 "무엇을(What)" 원하는지 기술합니다.
- **쿼리 최적화기(Query Optimizer):** 선언적인 쿼리를 받아 클러스터에서 가장 효율적으로 실행될 수 있는 물리적 계획으로 자동 변환해 줍니다.

---

## 5. 일괄 처리의 4가지 핵심 유스케이스

이 장의 마지막은 일괄 처리가 실제 산업에서 어떻게 쓰이는지로 마무리됩니다.

1. **ETL 파이프라인:** 시스템 간 데이터 이동 및 변환.
2. **데이터 분석:** 사전 집계된 대시보드 생성 및 분석가의 애드혹(Ad-hoc) 쿼리 지원.
3. **머신러닝:** 대규모 학습 데이터셋 전처리 및 모델 훈련.
4. **파생 데이터 생성:** 일괄 처리 결과를 운영 시스템(DB, 검색 엔진 등)에 벌크 로딩하여 사용자에게 서비스.

---

## 6. 다음 장으로의 연결: 유계(Bounded)에서 무계(Unbounded)로

11장에서 다룬 데이터는 **유계(Bounded)** 데이터, 즉 끝이 정해진 데이터셋이었습니다.

- **일괄 처리:** 데이터가 모두 모일 때까지 기다린 후 처리를 시작하며, 작업이 끝나면 종료됩니다.
- **스트림 처리 (12장):** 입력 데이터가 끝이 없는 **무계(Unbounded)** 데이터입니다. 작업은 종료되지 않고 데이터가 들어오는 대로 계속 실행됩니다.
