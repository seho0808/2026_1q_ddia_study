# DDIA Chapter 10: 일괄 처리 (Batch Processing)


## 1. 세 가지 시스템 유형

서비스, 일괄 처리 시스템, 스트림 처리 시스템은 데이터를 처리하는 방식에 따라 구분된다.

| 유형 | 입력 | 출력 | 지연 |
|------|------|------|------|
| **서비스 (온라인)** | 클라이언트 요청 | 응답 | 수 밀리초 |
| **일괄 처리 (오프라인)** | 유한한 대용량 데이터셋 | 결과 데이터 | 수 분~수 일 |
| **스트림 처리 (준실시간)** | 이벤트 스트림 | 이벤트 스트림 | 수 초 미만 |

### 일괄 처리의 특징

- **유한한 입력**: 데이터의 시작과 끝이 명확하다.
- **비동기적**: 사용자가 직접 기다리지 않으므로 처리량(throughput)이 중요하다.
- **결정론적**: 같은 입력에 대해 항상 같은 출력을 생성한다.

## 2. 유닉스 도구로 일괄 처리하기

### 로그 분석 예시

웹 서버 로그에서 가장 많이 방문한 URL 상위 5개를 구하는 명령어:

```bash
cat /var/log/nginx/access.log |
  awk '{print $7}' |
  sort             |
  uniq -c          |
  sort -r -n       |
  head -n 5
```

### 유닉스 철학

유닉스 파이프라인은 일괄 처리의 원형이다. 핵심 원칙은 다음과 같다:

1. **단일 목적**: 각 프로그램은 한 가지 일을 잘 수행한다.
2. **텍스트 스트림**: 프로그램 간 인터페이스는 표준화된 텍스트 스트림이다.
3. **조합 가능성**: 작은 프로그램을 파이프(`|`)로 조합하여 복잡한 작업을 처리한다.
4. **불변 입력**: 입력 데이터를 수정하지 않고 새로운 출력을 생성한다.

### 유닉스 도구의 한계

- **단일 머신**: 메모리와 디스크가 하나의 머신으로 제한된다.
- **병렬성 부족**: 다중 CPU 코어를 자동으로 활용하지 못한다.
- **분산 처리 불가**: 여러 머신에 걸친 대용량 데이터 처리가 어렵다.

→ 이 한계를 극복하기 위해 **맵리듀스(MapReduce)** 가 등장했다.

## 3. 맵리듀스와 분산 파일시스템

### HDFS (Hadoop Distributed File System)

맵리듀스는 분산 파일시스템을 기반으로 동작한다. HDFS는 GFS(Google File System)에서 영감을 받아 설계되었다.

```
┌─────────────────────────────────────┐
│           HDFS 아키텍처              │
│                                     │
│  NameNode (메타데이터 관리)           │
│     │                               │
│     ├─ DataNode 1 [블록 A, B, C]    │
│     ├─ DataNode 2 [블록 A, C, D]    │
│     └─ DataNode 3 [블록 B, D, E]    │
│                                     │
│  ※ 각 블록은 여러 노드에 복제됨       │
└─────────────────────────────────────┘
```

**특징:**
- 파일을 여러 블록(기본 128MB)으로 분할
- 각 블록을 여러 노드에 복제(기본 3벌) → 장애 내성
- 상용 하드웨어(commodity hardware)를 사용

### 맵리듀스 실행 흐름

```
입력 데이터
    │
    ▼
┌─────────┐
│  Map    │  → 키-값 쌍 생성
└─────────┘
    │
    ▼  (셔플/정렬)
┌─────────┐
│ Reduce  │  → 같은 키끼리 집계
└─────────┘
    │
    ▼
출력 데이터
```

### Map 단계

- 입력 데이터의 각 레코드를 처리하여 **키-값 쌍**을 출력
- 각 Mapper는 독립적으로 병렬 실행 (stateless)
- 입력 데이터의 파티션 수만큼 Mapper가 생성됨

```python
# 예: 단어 빈도 카운팅 - Map
def map(key, value):
    for word in value.split():
        emit(word, 1)
```

### Reduce 단계

- 같은 키를 가진 모든 값을 모아 집계
- 셔플(Shuffle) 단계에서 네트워크를 통해 데이터 이동

```python
# 예: 단어 빈도 카운팅 - Reduce
def reduce(key, values):
    emit(key, sum(values))
```

### 맵리듀스의 핵심 특징

| 특징 | 설명 |
|------|------|
| **데이터 지역성** | Mapper를 데이터가 있는 노드에서 실행 (네트워크 전송 최소화) |
| **장애 내성** | 실패한 태스크만 재실행 |
| **확장성** | 수천 대의 머신으로 수평 확장 |
| **멱등성** | 같은 태스크를 여러 번 실행해도 결과가 동일 |

## 4. 맵리듀스 워크플로우

### 조인(Join)

맵리듀스에서 여러 데이터셋을 조인하는 방식은 크게 두 가지다.

#### 4-1. 리듀스 사이드 조인 (Sort-Merge Join)

```
사용자 이벤트 로그          사용자 DB
user_id, event             user_id, name, age
─────────────────          ─────────────────
123, click                 123, Alice, 30
456, view                  456, Bob, 25
123, purchase

         │ Map: user_id를 키로 설정
         ▼
    셔플/정렬 (같은 user_id끼리 모임)

         │ Reduce: 두 데이터셋 병합
         ▼
123, Alice, 30, [click, purchase]
456, Bob, 25, [view]
```

**특징:**
- 모든 데이터가 네트워크를 통해 이동 → 비용이 큼
- 데이터 스큐(Skew) 문제 발생 가능 (특정 키에 데이터 집중)

#### 4-2. 맵 사이드 조인 (Broadcast Hash Join)

작은 데이터셋이 메모리에 올라갈 수 있을 때 사용:

```
작은 테이블(사용자 DB)       큰 테이블(이벤트 로그)
    → 모든 Mapper의         → 각 Mapper에서
      메모리에 로드            로컬로 조인 수행
```

**특징:**
- 셔플 없이 처리 → 매우 빠름
- 작은 테이블이 메모리에 올라갈 수 있어야 한다는 제약

### 그루핑과 집계

```
입력: (session_id, user_id, event)
Map: user_id를 키로 → (user_id, session_id) 출력
Reduce: 각 user_id에 대한 session 수 카운트
```

**데이터 스큐 처리:**
- 핫 키(hot key)가 있으면 특정 리듀서에 부하 집중
- 해결책: 랜덤 키를 붙여 부하 분산 후 두 번째 단계에서 최종 집계

## 5. 맵리듀스를 넘어서

### 맵리듀스의 단점

- **중간 결과를 HDFS에 저장**: 각 단계마다 디스크에 쓰고 읽는 과정이 반복됨
- **복잡한 워크플로우**: 여러 맵리듀스 작업을 체인으로 연결할 때 관리가 복잡
- **느린 속도**: 디스크 I/O로 인한 높은 지연

```
맵리듀스 체인:
MR1 → HDFS → MR2 → HDFS → MR3 → HDFS → 결과
      (쓰기)       (쓰기)       (쓰기)
```

### 데이터플로우 엔진 (Spark, Tez, Flink)

맵리듀스의 단점을 개선한 새로운 접근 방식:

```
Spark 실행 모델:
연산자1 → 연산자2 → 연산자3 → 결과
  (메모리 파이프라인, 필요할 때만 디스크 사용)
```

**개선점:**

| 항목 | 맵리듀스 | 데이터플로우 엔진 |
|------|---------|---------------|
| 중간 결과 | HDFS에 저장 | 메모리에 유지 |
| 파이프라인 | 불가 | 가능 |
| 연산자 종류 | Map/Reduce만 | 다양한 연산자 |
| 장애 복구 | 태스크 재실행 | 리니지(lineage) 기반 재계산 |
| 속도 | 느림 | 10~100배 빠름 |

### Apache Spark

```
RDD(Resilient Distributed Dataset) 기반:
- 불변(immutable) 분산 데이터 컬렉션
- 변환(transformation): lazy evaluation
- 액션(action): 실제 계산 트리거

val result = sc.textFile("input.txt")
  .flatMap(_.split(" "))
  .map((_, 1))
  .reduceByKey(_ + _)
  .collect()
```

### 장애 내성 비교

| 방식 | 장애 복구 전략 |
|------|------------|
| 맵리듀스 | 중간 결과를 HDFS에 저장 → 실패한 태스크만 재실행 |
| Spark | RDD 리니지 그래프로 손실된 파티션만 재계산 |
| 체크포인팅 | 특정 시점의 상태를 HDFS에 저장 (리니지가 너무 길 때) |

## 6. 고수준 API와 언어

### Hive와 Pig

직접 맵리듀스 코드를 작성하는 대신 선언적 언어로 처리:

```sql
-- Hive: SQL과 유사한 문법
SELECT user_id, COUNT(*) as event_count
FROM events
WHERE event_date = '2024-01-01'
GROUP BY user_id
ORDER BY event_count DESC
LIMIT 10;
```

```
-- Pig Latin
events = LOAD 'events' USING PigStorage(',');
filtered = FILTER events BY date == '2024-01-01';
grouped = GROUP filtered BY user_id;
counts = FOREACH grouped GENERATE group, COUNT(filtered);
DUMP counts;
```

### 특수 목적 시스템

| 시스템 | 용도 |
|--------|------|
| **Mahout** | 머신러닝 (맵리듀스 기반) |
| **Spark MLlib** | 머신러닝 (메모리 기반) |
| **GraphX / Pregel** | 그래프 처리 (PageRank 등) |
| **Impala / Presto** | 대화형 SQL 쿼리 (낮은 지연) |

### 분산 SQL 실행

```
쿼리 최적화 예시:

SELECT u.name, COUNT(e.id)
FROM users u JOIN events e ON u.id = e.user_id
WHERE e.type = 'purchase'
GROUP BY u.name

→ 옵티마이저가 자동으로 최적 실행 계획 수립:
   1. events 필터링 (type = 'purchase')
   2. users와 조인 (작은 테이블이면 맵사이드 조인)
   3. 집계
```

## 7. 핵심 요약

### 일괄 처리 시스템의 근본적 특성

```
┌─────────────────────────────────────────────┐
│            일괄 처리의 철학                   │
│                                             │
│  입력 데이터 = 불변(immutable)               │
│  출력 데이터 = 새로 생성                     │
│  부수효과 없음 = 순수 함수적 접근            │
│                                             │
│  → 재실행 가능 / 디버깅 용이 / 신뢰성 높음  │
└─────────────────────────────────────────────┘
```

### 발전 흐름

```
유닉스 파이프라인
       │ (아이디어 계승)
       ▼
  맵리듀스 (Hadoop)
       │ (단점 개선)
       ▼
데이터플로우 엔진 (Spark, Flink, Tez)
       │ (추상화)
       ▼
고수준 SQL/DSL (Hive, Spark SQL, Presto)
```

### 중요 개념 정리

| 개념 | 설명 |
|------|------|
| **데이터 지역성** | 계산을 데이터가 있는 곳으로 이동 (반대가 아닌) |
| **멱등성** | 같은 작업을 여러 번 실행해도 결과 동일 |
| **불변 입력** | 입력을 수정하지 않아 재실행 및 디버깅 용이 |
| **셔플** | 맵에서 리듀스로 데이터를 키 기준으로 재분배 |
| **데이터 스큐** | 특정 키에 데이터가 몰리는 현상 → 핫스팟 유발 |
| **리니지** | 데이터가 어떻게 만들어졌는지 추적 → 장애 복구에 사용 |

### 일괄 처리 vs 스트림 처리

> 일괄 처리는 **"유한한 데이터"** 를 처리하는 시스템이다.  
> 스트림 처리는 이 아이디어를 **"무한한 데이터 스트림"** 으로 확장한 것이다.
