# DDIA Chapter 11 — 스트림 처리 (Stream Processing)


## 개요

스트림 처리는 **이벤트가 발생하는 즉시** 처리하는 패러다임이다. 배치 처리가 고정된 데이터셋을 다루는 것과 달리, 스트림 처리는 끝이 없는(unbounded) 데이터를 실시간으로 다룬다.

---

## 1. 이벤트 스트림 전달 (Transmitting Event Streams)

### 이벤트란?
- 어떤 일이 발생했음을 기록한 **불변의 사실(immutable fact)**
- 타임스탬프 + 발생한 사건의 내용
- 예: 사용자 클릭, 주문 생성, 센서 측정값

### 메시지 브로커 (Message Broker / Message Queue)
생산자(Producer)와 소비자(Consumer) 사이의 중간 계층

| 특성 | 설명 |
|------|------|
| 비동기 처리 | 생산자와 소비자가 독립적으로 동작 |
| 내구성 | 메시지를 디스크에 저장 가능 |
| 부하 조절 | 소비 속도가 느려도 메시지 유실 없음 |
| 팬아웃 | 여러 소비자가 동일 메시지 수신 가능 |

#### 주요 메시지 브로커
- **AMQP/JMS 스타일**: RabbitMQ, ActiveMQ — 메시지 전달 후 삭제
- **로그 기반(Log-based)**: Apache Kafka, Amazon Kinesis — 메시지를 로그로 영구 보관

### 로그 기반 메시지 브로커

```
파티션 0: [이벤트1] [이벤트2] [이벤트3] [이벤트4] ...
파티션 1: [이벤트A] [이벤트B] [이벤트C] ...
파티션 2: [이벤트α] [이벤트β] ...
```

- 각 파티션은 **순서가 보장된 로그**
- 소비자는 **오프셋(offset)** 으로 읽은 위치를 추적
- 메시지를 소비해도 삭제되지 않음 → **재처리 가능**
- 처리량 = 디스크 순차 I/O 속도 (매우 빠름)

#### 소비자 그룹
```
토픽 파티션 [0, 1, 2, 3]
    ├── 소비자 그룹 A: 소비자1(파티션0,1), 소비자2(파티션2,3)
    └── 소비자 그룹 B: 소비자3(파티션0,1,2,3) — 독립적으로 처음부터 소비
```

## 2. 데이터베이스와 스트림 (Databases and Streams)

### 변경 데이터 캡처 (Change Data Capture, CDC)

DB의 변경사항을 스트림으로 내보내는 기법

```
MySQL/PostgreSQL
    → 바이너리 로그(binlog) 파싱
    → Kafka 토픽
    → 검색 인덱스(Elasticsearch), 캐시(Redis), 데이터 웨어하우스
```

- **파생 시스템(derived system)** 을 원본 DB와 동기화
- Debezium, Maxwell 같은 도구 활용
- DB를 **진실의 원천(source of truth)** 으로 유지

### 이벤트 소싱 (Event Sourcing)

애플리케이션 상태 변경을 **이벤트 로그로만** 저장하는 패턴

| 항목 | 일반 CRUD | 이벤트 소싱 |
|------|----------|------------|
| 저장 단위 | 현재 상태 | 이벤트 이력 |
| 과거 이력 | 일반적으로 소실 | 완전히 보존 |
| 상태 복원 | 직접 조회 | 이벤트 재생(replay) |
| 감사(Audit) | 별도 구현 필요 | 자동 |

```
이벤트 로그:
  [CartCreated] → [ItemAdded: 상품A] → [ItemAdded: 상품B] → [ItemRemoved: 상품A] → [OrderPlaced]

현재 상태 = 이벤트들을 순서대로 적용한 결과
```

### 불변성의 장점 (Immutability)
- 버그 발생 시 이벤트 재생으로 상태 복원 가능
- 여러 뷰(view)를 동일 이벤트로부터 파생 가능
- 실수로 삭제된 데이터도 복구 가능

## 3. 스트림 처리 (Processing Streams)

### 3가지 처리 패턴

#### 1) 이벤트 → 이벤트 (스트림 변환)
```
입력 스트림 → [필터/변환/집계] → 출력 스트림
```

#### 2) 이벤트 → DB 업데이트
```
이벤트 스트림 → [처리] → 검색 인덱스 / 캐시 / DB 업데이트
```

#### 3) 이벤트 → 사용자 알림
```
이벤트 스트림 → [처리] → 이메일 / 푸시 알림 / 대시보드
```

## 4. 스트림의 시간 문제 (Time in Stream Processing)

### 두 가지 시간 개념

| 시간 | 설명 | 예시 |
|------|------|------|
| **이벤트 발생 시간** (Event time) | 이벤트가 실제로 발생한 시각 | 사용자가 버튼을 누른 시각 |
| **처리 시간** (Processing time) | 시스템이 이벤트를 처리한 시각 | 서버가 요청을 받은 시각 |

### 문제: 지연 이벤트 (Straggler Events)
- 네트워크 지연, 오프라인 기기, 재시도 등으로 이벤트가 늦게 도착
- 처리 시간 기준으로 집계하면 이벤트 시간 기준 결과와 달라짐

### 해결: 워터마크 (Watermark)
```
워터마크 = 현재 처리 시간 - 허용 지연 시간(허용 범위)

워터마크보다 오래된 이벤트 = 지각생(straggler)으로 처리 or 무시
```

## 5. 윈도우 처리 (Windowing)

끝없는 스트림을 유한한 구간으로 나누어 집계

### 윈도우 유형

#### 텀블링 윈도우 (Tumbling Window)
```
|--- 1분 ---|--- 1분 ---|--- 1분 ---|
```
- 고정 크기, 겹침 없음
- 각 이벤트는 정확히 하나의 윈도우에 속함

#### 홉핑 윈도우 (Hopping Window)
```
|--- 5분 ---|
    |--- 5분 ---|
        |--- 5분 ---|
```
- 고정 크기, 일부 겹침
- 이동 간격(hop) < 윈도우 크기

#### 슬라이딩 윈도우 (Sliding Window)
- 새 이벤트가 들어올 때마다 윈도우가 이동
- "최근 5분 내의 이벤트" 같은 질의

#### 세션 윈도우 (Session Window)
```
[이벤트군1]  [이벤트군2]        [이벤트군3]
  ↑ 활동      ↑ 활동           ↑ 활동
         ↑ 비활동 구간     ↑ 비활동 구간
```
- 비활동 구간을 기준으로 세션 분리
- 사용자 행동 분석에 유용

## 6. 스트림 조인 (Stream Joins)

### 스트림-스트림 조인 (Stream-Stream Join)
```
클릭 스트림  ─┐
              ├─ 조인 (일정 시간 윈도우 내) ─→ 결과
검색 스트림  ─┘
```
- 두 스트림의 이벤트를 **시간 윈도우** 기준으로 조인
- 상태(state)를 일정 시간 유지해야 함

### 스트림-테이블 조인 (Stream-Table Join, 테이블 보강)
```
이벤트 스트림 (user_id=42, action=purchase)
    + DB 조회 (user_id=42 → name="홍길동", tier="VIP")
    → 보강된 이벤트 출력
```
- DB를 로컬에 캐시하거나 CDC로 동기화하여 저지연 조회

### 테이블-테이블 조인 (Table-Table Join)
- 두 CDC 스트림을 조인하여 구체화 뷰(materialized view) 유지

### 조인의 시간 의존성 문제
- 조인 결과는 **언제 처리되느냐**에 따라 달라질 수 있음
- 비결정적(non-deterministic) 가능성 존재

## 7. 내결함성 (Fault Tolerance)

### 문제: 정확히 한 번 처리 (Exactly-Once Semantics)

| 처리 보장 | 설명 | 위험 |
|----------|------|------|
| **최대 한 번** (at-most-once) | 실패 시 재처리 안 함 | 데이터 유실 |
| **최소 한 번** (at-least-once) | 실패 시 재처리 | 중복 처리 |
| **정확히 한 번** (exactly-once) | 중복 없이 정확히 처리 | 구현이 복잡 |

### 해결 방법

#### 마이크로배치 (Microbatching)
- 스트림을 작은 배치로 쪼개어 처리
- Spark Streaming이 사용하는 방식
- 배치 단위로 체크포인트 저장

#### 체크포인팅 (Checkpointing)
- 처리 상태를 주기적으로 저장
- 실패 시 마지막 체크포인트부터 재시작
- Apache Flink가 사용하는 방식

#### 멱등성 (Idempotency)
- 같은 연산을 여러 번 수행해도 결과가 동일하도록 설계
- 예: `UPDATE SET value = 42` (멱등) vs `UPDATE SET value = value + 1` (비멱등)

#### 트랜잭션 (Transactions)
- 입력 오프셋 전진 + 출력 쓰기를 원자적으로 수행
- Kafka Transactions, Flink의 2PC(Two-Phase Commit)

## 8. 스트림 처리 프레임워크

| 프레임워크 | 특징 |
|-----------|------|
| **Apache Kafka Streams** | 라이브러리 형태, 카프카와 긴밀한 통합 |
| **Apache Flink** | 낮은 지연, 강력한 상태 관리, exactly-once |
| **Apache Spark Streaming** | 마이크로배치, 배치와 통합 용이 |
| **Apache Storm** | 초기 스트림 처리 프레임워크 |
| **Google Cloud Dataflow** | Apache Beam 기반, 완전 관리형 |

## 9. 핵심 정리

```
배치 처리                     스트림 처리
───────────────────────────────────────────
유한한 데이터셋          vs   무한한 데이터 스트림
처리 후 결과 생성        vs   지속적으로 결과 업데이트
높은 지연(분~시간)       vs   낮은 지연(밀리초~초)
재처리 용이             vs   재처리 시 복잡성 존재
MapReduce, Spark        vs   Flink, Kafka Streams
```

### Lambda 아키텍처 vs Kappa 아키텍처

**Lambda 아키텍처**
```
데이터 → 배치 레이어 (정확, 느림)  ─┐
       → 스피드 레이어 (빠름, 근사) ─┤→ 서빙 레이어 → 쿼리
```
- 배치와 스트림 두 코드베이스를 모두 유지 → 복잡

**Kappa 아키텍처**
```
데이터 → 스트림 처리 레이어 → 서빙 레이어 → 쿼리
                ↑
    (재처리 시 새 버전의 컨슈머로 처음부터 재생)
```
- 스트림 처리 하나로 통일 → 단순
