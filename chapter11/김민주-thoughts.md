# 트랜잭션과 멱등성에 대한 느낀점

## Chapter 7 — 트랜잭션

ACID를 단순히 "데이터베이스가 제공하는 안전장치" 정도로만 알고 있었는데, 일관성(C)만 유일하게 애플리케이션의 책임이라는 부분이 인상 깊었다. 나머지 세 가지는 DB가 보장해줄 수 있어도, 비즈니스 로직의 불변식은 결국 개발자가 책임져야 한다는 점에서 트랜잭션을 맹목적으로 믿으면 안 되겠다는 생각이 들었다.

격리 수준이 단계별로 존재한다는 것도 새로웠다. 완벽한 직렬성이 성능 비용이 너무 크기 때문에 현실에서는 타협을 한다는 사실이, "DB가 알아서 해준다"는 막연한 믿음을 다시 생각하게 만들었다. Read Committed 수준에서도 같은 값을 두 번 읽을 때 다른 결과가 나올 수 있다는 게 직관적으로 이해하기 어려웠는데, Alice의 계좌 조회 예시를 보고 나서야 왜 격리 수준을 신중하게 골라야 하는지 실감했다.

쓰기 스큐(Write Skew) 문제가 특히 와닿았다. 두 트랜잭션이 각각 유효한 판단을 했는데 동시에 실행되면 전체적으로 불변식이 깨지는 상황은, 단순히 잠금 하나 걸어서는 해결이 안 된다는 점에서 동시성 제어가 생각보다 훨씬 복잡한 문제라는 걸 느꼈다.

## Chapter 11 — 멱등성 (스트림 처리 맥락에서)

exactly-once 의미론을 구현하는 방법 중 멱등성이 가장 현실적인 접근이라는 생각이 들었다. 트랜잭션을 쓰면 강하게 보장할 수 있지만 분산 환경에서의 2PC 비용이 크고, 마이크로배치는 지연이 생기는 반면, 멱등한 연산으로 설계하면 재처리를 두려워하지 않아도 된다는 점이 실용적으로 느껴졌다.

그런데 멱등성이 항상 쉽게 달성되는 게 아니라는 점도 눈에 들어왔다. `SET value = 42`는 멱등하지만 `SET value = value + 1`은 멱등하지 않은 것처럼, 연산의 형태를 조심해서 설계해야 한다. 막상 실무에서 스트림 파이프라인을 구성할 때 무심코 비멱등 연산을 집어넣으면 재처리 시 데이터가 틀어질 수 있겠다 싶어서, 설계 단계부터 재처리 가능성을 염두에 두는 습관이 중요하겠다고 느꼈다.

Chapter 7의 트랜잭션과 Chapter 11의 멱등성이 결국 같은 질문을 다른 방식으로 풀고 있다는 게 흥미로웠다. "한 번 처리된 것처럼 보이게 하려면 어떻게 해야 하는가"라는 문제를, 트랜잭션은 강한 보장으로, 멱등성은 연산 설계로 접근하는 것이다. 어떤 수단을 선택하느냐는 결국 일관성과 성능, 복잡성 사이의 트레이드오프를 어디서 타협하느냐의 문제였다.
