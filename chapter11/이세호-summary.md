# 서문

제공해주신 내용은 **데이터 집약적 애플리케이션 설계(DDIA) 2판**의 12장, **스트림 처리(Stream Processing)**의 도입부입니다. 일괄 처리(Batch Processing)의 한계를 짚어보고, 왜 현대 시스템에서 스트림 처리가 필수적인지를 설명하고 있습니다.

주요 내용을 핵심 위주로 정리해 드립니다.

---

## 1. 일괄 처리(Batch Processing)의 한계

이전 장(11장)에서 다룬 일괄 처리는 매우 강력하지만 결정적인 전제 조건이 있습니다.

- **데이터의 한정성(Bounded Data):** 데이터의 크기가 고정되어 있고 끝이 명확해야 합니다.
- **출력의 지연:** 정렬(Sorting)과 같은 작업은 전체 데이터를 다 읽어야만 출력을 시작할 수 있습니다.
- **주기적 실행의 문제:** 하루치 데이터를 모아서 처리하면, 결과가 반영되기까지 최대 하루의 지연 시간이 발생합니다.

## 2. 스트림 처리(Stream Processing)의 핵심 개념

실제 세상의 데이터는 대부분 **무한(Unbounded)**합니다. 사용자는 오늘도, 내일도 계속 데이터를 생성하기 때문입니다.

- **정의:** 데이터가 생성되는 즉시 혹은 점진적으로 처리하는 방식입니다.
- **목표:** 지연 시간(Latency)을 최소화하여 데이터 변화를 즉각적으로 결과에 반영하는 것입니다.
- **유사 사례:** 유닉스(Unix)의 `stdin`/`stdout`, TCP 연결, 파일 입출력 스트림, 오디오/비디오 스트리밍 등과 유사한 원리입니다.

---

### 📊 일괄 처리 vs 스트림 처리 비교

| 구분            | 일괄 처리 (Batch)            | 스트림 처리 (Stream)                 |
| --------------- | ---------------------------- | ------------------------------------ |
| **데이터 범위** | 한정됨 (Bounded)             | 무한함 (Unbounded)                   |
| **처리 단위**   | 고정된 크기의 덩어리 (Chunk) | 개별 이벤트 (Event) 또는 실시간      |
| **지연 시간**   | 높음 (시간 단위 ~ 일 단위)   | 낮음 (밀리초 ~ 초 단위)              |
| **완료 시점**   | 입력 데이터를 다 읽으면 종료 | 프로세스가 종료되지 않고 계속 실행됨 |

---

## 3. 이 장에서 다룰 내용

앞으로 12장에서는 스트림 처리를 세 가지 관점에서 깊게 파고들 예정입니다.

1. **표현 및 전송:** 이벤트 스트림을 어떻게 표현하고, 저장하며, 네트워크로 전송할 것인가?
2. **데이터베이스와 스트림:** 데이터베이스 내의 데이터 변화와 스트림은 어떤 관계가 있는가?
3. **연속적 처리:** 스트림을 지속적으로 처리하기 위한 도구와 애플리케이션 구축 방법은 무엇인가?

# Transmitting Event Streams - 서문

제공해주신 내용은 **이벤트 스트림의 전송(Transmitting Event Streams)**에 관한 핵심 원리를 다루고 있습니다. 일괄 처리에서의 '파일'이 스트림 처리에서는 어떻게 '이벤트'와 '토픽'으로 변환되는지 설명하는 부분입니다.

주요 내용을 구조화하여 정리해 드립니다.

---

## 1. 데이터의 최소 단위: 이벤트(Event)

일괄 처리에서 파일을 레코드로 파싱하듯, 스트림 처리에서는 **이벤트**가 기본 단위입니다.

- **정의:** 특정 시점에 발생한 사건에 대한 세부 정보를 담은 작고 독립적이며, **불변(Immutable)**인 객체입니다.
- **구성 요소:** 무엇이 일어났는지에 대한 데이터와 함께, 보통 사건이 발생한 시점을 나타내는 **타임스탬프**를 포함합니다.
- **예시:** \* **사용자 행동:** 페이지 조회, 상품 구매 등
- **기계 데이터:** 온도 센서 측정값, CPU 사용률 메트릭 등

- **인코딩:** 텍스트 문자열, JSON 또는 이진 형식(Avro, Protobuf 등)으로 인코딩되어 저장되거나 네트워크를 통해 전송됩니다.

---

## 2. 스트림의 구성 요소와 역할

이벤트 스트림 시스템은 데이터를 생성하는 쪽과 소비하는 쪽으로 나뉩니다.

- **생성자 (Producer / Publisher / Sender):** 이벤트를 1회 생성하여 내보내는 주체입니다.
- **소비자 (Consumer / Subscriber / Recipient):** 생성된 이벤트를 읽어서 처리하는 주체입니다.
- **토픽 (Topic / Stream):** 관련 있는 이벤트들을 하나로 묶는 논리적인 단위입니다. (파일 시스템의 '파일명'과 유사한 역할)

---

## 3. 전송 방식: 폴링(Polling) vs 알림(Notification)

생성자와 소비자를 연결하는 방식에는 두 가지 접근법이 있습니다.

### A. 폴링 (Polling)

- **방식:** 소비자가 주기적으로 데이터 저장소(파일이나 DB)를 체크하여 새로운 이벤트가 있는지 확인합니다.
- **문제점:** 실시간 처리를 위해 체크 주기(Interval)를 짧게 할수록, 새로운 데이터가 없을 때 발생하는 오버헤드가 커져 비용이 비효율적입니다.

### B. 알림 (Notification)

- **방식:** 새로운 이벤트가 발생하면 시스템이 소비자에게 즉시 알려주는 방식입니다.
- **한계:** 전통적인 관계형 데이터베이스(RDBMS)는 '트리거(Trigger)' 기능을 제공하지만, 기능이 제한적이고 설계 단계에서 핵심 요소가 아니었기에 스트림 처리에 최적화되어 있지 않습니다.

---

## 4. 결론 및 시사점

일괄 처리와 스트림 처리는 데이터의 연속성 측면에서 차이가 있습니다. 저자는 기존의 데이터베이스나 파일 시스템만으로는 저지연(Low-latency)의 연속 처리를 구현하는 데 한계가 있음을 지적하며, 이를 위해 **이벤트 알림을 전달하기 위한 전문적인 도구(예: 메시지 브로커)**가 필요함을 시사하고 있습니다.

# Transmitting Event Streams - Messaging Systems

제공해주신 내용은 메시징 시스템의 핵심 메커니즘인 **직접 통신, 메시지 브로커, 그리고 소비자 패턴(부하 분산 및 팬아웃)**에 대해 깊이 있게 다루고 있습니다.

일괄 처리의 "파일"에 대응하는 스트림 처리의 "메시지 전달 방식"을 중심으로 핵심 내용을 정리해 드립니다.

---

## 1. 메시징 시스템의 두 가지 접근법

### A. 직접 메시징 (Direct Messaging)

중간 매개체 없이 생성자와 소비자 사이의 직접적인 네트워크 통신을 사용합니다.

- **사례:** UDP 멀티캐스트(금융 업계), ZeroMQ, StatsD, 웹훅(Webhooks).
- **장점:** 중간 서버가 없어 지연 시간(Latency)이 매우 낮습니다.
- **단점:** 생성자와 소비자 모두 **항상 온라인** 상태여야 합니다. 소비자가 오프라인일 때 전송된 메시지는 손실될 가능성이 큽니다.

### B. 메시지 브로커 (Message Brokers)

메시지 스트림 처리에 최적화된 일종의 데이터베이스 역할을 하는 서버를 거치는 방식입니다.

- **사례:** RabbitMQ, ActiveMQ, Kafka, Google Cloud Pub/Sub 등.
- **특징:** 브로커가 메시지를 버퍼링하므로 소비자가 일시적으로 오프라인이어도 안전하며, **비동기적** 처리가 가능합니다.

---

## 2. 메시지 브로커 vs 데이터베이스

브로커는 데이터베이스와 유사하지만 몇 가지 결정적인 차이점이 있습니다.

| 구분            | 데이터베이스 (Database)                     | 메시지 브로커 (Message Broker)                   |
| --------------- | ------------------------------------------- | ------------------------------------------------ |
| **데이터 유지** | 명시적으로 삭제할 때까지 보관               | 소비자가 성공적으로 읽으면 자동 삭제 (일반적)    |
| **데이터 규모** | 대량의 기록 데이터를 저장 및 관리           | 큐가 짧을 때 최적의 성능 (작은 작업 세트 가정)   |
| **조회 방식**   | 쿼리 언어 및 보조 인덱스 지원               | 특정 패턴/토픽 구독 중심의 단순 선택             |
| **알림 방식**   | 데이터 변경 시 알림 기능이 약함 (폴링 필요) | 데이터 변경(메시지 도착) 시 소비자에게 즉시 통보 |

---

## 3. 다중 소비자 패턴 (Multiple Consumers)

토픽 하나에 여러 소비자가 붙을 때, 메시지를 전달하는 두 가지 주요 방식이 있습니다.

1. **부하 분산 (Load Balancing):** 각 메시지를 **단 하나의 소비자**에게만 전달합니다. 작업량이 많을 때 병렬 처리를 위해 사용합니다.
2. **팬아웃 (Fan-out):** 각 메시지를 **모든 소비자**에게 전달합니다. 여러 독립적인 서비스가 동일한 데이터 흐름을 각각 활용해야 할 때 사용합니다.

- _Kafka 예시:_ '소비자 그룹' 내에서는 부하 분산을, 서로 다른 '소비자 그룹' 간에는 팬아웃 방식을 동시에 사용할 수 있습니다.

---

## 4. 신뢰성과 메시지 재전송의 부작용

### 확인 응답 (Acknowledgments)

메시지 유실을 방지하기 위해 소비자는 처리가 완료되면 브로커에 **ACK(확인 응답)**를 보냅니다. ACK가 없으면 브로커는 처리가 실패한 것으로 간주하고 다른 소비자에게 메시지를 재전송합니다.

### 재전송 시 발생하는 문제

1. **순서 보장 파괴:** (위 본문 Figure 12-2 참고) 메시지 $m3$을 처리하던 소비자가 장애로 죽고, 그 사이 $m4$가 다른 소비자에게 처리되면, 나중에 재전송된 $m3$은 $m4$보다 늦게 처리되어 **순서가 뒤바뀔 수 있습니다.**
2. **독약 메시지 (Poison Pills):** 메시지 자체의 오류(예: 잘못된 JSON 형식)로 인해 어떤 소비자도 처리하지 못하고 무한 재시도에 빠지는 현상입니다.
3. **데드 레터 큐 (Dead Letter Queue, DLQ):** 무한 재시도를 막기 위해, 일정 횟수 이상 실패한 메시지를 별도의 큐로 격리하여 운영자가 사후 분석할 수 있게 합니다.

# Transmitting Event Streams - Log-based Message Brokers

제공해주신 내용은 현대 분산 시스템의 핵심인 **로그 기반 메시지 브로커(Log-based Message Brokers)**의 철학과 구조를 상세히 다루고 있습니다.

전통적인 메시징(JMS/AMQP)과 로그 기반 방식(Kafka 등)의 근본적인 차이를 중심으로 핵심 내용을 상세히 정리해 드립니다.

---

## 1. 메시징의 두 가지 철학: 일시적 vs 지속적

전통적인 메시징 시스템은 네트워크 패킷처럼 메시지를 **일시적인(Transient)** 것으로 간주합니다. 하지만 로그 기반 브로커는 데이터베이스나 파일 시스템처럼 **지속적인(Durable)** 기록으로 처리합니다.

- **전통적 메시징 (JMS/AMQP):** 메시지가 소비자에게 전달되고 확인(ACK)되면 즉시 삭제됩니다. 따라서 같은 메시지를 다시 읽을 수 없습니다.
- **로그 기반 메시징:** 메시지를 파일 끝에 추가(Append-only)하며, 소비자가 읽더라도 삭제되지 않습니다. 덕분에 과거의 데이터를 언제든 다시 읽는 '재처리'가 가능합니다.

---

## 2. 로그 기반 브로커의 구조: 파티션과 오프셋

대규모 데이터 처리를 위해 로그는 여러 개의 **파티션(Partition)**으로 나뉩니다.

- **파티션 (Sharding):** 하나의 토픽을 여러 파티션으로 나누어 서로 다른 머신에 분산 저장함으로써 처리량을 확장합니다.
- **오프셋 (Offset):** 파티션 내의 각 메시지에는 고유하고 순차적으로 증가하는 번호(오프셋)가 부여됩니다.
- 파티션 **내부**에서는 순서가 보장되지만, 파티션 **사이**에서는 순서 보장이 없습니다.

---

## 3. 로그 기반 vs 전통적 메시징 비교

| 구분            | 전통적 메시징 (JMS/AMQP)                  | 로그 기반 메시징 (Kafka, Kinesis)       |
| --------------- | ----------------------------------------- | --------------------------------------- |
| **메시지 삭제** | ACK를 받으면 즉시 삭제 (파괴적)           | 보존 기간이 지날 때까지 유지 (비파괴적) |
| **처리 방식**   | 메시지 단위의 부하 분산                   | 파티션 단위의 부하 분산                 |
| **순서 보장**   | 재전송 시 순서가 뒤섞일 수 있음           | 파티션 내에서는 엄격한 순서 보장        |
| **재처리**      | 불가능 (한 번 읽으면 끝)                  | 가능 (오프셋을 돌려 과거 데이터 재독해) |
| **적합한 사례** | 메시지별 처리가 비싸고 병렬성이 중요할 때 | 처리량이 매우 높고 순서가 중요할 때     |

---

## 4. 소비자 오프셋 관리 (Consumer Offsets)

로그 기반 시스템에서 소비자가 어디까지 읽었는지를 추적하는 방식은 데이터베이스의 **복제 로그 일련번호(LSN)**와 매우 유사합니다.

- **효율성:** 브로커가 모든 메시지의 ACK를 추적할 필요 없이, 소비자가 마지막으로 읽은 **번호(Offset)** 하나만 기록하면 되므로 오버헤드가 매우 적습니다.
- **장애 복구:** 소비자가 죽었다가 다시 살아나면, 마지막으로 기록된 오프셋부터 다시 읽기 시작합니다. (이 과정에서 일부 메시지가 중복 처리될 수 있습니다.)

---

## 5. 저장소 관리 및 디스크 공간

로그는 무한히 커질 수 없으므로 디스크 공간을 관리하는 메커니즘이 필요합니다.

- **세그먼트(Segment):** 로그를 작은 파일 단위로 나누어 오래된 세그먼트는 삭제하거나 아카이브 저장소로 옮깁니다.
- **링 버퍼(Ring Buffer):** 로그는 사실상 거대한 버퍼 역할을 합니다. 디스크 용량이 허용하는 한(수일~수주 분량) 데이터를 보관합니다.
- **계층형 저장소 (Tiered Storage):** 최신 데이터는 빠른 디스크에, 오래된 데이터는 저렴한 **객체 스토리지(S3 등)**에 저장하여 비용 효율성을 높입니다.

---

## 6. 로그 기반 방식의 운영적 장점

1. **재처리 및 실험 가능 (Replayability):** 새로운 기능을 개발했을 때, 어제의 데이터를 다시 흘려보내 결과가 어떻게 변하는지 테스트할 수 있습니다.
2. **소비자 간 격리 (Isolation):** 특정 소비자가 느려지거나 멈춰도 로그는 삭제되지 않고 쌓일 뿐이며, 다른 소비자에게 영향을 주지 않습니다.
3. **배치 처리와의 융합:** 로그에 저장된 데이터를 Iceberg 같은 테이블 형식으로 직접 쿼리하거나 데이터 웨어하우스로 보내기 용이합니다.

> **💡 핵심 요약:**
> 로그 기반 메시징은 메시지 전달을 **"이벤트의 기록(Record of events)"**으로 접근합니다. 이는 시스템의 **내결함성**을 높이고, 일괄 처리(Batch)와 스트림 처리 사이의 간극을 좁히는 혁신적인 접근법입니다.

# Databases and Streams - 서문

제공해주신 내용은 **데이터베이스와 스트림의 근본적인 연결 고리**를 다루고 있습니다. 전통적으로 분리된 영역이었던 두 분야가 '로그(Log)'라는 개념을 통해 어떻게 통합되는지, 그리고 이를 통한 '이벤트 소싱'과 '복제'의 원리를 설명합니다.

핵심 내용을 세분화하여 정리해 드립니다.

---

## 1. 철학의 전환: 상태(State) vs 이벤트(Event)

전통적인 데이터베이스와 스트림 방식은 데이터를 바라보는 관점이 다릅니다.

- **전통적 DB:** 데이터를 **수정 가능한 상태(Mutable State)**로 관리합니다. (예: 현재 사용자 이름 업데이트)
- **스트림 방식:** 모든 상태 변화를 **불변의 이벤트(Immutable Event)**로 기록합니다. (예: '이름이 A에서 B로 변경됨'이라는 사실을 기록)

---

## 2. 이벤트 소싱 (Event Sourcing)

이벤트 스트림을 **기록 시스템(System of Record)**으로 사용하는 설계 방식입니다.

- **동작 원리:** 데이터를 직접 수정하는 대신, 모든 상태 변화를 추가 전용(Append-only) 로그에 기록합니다.
- **파생 데이터:** 사용자가 조회하기 편한 형태의 데이터(Materialized View)는 이 로그를 읽어서 만들어낸 '파생된 결과'일 뿐입니다.
- **브로커의 역할:** 로그 기반 메시지 브로커(Kafka 등)는 이벤트를 영구 보존하고 저지연 알림을 줄 수 있어 이벤트 소싱에 매우 적합합니다.

---

## 3. 데이터베이스와 스트림의 근본적 관계

데이터베이스의 내부 동작 원리 자체가 사실상 '스트림'에 기반하고 있습니다.

### A. 복제 로그 (Replication Log)

- 리더 데이터베이스가 트랜잭션을 처리할 때 생성하는 **쓰기 이벤트의 스트림**입니다.
- 팔로워(Follower)는 이 스트림을 순서대로 읽어 자신의 복사본에 적용함으로써 리더와 동일한 상태를 유지합니다.

### B. 상태 머신 복제 (State Machine Replication)

- **원리:** 모든 이벤트가 쓰기 작업을 나타내고, 모든 복제본이 **동일한 이벤트를 동일한 순서**로 처리한다면, 모든 복제본은 결국 **동일한 상태**에 도달하게 됩니다.
- 이때 이벤트 처리는 반드시 **결정적(Deterministic)**이어야 합니다. 즉, 같은 입력에 대해 항상 같은 결과가 나와야 합니다.

---

## 4. 요약 및 시사점

저자는 "데이터베이스를 스트림으로 볼 수 있고, 스트림을 데이터베이스의 재료로 볼 수 있다"는 점을 강조합니다.

- **이벤트 스트림 =** 데이터베이스에 일어난 모든 일들의 기록.
- **데이터베이스 상태 =** 특정 시점까지의 이벤트 스트림을 모두 합친(Aggregation) 결과물.

이러한 통찰은 서로 다른 여러 데이터 시스템(DB, 검색 인덱스, 캐시 등) 간의 **데이터 일관성**을 유지하는 문제를 해결하는 핵심 열쇠가 됩니다.

# Databases and Streams - Keep Systems in Sync

제공해주신 내용은 여러 데이터 시스템(데이터베이스, 캐시, 검색 인덱스 등)을 운영할 때 발생하는 **데이터 동기화의 어려움**과 흔히 시도되는 **'이중 쓰기(Dual Writes)' 방식의 치명적인 결함**을 다루고 있습니다.

주요 내용을 핵심 위주로 정리해 드립니다.

---

## 1. 다중 데이터 시스템의 필연성

현대적인 복잡한 애플리케이션은 단 하나의 도구로 모든 요구사항을 충족할 수 없습니다. 따라서 보통 여러 기술을 조합하여 사용합니다.

- **OLTP 데이터베이스:** 사용자 요청 처리 (데이터의 원천)
- **캐시(Cache):** 응답 속도 향상
- **전체 텍스트 인덱스(Full-text Index):** 검색 쿼리 처리
- **데이터 웨어하우스(Data Warehouse):** 분석용

이처럼 데이터가 여러 곳에 복사되어 존재하기 때문에, 한 곳에서 데이터가 변경되면 다른 모든 곳도 **동기화(Sync)**되어야 합니다.

---

## 2. 이중 쓰기(Dual Writes) 방식과 그 문제점

동기화를 위해 애플리케이션 코드에서 데이터베이스와 검색 인덱스 등에 직접 데이터를 쓰는 방식을 '이중 쓰기'라고 합니다. 하지만 이 방식에는 두 가지 큰 문제가 있습니다.

### A. 경쟁 조건 (Race Condition)

가장 치명적인 문제입니다. 두 클라이언트가 동시에 같은 데이터를 수정할 때, 각 시스템에 도달하는 순서가 꼬일 수 있습니다.

- **상황:** 클라이언트 1은 값을 A로, 클라이언트 2는 B로 바꾸려 함.
- **결과:** 1. 데이터베이스에는 **A → B** 순서로 도착하여 최종 값이 **B**가 됨.

2.  검색 인덱스에는 **B → A** 순서로 도착하여 최종 값이 **A**가 됨.

- **결론:** 두 시스템의 데이터가 영구적으로 불일치하게 되며, 별도의 감지 메커니즘이 없다면 이 오류를 알아차리기도 어렵습니다.

### B. 부분적 실패 (Partial Failure)

한 시스템에 쓰는 것은 성공했지만, 다른 시스템에 쓰는 도중 네트워크 오류나 클라이언트 충돌로 실패하는 경우입니다.

- 이 문제를 해결하려면 모든 곳이 성공하거나 모두 실패하게 만드는 **원자적 커밋(Atomic Commit, 예: 2PC)**이 필요한데, 이는 구현 비용이 매우 크고 성능에 영향을 줍니다.

---

## 3. 해결을 위한 실마리: 단일 리더 모델

저자는 이 문제를 해결하기 위해 데이터베이스 복제에서 사용했던 **'리더-팔로워(Leader-Follower)'** 구조를 제안합니다.

- 데이터베이스를 **리더**로 삼아 쓰기 순서를 결정하게 함.
- 검색 인덱스나 캐시를 **팔로워**로 만들어 데이터베이스의 변경 사항을 그대로 따라가게 함.

> **💡 핵심 요약:**
> "앱이 여러 곳에 직접 쓰려고 하지 마라. 순서가 꼬이고 데이터가 깨진다. 차라리 한 곳(DB)이 대장이 되어 순서를 정하게 하고, 나머지는 그 순서를 기록한 로그를 받아 적게(Follow) 만드는 것이 훨씬 안전하다."

---

# Databases and Streams - Chnage Data Capture

제공해주신 내용은 **CDC(Change Data Capture, 변경 데이터 캡처)**의 정의, 구현 방법, 그리고 **이벤트 소싱(Event Sourcing)**과의 차이점을 심도 있게 다루고 있습니다.

데이터베이스의 내부 로그를 외부로 끌어내어 시스템 간 동기화 문제를 해결하는 핵심 원리를 정리해 드립니다.

---

## 1. CDC(Change Data Capture)의 정의와 가치

CDC는 데이터베이스에 기록된 모든 변경 사항을 관찰하고 추출하여, 이를 다른 시스템(검색 인덱스, 캐시, 데이터 웨어하우스 등)으로 복제하는 프로세스입니다.

- **동기화 문제 해결:** 앞서 언급된 '이중 쓰기'의 경쟁 조건(Race Condition)을 해결합니다. DB가 결정한 쓰기 순서대로 로그가 생성되고, 이를 소비자가 순서대로 읽기 때문입니다.
- **파생 데이터 시스템의 원천:** 검색 인덱스나 데이터 웨어하우스는 이제 DB의 변경 스트림을 구독하는 '소비자'가 됩니다.

---

## 2. CDC의 주요 구현 기술과 도구

과거에는 DB 내부 구현 세부사항이었던 복제 로그가 이제는 공용 API처럼 활용되고 있습니다.

- **주요 도구:**
- **Debezium:** MySQL, PostgreSQL, Oracle 등 다양한 DB의 로그를 표준 이벤트 스키마로 변환해주는 오픈 소스 프로젝트.
- **Kafka Connect:** 다양한 CDC 커넥터를 통해 스트림 데이터를 Kafka로 연동.

- **특징:** 대부분 **비동기 방식**입니다. DB는 소비자가 데이터를 읽었는지 기다리지 않고 커밋하므로, 시스템 성능에 영향을 주지 않지만 '복제 지연(Replication Lag)'이 발생할 수 있습니다.

---

## 3. 전체 상태 복구: 스냅샷과 로그 컴팩션

로그가 잘려나갔을 때 새로운 소비자(예: 새로 만든 검색 인덱스)가 전체 데이터를 복구하는 방법입니다.

- **초기 스냅샷 (Initial Snapshot):** 특정 시점의 DB 전체 복사본을 만들고, 그 시점의 로그 오프셋부터 변경 사항을 적용합니다.
- **로그 컴팩션 (Log Compaction):** \* **원리:** 같은 키를 가진 레코드 중 가장 최신값만 남기고 이전 기록을 삭제하는 방식입니다.
- **장점:** 로그의 크기가 전체 데이터셋의 크기에 비례하게 줄어듭니다. 덕분에 0번 오프셋부터 읽기만 해도 DB의 최신 전체 상태를 복구할 수 있어 별도의 스냅샷이 필요 없습니다. (Kafka가 지원)

---

## 4. CDC vs 이벤트 소싱 (Event Sourcing)

두 방식 모두 '로그'를 사용하지만, 추상화 수준에서 큰 차이가 있습니다.

| 구분            | CDC (Change Data Capture)          | 이벤트 소싱 (Event Sourcing)       |
| --------------- | ---------------------------------- | ---------------------------------- |
| **데이터 모델** | 가변적(Mutable) DB 사용            | 불변의(Immutable) 이벤트 로그 중심 |
| **이벤트 수준** | 저수준 상태 변경 (Row 단위)        | 고수준 비즈니스 의도 (Action 단위) |
| **도입 난이도** | 기존 DB에 쉽게 추가 가능           | 앱 설계 자체를 변경해야 함         |
| **로그 컴팩션** | 최신값만 남기기 가능 (Update 중심) | 불가능 (모든 히스토리가 의미 있음) |

---

## 5. 설계 시 주의사항: 스키마와 아웃박스 패턴

CDC를 도입할 때 DB 스키마가 외부 서비스에 노출되는 문제가 발생할 수 있습니다.

- **데이터 계약(Data Contracts):** DB 스키마 변경이 다운스트림 서비스를 깨뜨리지 않도록 관리해야 합니다.
- **아웃박스 패턴 (Outbox Pattern):** \* 내부 도메인 모델 대신, 외부 노출용 '아웃박스 테이블'을 별도로 둡니다.
- 비즈니스 로직과 아웃박스 기록을 **하나의 트랜잭션**으로 묶어 처리함으로써 이중 쓰기 문제를 피하면서도 내부/외부 모델을 분리(Decoupling)합니다.

> **💡 핵심 요약:**
> CDC는 데이터베이스를 단순한 '저장소'가 아닌 **'이벤트의 원천(Source of Events)'**으로 변환시킵니다. 이를 통해 복잡한 분산 시스템 간의 데이터 일관성을 우아하게 유지할 수 있습니다.

---

**다음 단계로 무엇을 도와드릴까요?**

- 본문에서 언급된 **'아웃박스 패턴'**의 구체적인 구현 예시가 궁금하신가요?
- 혹은 CDC를 통해 실시간 분석 시스템을 구축하는 **스트림 처리(Stream Processing)** 단계로 넘어갈까요?

# Databases and Streams - State, Streams, and Immutability

제공해주신 내용은 **불변성(Immutability)**의 철학이 어떻게 현대 데이터 시스템의 강력한 무기가 되는지를 다루고 있습니다. 가변적인 '상태'와 불변의 '로그'가 서로 대립하는 것이 아니라, 하나의 현상을 바라보는 두 가지 관점임을 설명합니다.

핵심 내용을 "철학, 장점, 한계"의 세 가지 축으로 정리해 드립니다.

---

## 1. 상태와 스트림의 수학적 관계

저자는 애플리케이션의 **상태(State)**와 **이벤트 스트림(Changelog)**의 관계를 미적분에 비유합니다.

- **적분 (Integration):** 이벤트 스트림을 시간에 따라 모두 합치면(연속적으로 적용하면) 현재의 **상태**가 됩니다.
- **미분 (Differentiation):** 상태의 변화를 시간에 따라 관찰하면 **이벤트 스트림**이 됩니다.
- **결론:** 데이터베이스(상태)는 사실 성능을 위해 스트림의 끝부분을 캐싱해둔 것에 불과합니다. 로그만 있다면 언제든 상태를 다시 계산해낼 수 있습니다.

---

## 2. 불변 이벤트(Immutable Events)의 4가지 장점

### ① 회계적 감사 가능성 (Auditability)

회계사가 실수했을 때 지우개로 지우지 않고 '수정 분개'를 추가하듯, 불변 로그는 시스템에서 일어난 모든 일의 완벽한 히스토리를 제공합니다. 이는 버그 복구 및 고객 서비스 대응에 필수적입니다.

### ② 풍부한 정보량

가변 DB에서 장바구니의 아이템을 삭제하면 그 사실이 영원히 사라지지만, 이벤트 로그는 "사용자가 고민하다가 삭제했다"는 분석적 통찰(Analytics)을 남깁니다.

### ③ 유연한 뷰 도출 (CQRS)

하나의 로그를 기반으로 여러 개의 **읽기 전용 뷰(Materialized Views)**를 만들 수 있습니다.

- 새로운 기능이 필요하면 기존 시스템을 건드리지 않고 로그를 처음부터 읽어 새로운 인덱스만 추가하면 됩니다.
- 쓰기에 최적화된 형태(로그)와 읽기에 최적화된 형태(인덱스, 캐시)를 분리하여 운영할 수 있습니다.

### ④ 동시성 제어의 단순화

여러 객체를 수정해야 하는 복잡한 트랜잭션 대신, 사용자 액션을 하나의 **자가 완결적(Self-contained) 이벤트**로 설계하여 로그에 추가하기만 하면 됩니다. 로그가 이벤트의 순서를 결정해주므로 동시성 문제가 획기적으로 줄어듭니다.

---

## 3. 불변성의 한계와 현실적인 문제

불변 히스토리를 영원히 보관하는 것이 항상 정답은 아닙니다.

- **데이터 파편화 및 성능:** 업데이트가 빈번한 데이터셋의 경우 로그가 너무 커져 컴팩션(Compaction)과 가비지 컬렉션의 성능이 중요해집니다.
- **법적/행정적 삭제 (GDPR):** "잊힐 권리"처럼 데이터를 영구 삭제해야 하는 경우, 불변성은 오히려 걸림돌이 됩니다.
- **해결책 - 크립토 슈레딩 (Crypto-shredding):** 데이터를 암호화해서 저장하고, 삭제가 필요할 때 해당 **암호화 키만 삭제**하여 데이터를 복구 불가능하게 만드는 방식입니다. (데이터는 남지만 읽을 수 없게 됨)

---

## 4. 요약 및 비교

| 구분     | 가변적 데이터베이스 (Mutable DB) | 불변 이벤트 로그 (Immutable Log)      |
| -------- | -------------------------------- | ------------------------------------- |
| **중점** | 현재의 최신 상태 (State)         | 발생한 사실의 기록 (History)          |
| **장점** | 읽기 성능 최적화, 익숙한 모델    | 감사 용이, 유연한 분석, 동시성 단순화 |
| **단점** | 히스토리 유실, 경쟁 조건 발생    | 저장 공간 증대, 데이터 삭제의 어려움  |

---

**이 장의 핵심을 이해하셨나요?** 이제 시스템 설계 시 **"어떻게 저장할 것인가"**보다 **"무엇이 일어났는가(이벤트)"**에 집중하면 훨씬 더 유연하고 견고한 시스템을 만들 수 있다는 점이 이 글의 요지입니다.

**다음 단계로 무엇을 도와드릴까요?**

- 본문에서 언급된 **CQRS(명령 조회 책임 분리) 패턴**을 실제 코드로 구현하는 예시를 보여드릴까요?
- 혹은 **데이터 삭제(GDPR 대응)**를 위한 기술적 대안들을 더 자세히 알아볼까요?

# Processing Streams - 서문

제공해주신 내용은 스트림을 확보한 후 이를 어떻게 활용하고 처리할 것인지에 대한 **스트림 처리(Stream Processing)**의 핵심 옵션과 배기(Batch) 처리와의 결정적인 차이점을 설명하고 있습니다.

핵심 내용을 세 가지 활용 방안과 스트림 처리만의 고유한 특성을 중심으로 정리해 드립니다.

---

## 1. 스트림의 세 가지 활용 방안

스트림 데이터를 확보한 후 할 수 있는 일은 크게 세 가지로 나뉩니다.

1. **저장소 업데이트:** 이벤트를 읽어 데이터베이스, 캐시, 검색 인덱스 등에 기록합니다. 이는 다른 클라이언트가 데이터를 쿼리할 수 있게 시스템 간 동기화를 유지하는 방법입니다. (일괄 처리의 결과물 생성과 유사)
2. **사용자에게 직접 전달:** 이메일 알림, 푸시 알림, 실시간 대시보드 시각화 등 사람(Human)을 최종 소비자로 하여 이벤트를 즉시 전달합니다.
3. **새로운 스트림 생성 (파이프라인):** 하나 이상의 입력 스트림을 처리하여 하나 이상의 **출력 스트림**을 만들어냅니다. 이를 처리하는 코드 단위를 **연산자(Operator)** 또는 **작업(Job)**이라고 부릅니다.

---

## 2. 스트림 프로세서(Stream Processor)의 특징

스트림 프로세서는 이전 장에서 다룬 유닉스 프로세스나 MapReduce 작업과 유사한 데이터 흐름 패턴을 가집니다.

- **읽기 전용 입력:** 입력 스트림을 변경하지 않고 읽기만 합니다.
- **추가 전용 출력:** 결과를 새로운 위치에 append-only 방식으로 기록합니다.
- **병렬화:** 맵리듀스와 유사하게 샤딩(Sharding) 및 병렬 처리 기법을 사용하여 확장할 수 있습니다.

---

## 3. 일괄 처리(Batch) vs 스트림 처리(Stream)의 결정적 차이

스트림 처리는 **"끝이 없다(Never ends)"**는 속성 때문에 일괄 처리와는 다른 설계 방식이 필요합니다.

| 구분                           | 일괄 처리 (Batch)                         | 스트림 처리 (Stream)                                               |
| ------------------------------ | ----------------------------------------- | ------------------------------------------------------------------ |
| **데이터 범위**                | 유한함 (Bounded)                          | **무한함 (Unbounded)**                                             |
| **정렬(Sorting)**              | 전체 데이터를 정렬할 수 있음              | 정렬이 불가능함 (끝이 없으므로)                                    |
| **조인 방식**                  | 정렬-병합 조인(Sort-merge join) 사용 가능 | 정렬 기반 조인 사용 불가                                           |
| **내결함성 (Fault-tolerance)** | 실패 시 처음부터 다시 실행 가능           | **처음부터 다시 시작하는 것이 불가능** (수년간 실행될 수 있으므로) |

---

## 4. 요약 및 시사점

저자는 스트림 처리가 일괄 처리의 개념을 계승하면서도, **'무한성'**이라는 특징 때문에 기존의 알고리즘(정렬 등)이나 복구 전략을 그대로 사용할 수 없음을 강조합니다. 따라서 스트림 처리 시스템은 **상태 관리(State Management)**와 **지속적인 내결함성**을 위한 새로운 접근법이 필요합니다.

---

**이어서 어떤 내용을 도와드릴까요?**

- 스트림이 끝나지 않는데 어떻게 데이터를 묶어서 처리하는지, **윈도잉(Windowing)** 개념에 대해 알아볼까요?
- 혹은 스트림 처리 도중 장애가 났을 때 데이터 유실 없이 복구하는 **내결함성 메커니즘**이 궁금하신가요?

# Processing Streams - Uses of Stream Processing

제공해주신 내용은 스트림 처리의 구체적인 활용 분야와 기술적 방법론(CEP, 분석, 뷰 유지보수, 검색 등)을 상세히 다루고 있습니다.

핵심 내용을 주제별로 분류하여 깔끔하게 정리해 드립니다.

---

## 1. 스트림 처리의 주요 활용 사례

스트림 처리는 단순히 데이터를 옮기는 것을 넘어, 실시간으로 복잡한 판단을 내리는 데 사용됩니다.

- **사기 탐지(Fraud Detection):** 신용카드 사용 패턴 변화 감지 및 차단.
- **거래 시스템(Trading):** 금융 시장의 가격 변화에 따른 자동 매매.
- **제조 및 모니터링:** 공장 기계의 상태를 감시하여 고장 징후 즉각 식별.
- **군사/지능 시스템:** 공격 징후 포착 및 경보 발령.

---

## 2. 복잡한 이벤트 처리 (CEP) vs 스트림 분석 (Analytics)

이 두 영역은 비슷해 보이지만 데이터에 접근하는 방식이 다릅니다.

| 구분     | 복잡한 이벤트 처리 (CEP)                | 스트림 분석 (Stream Analytics)                 |
| -------- | --------------------------------------- | ---------------------------------------------- |
| **목적** | 특정 **이벤트 패턴** 찾기 (시퀀스 매칭) | 대량 이벤트의 **통계적 요약** (집계)           |
| **비유** | 문자열에서의 '정규 표현식'과 유사       | 데이터베이스의 '집계 쿼리'와 유사              |
| **예시** | "A 발생 후 1분 내에 B가 없는 경우" 감지 | "최근 5분간의 초당 평균 요청 수" 계산          |
| **특징** | 상태 머신(State Machine) 내부 유지      | 윈도우(Window) 단위 집계, 확률적 알고리즘 사용 |

> **💡 데이터베이스와 쿼리의 관계 역전**
> 일반적인 DB는 **데이터가 고정**되어 있고 **쿼리가 일시적**이지만, CEP와 스트림 검색 시스템은 **쿼리가 고정(Standing Queries)**되어 있고 **데이터가 그 위를 흘러갑니다.**

---

## 3. 구체화된 뷰 유지보수 (Maintaining Materialized Views)

데이터베이스의 변경 내용을 스트림으로 추출하여 캐시나 검색 인덱스를 업데이트하는 방식입니다.

- **증분 뷰 유지보수 (Incremental View Maintenance, IVM):** \* 기존 DB의 뷰는 주기적으로 전체를 다시 계산(Batch)하여 비효율적이고 데이터가 신선하지 않았습니다.
- IVM은 **변경된 부분만 계산**하여 즉시 반영하므로 효율적이고 실시간성이 높습니다.
- 관련 기술: Materialize, RisingWave, ClickHouse, Feldera 등.

---

## 4. 스트림 검색 (Search on Streams)

전통적인 검색 엔진이 문서를 색인하고 나중에 쿼리를 던지는 방식이라면, 스트림 검색은 그 반대입니다.

- **작동 방식:** 사용자의 검색 쿼리(예: "특정 가격대의 부동산 알림")를 미리 등록해두고, 새로운 이벤트가 들어올 때마다 해당 쿼리에 매칭되는지 확인합니다.
- **도구:** Elasticsearch의 **Percolator** 기능이 대표적인 예입니다.

---

## 5. 이벤트 기반 아키텍처(EDA)와의 차이

메시지 전달 시스템과 스트림 처리는 비슷해 보이지만 지향점이 다릅니다.

- **액터 모델 (Actor Model):** 주로 동시성 관리와 일대일 통신에 집중하며, 메시지가 일시적(Ephemeral)인 경우가 많습니다.
- **스트림 처리:** 데이터 관리 기법에 집중하며, 비순환 파이프라인(Acyclic Pipeline) 구조로 결과물을 만들어냅니다. 내결함성(Fault-tolerance) 보장이 더 강력합니다.

---

## 6. 요약 및 시사점

스트림 처리는 단순히 "빠른 처리"를 넘어, **데이터가 들어오는 즉시 비즈니스 가치를 창출**하는 기술입니다.

- **정확성:** 확률적 알고리즘(HyperLogLog 등)을 써서 메모리를 아낄 수도 있지만, 스트림 처리 자체가 본질적으로 부정확한 것은 아닙니다.
- **시간의 중요성:** 통계를 낼 때 '윈도우(Window)'라는 개념이 핵심이 되며, 이는 다음에 이어질 "시간에 대한 추론" 섹션의 토대가 됩니다.

---

- 스트림 처리에서 가장 까다로운 **'윈도잉(Windowing)'과 시간 개념**에 대해 정리해 드릴까요?
- 혹은 본문에서 언급된 **확률적 알고리즘(Bloom Filter, HyperLogLog)**이 어떻게 작동하는지 궁금하신가요?

# Processing Streams - Reasoning About Time

제공해주신 내용은 스트림 처리에서 가장 까다롭고 중요한 주제 중 하나인 **시간(Time)에 대한 추론**을 다루고 있습니다. 이벤트가 실제로 발생한 시간과 시스템이 이를 처리하는 시간 사이의 간극, 그리고 이를 해결하기 위한 윈도잉(Windowing) 기법을 정리해 드립니다.

---

## 1. 이벤트 시간(Event Time) vs 처리 시간(Processing Time)

스트림 처리에서 시간을 정의하는 방식은 두 가지입니다. 이를 혼동하면 데이터에 심각한 왜곡이 생길 수 있습니다.

- **이벤트 시간 (Event Time):** 이벤트가 실제로 발생한 시점의 타임스탬프입니다. (예: 사용자가 클릭한 시간)
- **처리 시간 (Processing Time):** 스트림 프로세서가 이벤트를 읽어 들인 시점의 시스템 시계입니다.

### 왜 처리 시간을 쓰면 안 될까?

네트워크 지연, 시스템 장애, 혹은 과거 데이터 재처리(Backlog) 등의 이유로 지연이 발생하면 처리 시간 기준의 통계는 엉망이 됩니다. 예를 들어, 실제로는 매초 1건씩 발생한 이벤트라도 시스템이 1분간 멈췄다 가동되면, 처리 시간 기준으로는 특정 1초에 60건이 몰린 것처럼 '가짜 스파이크'가 나타납니다.

---

## 2. 낙오자 이벤트(Straggler Events) 대응 전략

이벤트 시간을 기준으로 윈도우(예: "37분~38분 사이의 데이터")를 설정하면, 네트워크 문제로 뒤늦게 도착하는 '낙오자'들을 어떻게 처리할지가 문제입니다.

- **대기 및 타임아웃:** 특정 시간 동안 이벤트가 안 오면 윈도우를 닫습니다.
- **전략 1 - 무시:** 낙오자 비율이 적다면 단순히 버립니다.
- **전략 2 - 수정(Correction):** 이미 계산된 결과를 취소(Retract)하고, 낙오자를 포함한 업데이트된 결과를 다시 발행합니다.
- **워터마크(Watermarks):** "이제 시간 $t$ 이전의 이벤트는 더 이상 오지 않는다"는 특수 메시지를 통해 윈도우를 안전하게 닫습니다.

---

## 3. 사용자 기기 시계의 불확실성 해결

모바일 앱처럼 오프라인 상태에서 이벤트를 버퍼링했다가 나중에 보내는 경우, 기기 시계를 믿기 어렵습니다(사용자가 시간을 임의로 바꿀 수 있음). 이를 위해 **세 가지 타임스탬프**를 로그에 남겨 오차를 보정합니다.

1. 이벤트 발생 시간 (기기 기준)
2. 이벤트 전송 시간 (기기 기준)
3. 이벤트 수신 시간 (서버 기준)

> **보정 공식:** $3번(서버 수신) - 2번(기기 전송)$을 통해 네트워크 지연 및 기기 시계 오차를 계산한 뒤, 이를 $1번$에 적용하여 실제 발생 시간을 추정합니다.

---

## 4. 윈도우의 종류 (Types of Windows)

데이터를 시간 단위로 묶는 방식에는 네 가지 주요 패턴이 있습니다.

| 종류                   | 특징                            | 비유                                        |
| ---------------------- | ------------------------------- | ------------------------------------------- |
| **텀블링 (Tumbling)**  | 고정 크기, 겹치지 않음          | 1분 단위로 딱딱 끊어서 계산                 |
| **호핑 (Hopping)**     | 고정 크기, 일정 간격으로 겹침   | 5분 치 데이터를 1분마다 슬라이드하며 계산   |
| **슬라이딩 (Sliding)** | 시간 간격 내의 모든 이벤트 포함 | 특정 시점 기준 과거 5분 이내의 기록들       |
| **세션 (Session)**     | 고정 크기 없음, 활동 기반       | 사용자가 접속해서 30분간 활동이 없을 때까지 |

---

## 5. 요약 및 주의사항

윈도잉 작업은 **임시 상태(State)**를 유지해야 합니다. 특히 슬라이딩 윈도우나 세션 윈도우는 이벤트가 끝날 때까지 데이터를 버퍼링해야 하므로, 처리량이 높을 경우 메모리나 디스크 용량 관리가 매우 중요합니다.

# Processing Streams - Stream Joins

제공해주신 내용은 스트림 처리의 정점이라 할 수 있는 **스트림 조인(Stream Joins)**에 관한 깊이 있는 설명입니다. 일괄 처리와 달리 끝이 없는 데이터 흐름 속에서 서로 다른 데이터를 어떻게 결합하는지, 그리고 그 과정에서 발생하는 '시간'과 '순서'의 문제를 핵심 위주로 정리해 드립니다.

---

## 1. 스트림 조인의 세 가지 유형

무한한 데이터셋에서 조인을 수행하려면 반드시 **상태(State)**를 유지해야 합니다. 조인의 대상에 따라 다음과 같이 세 가지로 나뉩니다.

### ① 스트림-스트림 조인 (윈도우 조인)

두 개의 이벤트 스트림을 특정 조건(주로 시간) 내에서 결합합니다.

- **예시:** 검색 이벤트와 해당 검색 결과에 대한 클릭 이벤트 결합 (클릭률 계산).
- **특징:** 클릭은 검색 직후에 일어날 수도, 며칠 뒤에 일어날 수도 있습니다. 따라서 특정 **시간 윈도우**(예: 1시간 이내)를 설정하고 그동안의 이벤트를 메모리나 디스크에 상태로 저장해 두어야 합니다.
- **작동:** 새로운 이벤트가 올 때마다 상대방의 '상태 인덱스'를 뒤져서 매칭되는 것이 있는지 확인합니다.

### ② 스트림-테이블 조인 (스트림 보충)

하나의 이벤트 스트림을 데이터베이스의 정보로 풍성하게 만듭니다(Enrichment).

- **예시:** '사용자 ID'만 있는 활동 로그에 '사용자 프로필(나이, 지역 등)' 정보를 붙여서 출력.
- **특징:** 매번 원격 DB를 조회하면 속도가 느려지므로, DB의 복사본을 스트림 프로세서 로컬에 둡니다.
- **핵심:** DB가 변경될 때마다 **CDC(변경 데이터 캡처)**를 통해 로컬 사본을 업데이트합니다. 사실상 '활동 스트림'과 '프로필 업데이트 스트림'의 조인입니다.

### ③ 테이블-테이블 조인 (구체화된 뷰 유지보수)

두 테이블의 변화를 모두 관찰하여 조인된 결과를 실시간으로 갱신합니다.

- **예시:** 소셜 미디어의 타임라인. '게시물(Posts) 테이블'과 '팔로우(Follows) 테이블'을 조인하여 각 사용자의 인박스를 업데이트.
- **수학적 비유:** 두 테이블 $u, v$의 조인 결과인 $u \cdot v$의 변화(미분)는 라이프니츠의 미분 법칙을 따릅니다.

$$(u \cdot v)' = u'v + uv'$$

즉, 게시물이 변할 때($u'$) 현재 팔로우 정보($v$)와 조인하고, 팔로우가 변할 때($v'$) 현재 게시물($u$)과 조인하여 결과를 갱신합니다.

---

## 2. 시간 의존성(Time-dependence)과 결정론

스트림 조인에서 가장 골치 아픈 문제는 **이벤트의 순서**입니다.

- **비결정론의 위험:** 여러 스트림에서 오는 이벤트들의 순서가 보장되지 않으면, 재실행 시 결과가 달라질 수 있습니다. 예를 들어 '세금 인상' 이벤트와 '판매' 이벤트 중 무엇이 먼저 처리되느냐에 따라 세금 계산이 달라집니다.
- **느리게 변하는 차원 (SCD):** 데이터 웨어하우스에서는 이를 해결하기 위해 특정 시점의 레코드 버전에 **고유 ID**를 부여합니다. 판매 시점의 세금 요율 ID를 직접 참조하게 함으로써, 나중에 다시 계산해도 동일한 결과가 나오도록 만듭니다.

## 3. 핵심 통찰 및 요약

| 구분          | 스트림-스트림              | 스트림-테이블              | 테이블-테이블             |
| ------------- | -------------------------- | -------------------------- | ------------------------- |
| **상태 범위** | 특정 시간 윈도우 내 이벤트 | 테이블 전체 (최신 버전)    | 테이블 전체 (모든 관계)   |
| **주요 목적** | 연관된 이벤트 매칭         | 데이터 풍성화 (Enrichment) | 복합 뷰/캐시 실시간 유지  |
| **상태 유지** | 양쪽 스트림 모두 버퍼링    | 테이블 사본만 유지         | 양쪽 테이블의 인덱스 유지 |

# Processing Streams - Fault Tolerance

제공해주신 내용은 스트림 처리의 마지막 난관인 **내결함성(Fault Tolerance)**과 **정확히 한 번(Exactly-once)** 처리 보장 방법을 다루고 있습니다. 멈추지 않는 스트림에서 어떻게 데이터 유실이나 중복 없이 장애를 복구하는지 핵심 위주로 정리해 드립니다.

---

## 1. 스트림 처리의 내결함성: 배치와의 차이

- **배치 처리:** 실패하면 해당 작업을 버리고 처음부터 다시 실행하면 됩니다(입력이 불변이므로).
- **스트림 처리:** 끝이 없으므로 "작업이 끝날 때까지 기다렸다가 결과를 출력"하는 방식이 불가능합니다. 따라서 **상태(State)**를 어떻게 유지하고 복구할지가 관건입니다.

---

## 2. 정확히 한 번(Exactly-once) 보장 전략

실제로는 여러 번 처리될 수 있지만, 외부에서 보기에 **단 한 번만 처리된 것과 같은 효과**를 내는 것이 목표입니다.

### ① 마이크로배치(Microbatching) 및 체크포인트

- **마이크로배치:** 스트림을 1초 단위 등 아주 작은 배치로 쪼개어 처리합니다(예: Spark Streaming).
- **체크포인트:** 스트림 사이에 '장벽(Barrier)'을 두어 상태를 주기적으로 저장소에 기록합니다(예: Apache Flink). 장애 시 마지막 체크포인트부터 다시 시작합니다.

### ② 원자적 커밋 (Atomic Commit)

이벤트 처리가 성공했을 때만 **[출력 데이터 + 상태 업데이트 + 입력 확인(ACK)]**이 한꺼번에(Atomically) 일어나도록 보장합니다. 내부 트랜잭션 관리를 통해 효율적으로 구현하며, Google Cloud Dataflow나 Kafka 트랜잭션이 이 방식을 사용합니다.

### ③ 멱등성 (Idempotence)

동일한 작업을 여러 번 수행해도 결과가 같은 성질을 이용합니다.

- **예시:** 카운터를 1 올리는 것은 멱등적이지 않지만(두 번 하면 2가 됨), 특정 키의 값을 특정 오프셋 번호와 함께 저장하는 것은 여러 번 실행해도 결과가 같습니다.
- **활용:** 메시지의 고유 오프셋을 DB에 함께 기록하면, 이미 처리된 오프셋인지 확인하여 중복 처리를 방지할 수 있습니다.

---

## 3. 장애 후 상태 복구 (Rebuilding State)

윈도우 집계나 조인을 위해 유지하던 '상태'를 복구하는 방법입니다.

- **원격 저장소 활용:** 상태를 외부 DB에 두지만, 성능 저하의 위험이 있습니다.
- **로컬 복제:** 상태 변경 사항을 별도의 전용 로그(예: Kafka의 컴팩션된 토픽)에 기록해두었다가 장애 시 이를 읽어 로컬 상태를 재구성합니다.
- **입력 재생:** 윈도우가 짧다면 그냥 과거 입력 데이터를 다시 읽어서 상태를 처음부터 계산해냅니다.

## 1. 스트림 처리의 본질: 연속적인 배치 처리

스트림 처리는 **무한한(Unbounded)** 데이터셋을 대상으로 한다는 점을 제외하면, 11장에서 다룬 일괄 처리(Batch)와 매우 유사합니다. 여기서 메시지 브로커와 이벤트 로그는 배치 시스템의 **파일 시스템** 역할을 수행합니다.

## 2. 메시지 브로커의 두 가지 모델

| 모델                | 방식                                          | 적합한 사례                                      |
| ------------------- | --------------------------------------------- | ------------------------------------------------ |
| **AMQP/JMS 스타일** | 개별 메시지를 소비자에게 할당 및 처리 후 삭제 | 비동기 RPC, 태스크 큐 (순서가 중요하지 않을 때)  |
| **로그 기반 모델**  | 샤드(파티션)의 모든 메시지를 순서대로 전달    | 스트림 처리, 파생 상태 생성 (재처리가 필요할 때) |

---

## 3. 데이터베이스와 스트림의 통합

데이터베이스의 쓰기 작업을 **스트림**으로 바라보는 관점은 시스템 통합에 강력한 기회를 제공합니다.

- **CDC(변경 데이터 캡처) & 이벤트 소싱:** DB 히스토리를 스트림으로 추출.
- **시스템 동기화:** 추출된 스트림을 통해 검색 인덱스, 캐시, 분석 시스템을 실시간으로 업데이트.
- **로그 컴팩션:** 로그만으로 DB의 전체 상태를 복구할 수 있게 함.

---

## 4. 스트림 조인의 세 가지 유형 (복습)

1. **스트림-스트림:** 두 이벤트 간의 시간적 연관성 검색 (예: 30분 내 검색과 클릭).
2. **스트림-테이블:** 이벤트에 DB 정보를 추가하여 풍성하게 만듦 (Enrichment).
3. **테이블-테이블:** 두 DB의 변경 사항을 결합하여 실시간 구체화된 뷰 유지.

---

## 5. 시간과 내결함성

- **시간의 복잡성:** 처리 시간(Processing time)과 이벤트 시간(Event timestamps)을 엄격히 구분해야 하며, 늦게 도착하는 낙오자 이벤트(Stragglers) 처리가 필수적입니다.
- **정확히 한 번(Exactly-once):** 장애 발생 시 부분적인 출력을 폐기하고 중복을 방지하기 위해 **마이크로배치, 체크포인트, 트랜잭션, 멱등성 쓰기** 기법을 사용합니다.
